## Proposed changes
1. Keep a single immediate-send helper on `Input`: `send_queued_row_immediately(conversation_id, query_id, text, is_command, trigger, ctx)`. The panel Send-now event path passes `QueuedPromptSendNowTrigger::SendNowButton`; the Enter path passes `QueuedPromptSendNowTrigger::EnterOnEmptyInput`. The helper must remain the only place that removes fired rows and emits `QueuedPromptSentNow` so the two triggers cannot diverge.
2. Keep Enter sendability owned by `QueuedPromptsPanelView`. The panel already knows whether it is visible, whether pane-level sending is available, whether a row is locked, whether an inline edit is active, and whether the CLI-agent rich input is open. The input should ask the panel predicate instead of duplicating that logic.
3. Keep the host input editor handle in the panel and read emptiness live at decision time. The cached empty/non-empty state should only dampen re-render notifications for the header hint; it must not be the source of truth for dispatch. This prevents same-update buffer changes from causing stale Enter behavior.
4. In `Input::input_enter`, evaluate Enter-to-send after menu acceptance/newline decisions and before normal submit/queue handling. If the panel predicate says Enter can send, look up the active conversation's queue head, skip locked rows, and send exactly that row through `send_queued_row_immediately`.
5. Do not add a new feature flag. The behavior remains covered by `FeatureFlag::QueueSlashCommand` through the existing panel `should_render` gate.
6. Keep pane-level permission updates flowing from the host into the panel through `set_can_send_prompt`. Read-only viewer state must disable row Send-now buttons and the Enter path together, while preserving edit/delete/reorder behavior.
7. Render the header hint inside `render_header` using the existing `render_keystroke_with_color_overrides` keycap component and theme-derived colors. The hint should be driven solely by `should_show_enter_hint` so display and dispatch stay aligned.
