# TECH: TUI Warping Indicator (CODE-1816)

## Context

While an agent turn is running, the TUI shows no in-progress indicator. Per the Figma design ([TUI file, node 323-17216](https://www.figma.com/design/yg5nbPZuGoAszHS3Rhvehu/TUI?node-id=323-17216&m=dev)), a `⋮ Warping... (1s)` row should render between the transcript and the input box while the selected conversation is in progress — a standalone row in the session view's column, not part of the streaming agent block:

- Layout: `<spinner> <Warping...> <(Ns)>` on one row, one space (1 cell) between parts, one blank row above (separating it from the transcript's last row).
- Spinner: follows the prototype's rotation choreography — a 180° rotation right, a 180° rotation back left, a rest at vertical, then fast spins right (one and a half turns), restarting. The prototype's motion data isn't machine-readable via the Figma API (the CharAni variants only expose per-frame glyphs), so the choreography is encoded from designer/user observation, mapping each 45° step to the nearest three-dot orientation glyph (`⋮ ⋰ ⋯ ⋱`, repeating every 180°). Non-bold, themed bright magenta.
- "Warping...": bold, themed bright-magenta base with a shimmer band sweeping left→right toward a contrast-aware theme text color. This maps the animation component's [`Lilac-200` and `white` tokens](https://www.figma.com/design/yg5nbPZuGoAszHS3Rhvehu/TUI?node-id=64-9416&m=dev) (`#d2b5ff` and `#f1f1f1`) to semantic theme roles instead of hardcoding the design hex values, with the peak selected against the resolved terminal background so it remains visible in light themes.
- `(Ns)`: elapsed whole seconds since the request started, using the theme foreground at `sub_text_opacity`.

There is no `PRODUCT.md`; the Figma node above is the behavior source of truth.

### Current state

- GUI shimmer: `ShimmeringTextElement` computes a moving band center and per-glyph intensity, then lerps base→shimmer color per glyph ([`crates/warpui_core/src/elements/gui/shimmering_text.rs:119-182 @ fbe26f9`](https://github.com/warpdotdev/warp/blob/fbe26f99c2b508c2e450db4ceccfbdb2a0c8766e/crates/warpui_core/src/elements/gui/shimmering_text.rs#L119-L182)); config in [`shimmering_text/config.rs @ fbe26f9`](https://github.com/warpdotdev/warp/blob/fbe26f99c2b508c2e450db4ceccfbdb2a0c8766e/crates/warpui_core/src/elements/gui/shimmering_text/config.rs) (period 3s, radius 6, padding 8). The math is plain `f32`/`Duration`; only the glyph/ligature mapping is GUI-specific. `elements::gui` is always compiled in `warpui_core`, so these types are already visible to `warp_tui`.
- GUI warping indicator ([`app/src/ai/blocklist/block/view_impl/common.rs:203,551 @ fbe26f9`](https://github.com/warpdotdev/warp/blob/fbe26f99c2b508c2e450db4ceccfbdb2a0c8766e/app/src/ai/blocklist/block/view_impl/common.rs#L203)) uses gray→white shimmer colors and has no rotating spinner; the lilac three-dot spinner is new to the TUI design, so only the shimmer math is reusable.
- TUI text: `TuiText` applies one `TuiStyle` to the whole string ([`crates/warpui_core/src/elements/tui/text.rs @ fbe26f9`](https://github.com/warpdotdev/warp/blob/fbe26f99c2b508c2e450db4ceccfbdb2a0c8766e/crates/warpui_core/src/elements/tui/text.rs)) — no per-character color today, but `TuiElement::render` writes `ratatui` cells directly and `Color::Rgb` is supported.
- Session surface: `TuiTerminalSessionView::render` already lays out a `TuiFlex::column()` of transcript / input box / footer (`crates/warp_tui/src/terminal_session_view.rs`), so a conditional row slots in between the transcript and the input box. The view holds the surface's `ConversationSelectionHandle` (for the selected conversation); `BlocklistAIHistoryModel` resolves the conversation, whose `status().is_in_progress()` and `latest_exchange().start_time` provide the visibility condition and animation anchor.
- GUI parity: the GUI warping indicator's core visibility condition is `conversation.status().is_in_progress()` (`app/src/ai/blocklist/block/status_bar.rs`, `render_warping_indicator_for_latest_exchange`); its other clauses (passive requests, long-running-command takeover states) are GUI/PTY concepts with no TUI counterpart today.
- Redraws are invalidation-driven only ([`spawn_tui_driver`, `crates/warpui_core/src/runtime/mod.rs:381-471 @ fbe26f9`](https://github.com/warpdotdev/warp/blob/fbe26f99c2b508c2e450db4ceccfbdb2a0c8766e/crates/warpui_core/src/runtime/mod.rs#L381-L471)); the TUI paint path has no repaint scheduling. The GUI's generalizable animation mechanism is `PaintContext::repaint_after`/`repaint_at` with earliest-deadline-wins coalescing ([`crates/warpui_core/src/presenter.rs:604-629 @ fbe26f9`](https://github.com/warpdotdev/warp/blob/fbe26f99c2b508c2e450db4ceccfbdb2a0c8766e/crates/warpui_core/src/presenter.rs#L604-L629)) plus the `LiveElement` wrapper ([`crates/warpui_core/src/elements/gui/live.rs @ fbe26f9`](https://github.com/warpdotdev/warp/blob/fbe26f99c2b508c2e450db4ceccfbdb2a0c8766e/crates/warpui_core/src/elements/gui/live.rs)); `ShimmeringTextElement` self-schedules the same way.
- Every TUI draw re-executes `TuiElement::render` over the retained element tree into a fresh buffer ([`TuiPresenter`, `crates/warpui_core/src/presenter/tui.rs:133-251 @ fbe26f9`](https://github.com/warpdotdev/warp/blob/fbe26f99c2b508c2e450db4ceccfbdb2a0c8766e/crates/warpui_core/src/presenter/tui.rs#L133-L251)), so elements that derive visuals from the wall clock inside `render()` animate correctly on cached-element repaints, with no view re-render or relayout.
- `BlocklistAIHistoryModel` emits `UpdatedConversationStatus` on every conversation status change and `AppendedExchange` when a request starts, each carrying the terminal surface id — the signals the session view needs to re-render the indicator in and out and re-anchor its counter.
- Theme/styles: the TUI is pinned to `dark_theme()` ([`crates/warp_tui/src/session.rs:64-66 @ fbe26f9`](https://github.com/warpdotdev/warp/blob/fbe26f99c2b508c2e450db4ceccfbdb2a0c8766e/crates/warp_tui/src/session.rs#L64-L66)); styles come from `TuiUiBuilder` ([`crates/warp_tui/src/tui_builder.rs @ fbe26f9`](https://github.com/warpdotdev/warp/blob/fbe26f99c2b508c2e450db4ceccfbdb2a0c8766e/crates/warp_tui/src/tui_builder.rs)), which reads `terminal_colors().normal/bright`.

## Proposed changes

### 1. Shared shimmer math in `warpui_core`

- New feature-independent module `crates/warpui_core/src/elements/shimmer_math.rs`: move `ShimmerConfig` here plus free functions for the band center (glyph count + elapsed time + config), per-index intensity, and the base→peak `ColorU` lerp, operating on plain `usize`/`f32` (no `GlyphIndex`).
- Refactor `ShimmeringTextElement` to call the shared functions (mapping its `GlyphIndex` at the call site) and re-export `ShimmerConfig` from its current path so `warpui::elements::shimmering_text::ShimmerConfig` consumers (e.g. `app/src/ai/loading/shimmering_warp_loading_text.rs`) are unaffected.

### 2. Generalizable TUI repaint scheduling (framework, mirrors the GUI)

- Paint-context plumbing: give the TUI paint pass `repaint_after(Duration)` with earliest-deadline-wins coalescing, mirroring the GUI `PaintContext`. Preferred shape: a dedicated `TuiPaintContext` (wrapping `rendered_views` + `repaint_at: Option<Instant>`) passed to `TuiElement::render`, mirroring the GUI's LayoutContext/PaintContext split; this is mechanical signature churn across `crates/warpui_core/src/elements/tui/*`. (Lighter alternative if churn is unwanted: add the accumulator to the existing `TuiLayoutContext`, which `render` already receives.) Requested delays are floored (`MIN_REPAINT_DELAY`, 10ms) so a misbehaving element can't busy-loop the scheduler. The shared `use_view` move-out/move-in pattern lives on one `TuiViewMapContext` trait implemented by the layout/paint/presentation contexts.
- Surface the deadline: `TuiPresenter::paint` collects the winning deadline into `TuiFrame` (new `repaint_at: Option<Instant>` field). Elements speak relative (`repaint_after`); the frame/driver speak absolute (`repaint_at`), so post-paint scheduling doesn't drift by paint duration.
- Drive it: in `spawn_tui_driver`, after each draw, if the frame carries a deadline, schedule/replace a single one-shot foreground timer that calls `screen.draw(ctx)` at that instant (each draw paints the full tree, so its deadline is authoritative and replacing cancels the stale timer). This is a paint-only redraw over retained elements — no view invalidation, no transcript relayout; `TuiPresenter` tracks an explicit "invalidate ran this frame" flag so such redraws reuse the cached root element even for views with no child views. Self-sustaining like the GUI: each draw reschedules only if some element requested another repaint, so the TUI is fully idle when nothing animates.
- `TuiAnimated` (`crates/warpui_core/src/elements/tui/animated.rs`): the reusable animation element — `TuiAnimated::new(repaint_interval, build_frame)` requests `repaint_after(repaint_interval)` on every render and re-invokes `build_frame: Fn() -> Box<dyn TuiElement>` at the start of every layout pass, delegating render/cursor/present/event dispatch to the built child. Animation on a cached element tree needs both halves and neither is useful alone: timed repaints re-run layout+paint over the *cached* element trees without re-rendering any view (so a plain `TuiText` would repaint the string it was built with at view-render time forever), and rebuilt content without repaints never gets a pass to rebuild in — hence one fused element rather than separate scheduling/content wrappers (the same shape as SwiftUI's `TimelineView`). Elements whose *paint output* is clock-derived but whose content is fixed (the shimmer) skip it and call `repaint_after` directly. Multiple animated elements coalesce to the earliest deadline per frame.
  - Why not tick `ctx.notify()` instead: notify is view-granular. It dirties only the notifying view (`crates/warpui_core/src/core/view/context.rs:445` → `notify_view_observers`, `crates/warpui_core/src/core/app.rs:4087`), and the presenter then re-runs that view's whole `render()` down to its `TuiChildView` boundaries, through the effect queue. At the indicator's 20Hz cadence that would rebuild the session view's entire element tree and re-read its models twenty times a second when only one glyph and one counter string changed. `TuiAnimated` scopes recomputation to exactly the clock-dependent leaves, inside the existing paint-driven pass, with no view invalidation. (This is also where the TUI intentionally diverges from the GUI, which ticks time-varying *text* via view-level notify timers — e.g. the summarization elapsed counter, `app/src/ai/blocklist/block/view_impl/common.rs:305`, driven by 1s notify loops like `app/src/ai/blocklist/block/status_bar.rs:686` — and uses its repaint-only `LiveElement` wrapper for content kept fresh by those re-renders; only paint-time *visuals* like the GUI shimmer are clock-derived. A GUI `Animated` twin would slot into `elements/gui/` if ever needed.)
- Keyframe timing (`crates/warpui_core/src/elements/animation.rs`, feature-independent alongside `shimmer_math`): `Keyframe<T>` (constructed via `Keyframe::new(value, hold)` or `Keyframe::from_millis(value, hold_ms)`) and `KeyframeTimeline<T>`, a looping timeline that precomputes cumulative end offsets at construction and answers `value_at(elapsed)` by binary search. The generic home for discrete frame-stepped choreography (the spinner today).
- `AnimationClock` (same module): the monotonic-safe anchor for animations whose starting offset comes from the wall clock — it captures the initial elapsed `Duration` plus a build-time `Instant` and reports `elapsed()` as their sum. Never reconstruct an anchor as `Instant::now() - elapsed`: that subtraction panics when the wall-clock offset exceeds the monotonic clock's range (suspend, clock skew, stale exchanges).
- There must be exactly one native animation mechanism: no per-view tick timers. (A prior TUI spinner effort showed a manual per-view tick forcing full transcript relayout alongside native repaint scheduling causes redundant redraws and jank.)

### 3. New TUI shimmering-text element

- `TuiShimmeringText` in `crates/warpui_core/src/elements/tui/` (gated by the existing `tui` feature, alongside `text.rs`): renders a single-line string writing one cell per char with `fg: Color::Rgb(lerp(base, peak, intensity))` plus optional `Modifier::BOLD`.
- Colors are computed inside `render()` from an `AnimationClock` passed at construction and a `ShimmerConfig`, and the element calls `repaint_after` each render — mirroring the GUI `ShimmeringTextElement`, minus the glyph-layout state handle (TUI cells are 1:1 with chars).

### 4. Warping indicator row in `warp_tui`

- `crates/warp_tui/src/tui_builder.rs`: semantic style additions — `warping_base_color` / `warping_spinner_style` (`terminal_colors().bright.magenta`) and `warping_shimmer_color` (a theme text color selected for contrast against the resolved terminal background); the timer suffix reuses `muted_text_style`.
- New module `crates/warp_tui/src/warping_indicator.rs`: `render_warping_indicator(elapsed, app)` composes a `TuiFlex::row()`: spinner, a space, `TuiShimmeringText::new("Warping...")` (bold, bright-magenta→contrast-aware theme text), a space, and the `(Ns)` counter. One `AnimationClock::starting_at(elapsed)` drives all three parts so they stay phase-locked. The spinner and counter are `TuiAnimated` leaves, each rebuilding a styled single-row `TuiText` on its own cadence (the spinner reads `SPINNER_TIMELINE.value_at(clock.elapsed())` at 80ms; the counter formats elapsed seconds at 1s); the shimmer label self-schedules, so the row needs no wrapper.
- `app/src/ai/agent/mod.rs`: `AIAgentExchange::time_since_start()` — the wall-clock time since `start_time`. Lives on the exchange because `warp_tui` has no chrono dependency and `AIAgentExchange`'s fields aren't re-exported through `tui_export`.
- `crates/warp_tui/src/terminal_session_view.rs`: `render` inserts the indicator into the root column between the transcript `flex_child` and the input box — wrapped in a `TuiContainer` with one row of top padding — while the selected conversation's `status().is_in_progress()` (matching the GUI's condition; resolved via `ConversationSelection::selected_conversation_id` → `BlocklistAIHistoryModel::conversation`). The `(Ns)` counter anchors to the latest exchange's `time_since_start()`, so animation survives element-tree rebuilds and the indicator stays up across tool-call exchanges within a turn. Appearance/removal is re-render driven: the view subscribes to `BlocklistAIHistoryModel` (notify on `UpdatedConversationStatus` / `AppendedExchange` for this surface) and to the conversation-selection model (notify on any selection change).
- `TuiAIBlock` and `TuiTranscriptView` need no changes: the indicator is not a block section.

### 5. Constants

The spinner choreography (`SPINNER_TIMELINE`, a `KeyframeTimeline<&str>` static built from `Keyframe::from_millis` entries: 200ms per 45° step for the two half-turns, a 200ms rest at vertical, and 80ms per step for the 540° fast spin; 2.68s total loop) lives in `warping_indicator.rs` with doc comments; the spinner's repaint interval (80ms, matching the timeline's shortest hold) and the counter's (1s) are inlined at their `TuiAnimated::new` calls. The shimmer uses `ShimmerConfig::default()` (3s period, radius 6, padding 8) and its own 100ms repaint constant inside `TuiShimmeringText`. The Figma prototype's frame timings aren't readable via the MCP API, so these are tunable defaults to adjust by eye. (The GUI shimmer repaints at 32ms/30fps; a TUI band over a 10-char string moves ~1 cell per 120ms at the 3s period, so ~100ms repaints are visually equivalent and cheaper.)

Open question: elapsed format for long turns — the design shows `(1s)`; the proposal keeps plain seconds (`(94s)`) rather than `1m 34s`.

## Testing and validation

- Build/lint: `cargo check -p warp_tui -p warpui_core`, then `./script/format` and the presubmit clippy invocation.
- Unit tests (`*_tests.rs` convention):
  - Shimmer math (`crates/warpui_core/src/elements/shimmer_math_tests.rs`): band center sweep/loop, single-glyph degenerate case, intensity peak/radius falloff, color-lerp endpoints.
  - Spinner frame selection from elapsed time, a rendered `⋮ Warping... (0s)` row that requests a repaint, and shimmer isolation from the spinner (`crates/warp_tui/src/warping_indicator_tests.rs`).
  - Keyframe timeline hold/loop/zero-hold/empty behavior (`crates/warpui_core/src/elements/animation_tests.rs`); `TuiAnimated` rebuilds its frame on every layout pass, requests a repaint every render, and paints nothing before its first layout (`crates/warpui_core/src/elements/tui/animated_tests.rs`).
  - `TuiShimmeringText` cell colors: all-base before the band reaches the text, peak at mid-band, bold modifier, repaint request (`crates/warpui_core/src/elements/tui/shimmering_text_tests.rs`).
  - Repaint plumbing: static frames carry no deadline, `TuiFrame` carries the earliest requested deadline, `TuiAnimated` requests one every paint, and a paint-only repaint reuses the cached root without re-rendering a child-view-less view (`crates/warpui_core/src/presenter/tui_tests.rs`); `AnimationClock` preserves an initial offset far beyond process uptime without panicking (`crates/warpui_core/src/elements/animation_tests.rs`).
- Manual: `script/run-tui`, run a prompt, verify spinner rotation, shimmer sweep, ticking `(Ns)`, spacing/colors against the Figma node, the row disappearing when the turn completes, and that an idle TUI schedules no repaints (no busy redraw loop).

## Parallelization

Not proposed. The work is a single dependency chain (shimmer math → repaint plumbing/`TuiShimmeringText` → `warp_tui` integration) inside two crates with shared signatures (`TuiElement::render` churn touches everything the later steps build on); splitting it across agents would serialize on merge conflicts rather than save wall-clock time. Steps 1 and 2 are mutually independent and could be done in either order, but each is small.

## Risks and mitigations

- `TuiElement::render` signature change touches every TUI element and its tests: purely mechanical; do it as its own commit so review is trivial.
- Runaway repaint loops (an element always requesting repaints): the indicator only renders while the conversation is in progress, and the driver schedules exactly one coalesced timer per frame; the manual idle check above guards regressions.
- Elapsed-counter width changes (`(9s)` → `(10s)`) shift row layout by one cell mid-animation: acceptable per design (counter is the row's last element, nothing to its right).
