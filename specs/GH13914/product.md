# Product Spec: Inline code chip background extends past glyph ink on the right

**Issue:** [warpdotdev/warp#13914](https://github.com/warpdotdev/warp/issues/13914)
**Figma:** none provided

## Summary

Inline code chips (the pill-shaped background behind `` `code` `` spans in Markdown-rendered text) paint their background rectangle using each glyph's advance width, not its visual ink width. On the right edge of the chip, this leaves visible empty background color between the last glyph's ink and the chip's right border/edge — the chip looks asymmetric, with tighter padding on the left than on the right.

The fix stops deriving the chip's horizontal edges from advance width. Instead it anchors them to the first and last visible glyph's true **ink** extents, plus an **equal** fixed design padding on each side, so the chip has literally equal ink-derived padding left and right. Interior glyph positions remain advance-based (that is the text itself, unchanged). This is a paint-time-only change to the background/border rectangle; it does not affect text selection, caret placement, copy/paste, or the underlying Markdown content.

This ink-edge geometry is scoped to the **chip family**: inline-code chips today, and kbd keycaps (`` <kbd> ``, PR [#13916](https://github.com/warpdotdev/warp/pull/13916), pending) once they land, since kbd renders by reusing the inline-code chip's styling. kbd keycaps are in fact where the left/right asymmetry was first noticed. All other backgrounded/bordered text (search-match highlights, background-only inline code, any other `TextStyle::border`/`TextStyle::background_color` consumer) is explicitly out of scope and keeps its existing geometry unchanged. See `tech.md` for the exact chip-family predicate.

## Problem

When Warp renders Markdown inline code (for example, in agent conversation panes, README/chat message rendering, or any rich text using the `warpui_core` text layout system), the background/border decoration behind the code span is computed from each glyph's *advance width* — the distance the text cursor moves after drawing the glyph, which is the sum of the glyph's left-side bearing, its ink, and its right-side bearing. Advance width is a spacing metric for positioning the *next* glyph, deliberately wider than a glyph's visible ink so adjacent glyphs don't collide; it is not a visual bounding box.

Because the chip's left edge is anchored at the first glyph's origin (advance origin, roughly one left-side bearing to the left of the ink) while the right edge is anchored at the last glyph's advance endpoint (roughly one right-side bearing past the ink), the chip is not visually balanced: the gap between the last character's ink and the chip's right edge is consistently larger than the gap between the chip's left edge and the first character's ink. This is most noticeable with narrow-ink final characters (e.g., `code)`, `foo;`, or any run ending near a period, comma, or narrow letter) where the extra advance spacing after the glyph becomes visible whitespace inside the colored chip.

This is purely a rendering/paint issue — it does not affect text selection, caret placement, copy/paste, or the underlying Markdown content, all of which continue to use the existing glyph geometry unchanged.

## Goals

- The chip's background/border has **equal** ink-derived design padding on the left and right — literally the same gap between the code's ink and the chip edge on both sides. This is the core promise: the advance-vs-ink asymmetry is eliminated, and (unlike an earlier framing of this fix) the residual left-only `block_padding` skew is also removed *for chips*, so "equal padding" is achieved outright, not merely approximated.
- As a deliberate consequence of achieving equal padding, the chip's left edge **tightens** versus today, because the legacy left-only `block_padding` shift (`2 × block_padding`, `= font_size / 5`, ≈2.6px at a 13px font as a *typical* magnitude — clearly visible, not sub-pixel) is compensated out for chips; the exact pixel shift also depends on the switch from advance- to ink-anchoring, so it is glyph/font-dependent rather than a fixed delta. This is intended behavior, not preservation: the chip reads as noticeably snugger on both the left and the right, with both gaps equal. Non-chip bordered/backgrounded runs keep the legacy left shift and do not move.
- The ink-edge geometry is scoped to the **chip family** (inline-code chips today, kbd keycaps once PR #13916 lands) via an explicit, named predicate in the paint path (see `tech.md`). It is not applied to any other `TextStyle::border`/`TextStyle::background_color` consumer.
- The fix applies uniformly regardless of platform (macOS/CoreText or Linux-winit/cosmic-text text layout backends), because it sources per-glyph ink from a single measurement that both backends already implement with matching semantics (see `tech.md`).

## Non-goals

- Changing the border radius, border width, or color of inline code chips (only the horizontal positioning of the existing background/border rectangle is affected). Note: the horizontal application of `block_padding` *is* changed for chips — its legacy left-only shift is compensated out so left/right padding is equal (see Goals and `tech.md`); its vertical (top/bottom) insets are unchanged, and its behavior for non-chip runs is untouched.
- Changing font metrics, hinting, ligature shaping, glyph draw positions, or caret geometry — the ink measurement is read *only* to position the background/border rectangle; nothing about how text is laid out or where the caret lands changes.
- Any change to the terminal UI (TUI) Markdown renderer (`crates/warp_tui/src/tui_markdown.rs`). The TUI renders inline code with a foreground-color-only style patch (no `background_color`/border) over `ratatui`'s per-cell grid, which has no advance-vs-ink distinction to begin with — this issue does not reproduce there and no change is proposed for it (confirmed in `tech.md`).
- Fixing analogous background/border asymmetry for backgrounded/bordered text **outside the chip family** (search-match highlights, background-only inline code, any other `TextStyle::border`/`TextStyle::background_color` consumer). The ink-edge geometry is intentionally gated on the chip-family predicate; all such non-chip runs keep their existing advance-based geometry, including the legacy `block_padding` left-only shift, unchanged. See `tech.md` for the predicate and the confirmed blast radius.
- Adding a *design-space* per-glyph ink API (e.g. a new CoreText `CTFontGetBoundingRectsForGlyphs` binding). The fix uses the already-plumbed rasterized-ink measurement; a design-space alternative is noted in `tech.md` as a possible future refinement, not adopted here.

## User experience

### Current behavior (broken)

1. Warp renders a chat/agent message (or any rich-text surface) containing inline code, e.g. `` This is `code`. ``.
2. The code chip's background is visibly wider on the right of the last glyph (`e`) than the gap on the left before the first glyph (`c`).
3. This asymmetry is most visible with narrow trailing glyphs and is a subtle but persistent visual defect across every inline code span in every rendered Markdown surface.

### Expected behavior (after fix)

1. The same message renders with the code chip's background hugging the code's ink, with a fixed, **equal** design padding on the left and right — the two gaps are literally the same size.
2. The chip's left edge **tightens** versus today — the legacy left-only `block_padding` residual (`2 × block_padding`, `= font_size / 5`, ≈2.6px at a 13px font as a *typical* magnitude) is compensated out for chips to make the padding equal, and the edge switches from advance- to ink-anchored, so the exact pixel shift is glyph/font-dependent. Code spans at the start of a line, or immediately following another inline element, sit noticeably snugger on the left. This is a clearly visible, intended shift, not a preserved position.
3. Because each chip's edges are derived from its own glyphs' ink, chips are independently correct — there is no cross-chip "sawtooth." (An earlier candidate approach shifted the whole rect by a fraction of advance and accepted a subtle sawtooth in vertically-stacked chips; the ink-edge approach makes that trade-off moot because each chip gets its own exact correction.)
4. The extra space the fix "gives back" on the right was, and remains, the space between two words — e.g. in `This is `code``, the visual glyph gap between "is" and "`code`" is unchanged; only the region of that gap that renders as chip-background color moves.

### Edge cases

- **Empty code span (`` `` ``):** no glyphs to measure; the chip should not attempt to paint a malformed or negative-width background. Existing empty-run handling (no first glyph → no background painted) is preserved.
- **Zero-ink edge glyph (e.g. a trailing space inside a code span):** a glyph with no ink has no ink edge to snap to. The affected edge (left or right) falls back to that glyph's advance-based edge, offset outward by the same fixed design padding used everywhere else (`advance-edge ± pad`), so the chip never collapses to zero or negative width on whitespace-terminated spans and still gets its normal padding. See `tech.md` for the exact fallback rule.
- **Ink overshooting the advance (e.g. an italic `f` whose ink extends past its advance box):** ink-based edges contain such overshoot naturally. Note that today's advance-based right edge can actually *clip* such overshoot; the ink-edge fix removes that incidental clipping. This is a small, correct-direction side effect, not a regression.
- **Ligatures:** a run with a shaped ligature (e.g. a code font ligature for `->` or `!=`) has one glyph covering multiple characters; its ink is measured the same as any other glyph's — no special-casing is introduced or required.
- **Right-to-left (RTL) text inside a code span:** the fix must not assume left-to-right glyph ordering when determining which edge is "leading" vs "trailing." The visually-leftmost and visually-rightmost glyphs are identified order-independently (see `tech.md`), and their ink is what drives the left/right edges.
- **Emoji or wide glyphs inside code spans:** wide glyphs typically have less disproportionate advance-vs-ink padding relative to their size, so the visual effect is smaller, but ink measurement applies uniformly; no separate threshold or cutoff is introduced.
- **Truncated/ellipsized code chip (chip cut off by line-width clipping):** the existing `visible_left`/`visible_right` clamping to the drawn glyph span (added for a prior fix, see `test_run_background_clamped_to_visible_glyph_span`) continues to apply; the ink edges are derived from the *visible* first/last glyphs, and the existing intersection clamp still bounds the painted rect to `visible_bounds`.
- **Border-only or background-only run (not a chip):** the chip-family predicate requires **both** a border and a background color (how the inline-code chip, and future kbd keycaps, are constructed). A run with only one of the two is not a chip and keeps the legacy advance-based geometry — the ink-edge treatment does not apply. This is intentional: it prevents the fix from silently altering search highlights (background-only) or any hypothetical border-only style. See `tech.md` for the predicate.

## Success criteria

1. For any inline code span, the left ink gap and the right ink gap are **equal** — both are exactly the fixed design padding (`ink_left − pad` on the left, `ink_right + pad` on the right, with the legacy left-only `block_padding` shift compensated out for chips). This is a geometry-level equality, verified by a unit test asserting `left_gap == right_gap == pad` for a chip run, not merely a visual approximation. The multi-pixel advance-vs-ink gap this issue targets is eliminated, and no `block_padding` residual survives on chips.
2. As the deliberate consequence of criterion 1, the chip's left edge **moves inward** versus today's rendering: the legacy left-only `block_padding` residual (`2 × block_padding`, `= font_size / 5`, ≈2.6px at a 13px font — the typical magnitude of what's removed, not a fixed delta) no longer appears on the left edge, and the edge itself is now ink-anchored rather than advance-anchored, so the exact pixel shift is glyph/font-dependent. The verifiable, geometry-level property is: (a) the rendered chip has **zero** `block_padding` residual on the left (`horizontal_block_padding == 0` for chips), and (b) the left edge is derived from `first_ink_left − pad`, not from the legacy `visible_left − 2×block_padding` formula — both verified by a unit test asserting a chip run's rect uses the ink-edge-minus-pad formula with no block_padding term. This is the intended behavior, not a preserved position. Non-chip bordered/backgrounded runs keep the legacy left shift and their left edge does not move (verified by a test asserting a non-chip run retains the legacy advance-based geometry).
3. No regression to caret placement, text selection, or copy/paste behavior for text inside or adjacent to inline code spans (these are driven by `caret_positions`/glyph `position_along_baseline`, which this fix does not touch).
4. No change to the TUI Markdown rendering of inline code (confirmed unaffected, per Non-goals).
5. Existing test `test_run_background_clamped_to_visible_glyph_span` in the `warpui_core` text-layout tests continues to pass (updated to use chip-family styling and reflect the new ink-edge geometry, without weakening its truncation-clamping assertion).
6. A zero-ink trailing glyph (e.g. a code span ending in a space) still paints a well-formed chip via the advance-edge fallback (verified by a unit test).
7. The chip-family predicate itself has unit coverage: `TextStyle::is_chip` returns true only for a run with both a border and a background color, and false for border-only, background-only, and plain runs (verified by `test_is_chip_predicate`).

## Open questions

1. **Padding value (design/taste, not correctness).** The chip's per-side padding is a fixed fraction of the font size — currently `font_size / 12` — applied **equally** to both ink edges. It is purely a design value ("how much air around the code reads best"), no longer doing any left-edge-preservation duty: since the equal-padding fix compensates out the legacy `block_padding` left shift for chips, the constant is not calibrated to land near any old advance-based edge. This is a single, well-commented, easily-tunable constant; the exact fraction merits a visual sign-off. Mis-tuning it only changes the uniform padding — it can never re-introduce the left/right asymmetry, because the ink edges themselves are exact (see `tech.md`).
2. **macOS right-edge AA residual (visual sign-off).** The rasterized-ink measurement on macOS includes a deliberate ~1px anti-aliasing fudge on the right that the Linux backend does not (see `tech.md`). This makes the right ink edge read up to ~1 device pixel wider on macOS than on Linux for identical text — far smaller than the advance gap being removed, but worth a cross-platform visual check before merge.
3. **Scope: other `TextStyle::border`/`background_color` consumers — RESOLVED.** The ink-edge treatment is scoped to the **chip family** (inline-code chips today, kbd keycaps once PR #13916 lands) via an explicit, named predicate (`TextStyle::is_chip` — a run carrying both a border and a background color, which is how chips are constructed and how kbd will reuse the path). It does **not** apply to any other backgrounded/bordered run: search-match highlights, background-only inline code, and any border-only style keep their existing advance-based geometry. See `tech.md` for the predicate and where it gates the paint path.
4. **Pre-existing `block_padding` left-only offset — RESOLVED for chips (found while speccing this fix).** The existing `block_padding` term only offsets the rect's left edge (it algebraically cancels out of the right edge; see `tech.md`), which by itself would leave a `2 × block_padding` (`= font_size / 5`, ≈2.6px at a 13px font) left/right imbalance — clearly visible, not sub-pixel. The equal-padding fix now **compensates this out for chip-family runs**, so chips have literally equal ink padding on both sides. The legacy left-only shift is deliberately left untouched for every non-chip bordered/backgrounded run (changing it globally would alter the visual footprint of unrelated runs); scoping the compensation to chips via the predicate keeps the blast radius bounded.
