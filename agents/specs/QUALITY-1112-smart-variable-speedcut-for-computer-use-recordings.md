*Spec: Smart variable-speed/cut for computer-use video recordings (QUALITY-1112)*

== PRODUCT ==
*Summary:* On Linux, published computer-use recordings are captured at 1x, retain the moments while the client is carrying out meaningful agent work at 1x, and remove blocked/thinking gaps with no on-screen delta. The transformation is client-side in the existing post-stop finalization path; no server-side segmentation and no player-side post-processing are introduced.

*Key design choices:* (1) record the start and finish of each successful `UseComputer` action group in the client, (2) expand each real action window by deterministic asymmetric margins (250 ms before, 1000 ms after) and linger overlay pills across the post-action margin so they stay readable, (3) cut with one post-stop ffmpeg pass and remap ASS overlay timestamps through the same source-to-output map, and (4) treat a recording with no committed action group as an error rather than publishing an unexplained artifact, except that conversation cancellation preserves its `Cancelled` result while always discarding the capture without upload.

*Plain-language cut explanation:* The recorder first makes a normal 1x master. After capture stops, ffmpeg reads that master and `trim` selects only the time ranges that contain action work (like cutting unwanted strips out of film). Each selected strip gets `setpts=PTS-STARTPTS`, which relabels its first frame as time zero so its old gap timestamp is not carried forward. `concat` then glues the strips end-to-end in source order. It does not wait 1.5 seconds, invent frames, or speed up action frames; it simply removes the unselected source frames and creates a continuous output timeline. The ASS overlay is rendered after this map is known, using the same source-to-output conversion, so pills move with their actions.

*Behavior* (numbered, testable invariants from the user's/consumer's view):
1. A Linux recording with at least one successfully completed meaningful `UseComputer` call contains each action group at approximately real time (1x). A group starts at the wall-clock offset when the client begins executing that `UseComputer` call and ends at the wall-clock offset immediately after its complete action sequence (and any requested post-action screenshot) finishes. There is no artificial 1.5 s wait.
2. `source_duration` means the actual 1x source video duration available at finalization: the elapsed capture duration from `RecordingHandle` start until explicit stop, auto-stop, or ffmpeg early exit, clamped to the finalized media timeline. It is not the configured `max_duration`, except when that limit is what stopped capture, and it is not a delay added after an action. Action start/finish offsets are clamped to this duration.
3. A qualifying action group is a successful `UseComputer` dispatch containing at least one real interaction (keyboard, typing, pointer, scroll, or other non-wait action). A dispatch containing only `Wait`/zero-duration no-op actions is not committed to the recording timeline. Pointer-only groups still commit with an empty label list so their visual effects are retained even though no pill is rendered. If a mixed batch contains waits and real interactions, the entire client-executed batch is one contiguous group from its start to its finish; the explicit wait is not split into an inferred gap.
4. A single `UseComputer` call may contain multiple sub-actions (for example click → type → scroll). The client executes them sequentially, records one group start before the sequence and one group finish after the sequence, and keeps all of that contiguous group at 1x. Renderable labels remain one row in the existing action order. Instrumenting every low-level sub-action separately is not required for this version because it would split an API-level action group and could introduce cuts between sub-actions; a future per-sub-action timeline can build on the same start/finish fields.
5. The action window that drives the cut is `[start_offset, finish_offset]`, with a minimum of one source frame when start and finish fall in the same frame. This window expands by 250 ms before its start and 1000 ms after its finish, then clamps to `[0, source_duration]`. The margins are asymmetric because the pre-action frames are mostly the removed thinking/blocked gap while the post-action frames show the action's on-screen effect (and keep its overlay pill readable). Neither margin is a transition and neither changes playback rate.
6. Group intervals are sorted by source start, and duplicate, adjacent, or overlapping expanded windows are merged into one contiguous source segment. Every group represented in a merged segment remains eligible for overlay remapping; no group is dropped because it has no labels.
7. Source gaps between merged segments are removed entirely. Each retained segment is reset to a zero-based PTS and concatenated in source order, producing an output timeline that starts at 0, advances monotonically, has no black frames or timestamp gaps, and preserves source frame cadence and playback rate inside each segment.
8. Every rendered pill remains aligned with its source action after cuts and stays visible long enough to read. An overlay event uses the group's *display* interval — the action window's start, lingering the full post-action margin (1000 ms) past `finish_offset` so a pill never extends past kept frames, and bounded by the next group's start so pills never overlap — which is distinct from the `[start_offset, finish_offset]` action window that drives the cut. The display interval is remapped through the source-to-output segment map before ASS generation. Events wholly in removed gaps are omitted; events touching a cut boundary are clamped to the retained boundary. The remap is done at `Duration` precision before ASS's centisecond formatting.
9. A recording with no committed meaningful action group is an explicit finalization error (`StopRecordingResult::Error`) for normal finalization reasons, is not uploaded as a video artifact, and has its temporary capture files cleaned up. This includes recordings containing no `UseComputer` calls and recordings containing only waits/no-ops. Conversation cancellation is the deliberate exception: it returns the existing `StopRecordingResult::Cancelled` result and discards the capture before the empty-action check, so it also never uploads a video. This policy is bundled in this PR because the finalization function already owns the action list and upload decision; it prevents the new cut feature from silently publishing an artifact with no action timeline.
10. If a recording has committed actions but segment computation or ffmpeg cut/overlay processing fails, the existing best-effort behavior remains: upload the untouched 1x source rather than a partial/corrupt artifact. Existing cleanup and terminal finalization behavior remains intact.
11. The Linux recorder no longer applies the server/default `playback_speed_multiplier` to the live capture or smart-cut output. The request field remains accepted for wire compatibility. On macOS, where `burn_in_action_log` is currently a no-op and capture is avfoundation, the existing uniform `setpts` behavior remains the documented fallback; no macOS cut implementation is part of this change. Other unsupported/no-op platforms remain unchanged.
12. `max_duration` remains an input wall-clock bound (`-t` before `-i`) and `max_size_bytes` remains the hard capture-file bound (`-fs`). Both apply to the new 1x Linux master, which can be larger than the old uniformly sped-up output and may therefore reach the size cap sooner in wall-clock time. No multiplier-based scaling of `-fs` or server policy change is introduced here; a size-limited capture still finalizes and cuts the prefix that exists.
13. Current capture metadata/API semantics are preserved: `RecordingOutput`/`StopRecordingResult` continue to report the recorder's wall-clock capture duration and capture-file size as they do today, while the uploaded video stream has the compact duration equal to the sum of retained segments. The no-audio capture contract remains video-only (`concat` uses `a=0`); adding audio is out of scope.

*Design alternatives*:
- *Live uniform speedup:* rejected because it speeds up the very action sections that must remain readable, cannot remove only blocked gaps, and leaves source-time overlay events tied to the wrong playback clock.
- *Player/server post-processing:* rejected by the explicit client-side decision and because the artifact must be correct when downloaded or viewed without a specialized player; server code must not compute segments.
- *Physical per-segment files followed by an external concat:* rejected as extra temporary-file lifecycle and upload complexity. One post-stop `filter_complex` pass is selected.
- *What ffmpeg does:* `trim` selects source frame ranges; `setpts=PTS-STARTPTS` restarts timestamps for each selected strip; `concat` appends the strips. This is selected over a global speed filter because it removes only dead source frames and preserves 1x action frames.
- *Dispatch-only fixed 1.5 s window:* rejected as the primary action end because it cannot represent a long or multi-sub-action client execution and makes the reader think the client waits 1.5 s. Finish time is recorded when the complete `UseComputer` sequence returns. A one-frame minimum handles an instantaneous call; the surrounding margins (and the overlay's post-finish linger) provide the visible context that the old fixed duration used to provide.
- *Per-low-level-sub-action finish timestamps:* evaluated but not selected for this PR. The actor's API already executes a `UseComputer` batch sequentially and existing overlays intentionally group that batch into one row; splitting it would require a new callback/event contract through every platform actor and could cut between sub-actions. One start/finish pair per successful API call captures the entire sequence without guessing internal boundaries.
- *No-action no-op:* rejected because it silently publishes an artifact with no committed action evidence. Return an explicit error and clean up instead for normal finalization; preserve the existing cancellation contract as a deliberate exception because cancellation already drops the handle and guarantees no upload. This is small enough to bundle because `finalize_recording` already owns the action list and upload decision.
- *Symmetric 500 ms margins:* the original cut used 500 ms on both sides, but once dead gaps were removed that left overlay pills flickering for roughly one frame on instantaneous actions. Replaced with asymmetric margins — 250 ms pre (short, since pre-action frames are mostly removed dead time) and 1000 ms post (retains the action's effect and gives the pill a readable window) — clamped, then merged.
- *Segment only labeled actions:* rejected because clicks, mouse movement, and drags can change the screen while intentionally producing no pill. The timeline records meaningful no-label interactions, while ASS generation skips empty label lists.
- *Scale `-fs` by the old multiplier:* rejected because it weakens the configured hard disk bound and makes resource use depend on an optional server hint. The existing capture bound remains authoritative; server tuning can follow separately.
- *Deprecate `playback_speed_multiplier` immediately:* rejected because old servers and macOS still send/use it. Retain the field and document Linux smart-cut precedence; do not add a new server field in this PR.

*Open questions resolved:*
- *Source duration:* the actual 1x capture elapsed duration available at finalization, clamped to the finalized media timeline; it is not a post-action delay and not automatically the configured maximum.
- *Action start/finish:* one start/finish pair per successful `UseComputer` API call. Start is captured immediately before client execution; finish is captured immediately after the complete sequential batch (including its explicit waits and requested post-action screenshot) returns. There is no artificial 1.5 s wait.
- *Multiple sub-actions:* one call-level contiguous group; labels remain a single ordered row. Low-level per-action timing is explicitly evaluated and deferred because it needs a cross-platform actor callback contract.
- *Margin:* asymmetric per finish-based group — 250 ms before the start and 1000 ms after the finish; the overlay pill lingers the full post-action margin past `finish_offset` so it stays readable. No server-configurable margin is added.
- *Grouping:* sort by start, clamp to source duration, enforce one frame for equal-frame groups, expand/merge overlapping or touching intervals, and map the merged segments in source order.
- *No-action policy:* explicit finalization error with no upload for normal finalization reasons, bundled in this PR because the existing finalize function already owns the action list and upload decision. Conversation cancellation intentionally returns `Cancelled` before that guard and drops the handle, which also guarantees no upload; it is not a follow-on.
- *Playback multiplier:* retained for request compatibility and macOS fallback; ignored for Linux smart capture/cut. Processing failure with committed actions uploads the unchanged 1x source; an empty action list errors instead.
- *Platform:* Linux x11grab plus post-stop ASS/cut is the supported implementation. macOS remains uniform-speed avfoundation because the shared burn-in API is explicitly a no-op off Linux; Windows/no-op backends are unchanged.
- *Audio:* current capture has no audio input, so the graph is video-only and uses `concat ... a=0`; no audio synchronization policy is needed.
- *Metadata:* preserve current capture-time `duration` and capture-file `size_bytes` fields rather than changing the wire/API contract in this feature.
