# Spec: Make NLD opt-in with toggle slash commands

Linear: [CODE-1893](https://linear.app/warpdotdev/issue/CODE-1893/make-nld-opt-in-toggleable-with-slash-command)
Target repo: `warpdotdev/warp` (base `master`)
Requester: kevin (kevin@warp.dev)

== PRODUCT ==

*Summary:* Natural language detection (NLD) in the TUI is currently opt-out: the shared setting `ai_autodetection_enabled_internal` (`AIAutoDetectionEnabled`) defaults to `true`, so the TUI auto-detects natural-language input and routes it to the agent input by default. This change flips NLD to opt-in (default `off`) and exposes two explicit TUI slash commands — `/enable-natural-language-detection` and `/disable-natural-language-detection` — so a user can turn NLD on or off from the TUI input, since the TUI has no AI settings page for this toggle.

*Key design choices:*
1. **Flip the shared setting's default to `false` (global opt-in).** `ai_autodetection_enabled_internal` is `surface: SettingSurfaces::ALL` and `sync_to_cloud: Globally(Yes)` with no surface-conditional default today, so flipping its default is a single one-line change that makes NLD opt-in on every surface that gates on it. A TUI-only default would require brand-new per-surface-default infrastructure and is incoherent with cloud sync (the synced stored value would override any per-surface default once set), so it is rejected (see Design alternatives).
2. **The two slash commands are TUI-only.** They are gated into the static-command registry by `settings_mode == SettingsMode::Tui` and classified by `TuiSlashCommand` so only the TUI executes them. The GUI keeps its existing AI-settings-page toggle (`AISettingsPageAction::ToggleAIInputAutoDetection`); adding the commands to the GUI is out of scope for this request.
3. **Leave the existing `nld_in_terminal_enabled_internal` migration untouched.** That migration (`SettingsInitializer::handle_user_fetched`) is GUI-only (it seeds a GUI-surface setting) and reads the live value of `ai_autodetection_enabled_internal`, so flipping the default naturally produces opt-in-consistent seeding with no code change; removing the (overdue) migration is separate cleanup and out of scope.

*Behavior* (numbered, testable invariants from the user's view):
1. **Default is opt-in.** A fresh user who has never touched NLD has NLD *off*: `is_ai_autodetection_enabled(app)` returns `false` (assuming AI is otherwise enabled), so the TUI input does not auto-detect natural language and does not auto-route to agent input on detection. (The TUI input-mode policy falls back to `AI_LOCKED_CONFIG`.)
2. **`/enable-natural-language-detection` turns NLD on.** Running the command in the TUI sets `ai_autodetection_enabled_internal` to `true`, persists it, syncs it to cloud (per the setting's `sync_to_cloud`), and the TUI reactively re-runs input detection (existing `AISettingsChangedEvent::AIAutoDetectionEnabled` subscription). A confirmation hint is shown.
3. **`/disable-natural-language-detection` turns NLD off.** Running the command sets `ai_autodetection_enabled_internal` to `false`, persists + syncs, and the TUI reactively stops auto-detecting. A confirmation hint is shown.
4. **Commands are TUI-only.** The two commands appear in the TUI slash menu (when AI is enabled) and are executable there; they do *not* appear in the GUI slash menu. The GUI continues to expose the toggle only via the AI settings page.
5. **Commands require AI enabled.** The commands are gated by `Availability::AI_ENABLED`; when all AI is disabled (`is_any_ai_enabled` false) the commands are hidden, matching every other AI-gated slash command.
6. **Telemetry records the toggle with a slash-command origin.** Each enable/disable emits `TelemetryEvent::AgentModeToggleAutoDetectionSetting { is_autodetection_enabled, origin: AgentModeAutoDetectionSettingOrigin::SlashCommand }`, and `record_static_slash_command_accepted` fires for the accepted command (consistent with other TUI static commands). The existing `SettingsPage` and `Banner` origins are unchanged.
7. **Existing-user values are preserved.** Users who previously explicitly enabled or disabled NLD keep their stored value; only users who never set it move to the new opt-in default (`false`). The migration's early-return guard (`is_value_explicitly_set`) ensures already-migrated users are not re-seeded.
8. **GUI side effect is intentional and consistent.** Because the setting is shared, the GUI's "Autodetect terminal commands in agent input" (AgentView) / "Natural language detection" (non-AgentView) toggle also defaults to off for users who never set it. The GUI's *terminal-input* NLD (`nld_in_terminal_enabled_internal`) is a separate GUI-only setting that already defaults to `false` and is unaffected.

*Design alternatives* (per decision point with more than one reasonable approach):
- **Default-flip scope — global vs TUI-only.**
  - *Global flip (chosen)*: set `default: false` on `ai_autodetection_enabled_internal`. One line; coherent with the setting being `surface: ALL` + cloud-synced; makes NLD opt-in everywhere. Con: also flips the GUI agent-view command-autodetection default to off (acceptable; GUI keeps its settings-page toggle and its separate GUI-only terminal-NLD setting which is already opt-in).
  - *TUI-only flip (rejected)*: would need either (a) new per-surface-default support in `define_settings_group!` (does not exist; sizable macro + persistence change), or (b) a new TUI-specific gate setting that ANDs into the TUI's `is_ai_autodetection_enabled`. (a) is incoherent with cloud sync — the synced stored value overrides a per-surface default once any surface sets it, so the per-surface default only matters for never-toggled users and still syncs the same single value. (b) diverges the TUI from the shared setting the request refers to ("make NLD opt-in") and adds a new cloud setting. Both are materially larger and not what "make NLD opt-in" asks for. Re-evaluate only if kevin explicitly wants GUI to keep its current opt-out default.
- **Slash-command surface — TUI-only vs both.**
  - *TUI-only (chosen)*: register the two commands only when `settings_mode == SettingsMode::Tui`; classify + execute via `TuiSlashCommand` + `execute_tui_slash_command`. Matches the request ("for our TUI app"); the GUI already has the settings-page toggle, so GUI commands are redundant.
  - *Both surfaces (rejected for now)*: register unconditionally and add a GUI handler in `Input::execute_slash_command`. Reasonable future extension (especially if the global flip lands and GUI users want a quick on/off), but not requested. Documented so it can be added later without redesign.
- **Migration — leave vs modify vs remove.**
  - *Leave as-is (chosen)*: the migration reads `*ai_settings.ai_autodetection_enabled_internal` (live value). Under the global flip, never-toggled existing users now read `false`, so un-migrated users get `nld_in_terminal_enabled_internal = false` (opt-in-consistent); already-migrated users hit the `is_value_explicitly_set` early return and are unchanged. No code change needed.
  - *Modify (rejected)*: not needed — reading the live value already produces the right behavior.
  - *Remove (rejected as scope creep)*: the TODO says remove ~6 weeks from 2/5/26 (overdue), but removing it is unrelated cleanup; doing it here would expand blast radius and mix concerns. Track separately.

*Risks / blast radius:*
- **GUI behavior change.** The global flip changes the GUI's "Autodetect terminal commands in agent input" / "Natural language detection" default to off for never-toggled users. Mitigation: the GUI settings page still exposes the toggle; existing users who set it keep their value. If this is unacceptable, switch to the TUI-only approach (larger).
- **Cloud sync.** Because the setting is cloud-synced, a user who toggles on one surface/device propagates to all. This is existing behavior, not new, but the default change means new users start off everywhere. No mitigation needed (consistent opt-in posture).
- **Telemetry schema.** Adding a `SlashCommand` origin variant is additive (new serde value); existing consumers reading `origin` must handle the new value. Verify any downstream analytics/dashboard filter that enumerates origins is updated to include `slash_command`.
- **`TuiSlashCommand` exhaustiveness.** Adding variants requires updating every match on the enum (the `from_static_command` classifier + `execute_tui_slash_command` + any tests). The compiler enforces this.
- **Migration interaction.** Under the global flip, existing GUI users who never set the autodetection and never ran the migration will now seed `nld_in_terminal_enabled_internal = false` (instead of `true`). This is the intended opt-in behavior, not a regression, but it is a behavior change for that cohort.
