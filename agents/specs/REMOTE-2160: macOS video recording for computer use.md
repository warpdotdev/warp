*Spec: macOS video recording for computer use (REMOTE-2160)*

== PRODUCT ==
*Summary:* Add real video recording for computer use on macOS cloud runners (Namespace macOS VMs), mirroring the existing Linux `x11grab` recorder. Today `create_recorder()` on macOS returns a no-op (`RecordingError::Environment`), so computer-use recordings are unavailable on macOS. After this change, a macOS run with computer use enabled produces a real MP4 (H.264 / yuv420p) of the main display and publishes it through the same path Linux uses — provided the macOS sidecar image bundles ffmpeg (dev channel first, matching Linux).

*Key design choices:*
1. Mirror the Linux recorder, don't reinvent. The macOS recorder shells out to the same ffmpeg subprocess lifecycle (spawn → wait for first output → SIGINT to finalize → bounded by `-t`/`-fs`), swapping only the input device from `x11grab` (`$DISPLAY`) to `avfoundation` (`Capture screen 0:none`). Reuse the shared `RecordingConfig` / `RecordingOutput` / `RecordingError` types and the `RecordingHandle` lifecycle unchanged.
2. Select the avfoundation screen by NAME, not index. `"Capture screen 0"` is parsed directly by ffmpeg (`sscanf(ctx->video_filename, "Capture screen %d", ...)`) and is stable/English; the integer index shifts when the camera count changes (cameras precede screens in avfoundation's combined index space). Robust in headless Namespace VMs (one display, typically no cameras), and matches the macOS screenshot path's main-display-only behavior (`screencapture -m`).
3. Gate ffmpeg on the dev channel for BOTH platforms, consistently. macOS bundles a static darwin/arm64 ffmpeg into the `oz-macos-sidecar` application image only when `WARP_CHANNEL=dev`, mirroring `xvfb-sidecar/Dockerfile`'s existing dev-gate. Stable macOS ships without ffmpeg → recording stays unsupported (no-op) on stable, exactly as on stable Linux today. Promoting ffmpeg to stable is a separate, independent decision.
4. The committed spec lives in the `warp` repo; the change spans two repos — `warp` (the macOS recording substrate) and `warp-agent-docker` (bundling ffmpeg into the sidecar image). This PR carries the `warp` substrate plus this spec file; a coordinated PR on `warp-agent-docker` carries the packaging change. Both are described here as one contract.

*Behavior* (numbered, testable invariants from the consumer's view):
1. On macOS, `create_recorder()` returns a real recorder (not no-op) when ffmpeg is on PATH. `Recorder::start` launches an ffmpeg `avfoundation` capture of the main display and resolves only once capture is confirmed live (the output file has grown). `Recorder::stop` finalizes a playable MP4 and returns a `RecordingOutput` with a real path, duration, dimensions, and size.
2. On macOS, when ffmpeg is NOT on PATH (stable sidecar, or a dev image before bundling lands), `Recorder::start` fails fast with `RecordingError::Environment` naming ffmpeg — recording is reported unsupported, never silently produces nothing. Same shape as Linux without ffmpeg.
3. The macOS `WARP_MOCK_RECORDER` path is unchanged: setting it still returns the mock recorder (fixture-copy) for UI/publish-flow testing without real capture. The real recorder does not interfere with the mock path, and the mock path still takes precedence in `create_recorder()`.
4. The produced MP4 is playable and bounded: H.264 (`libx264`), `ultrafast` preset, `yuv420p` pixel format, `+faststart` movflag; dimensions forced even (the libx264/yuv420p requirement); duration ≤ `RecordingConfig.max_duration` and size ≤ `max_size_bytes`, enforced by ffmpeg `-t` / `-fs`. Encode settings identical to Linux recordings.
5. The recording captures the cursor and mouse clicks (`-capture_cursor 1 -capture_mouse_clicks 1`) so computer-use replay is intelligible.
6. Abandoned recordings are cleaned up: if a `RecordingHandle` is dropped without `stop`, the partial MP4 and its log are removed (ffmpeg is killed on drop). Same contract as Linux.
7. A recording ffmpeg finalizes produces `RecordingCompletionStatus::Completed`; one that exits before an explicit `stop` (limit reached or crash) produces `StoppedEarly`. `poll_exit` reports the kind. Same semantics as Linux.
8. Bundling: a dev `oz-macos-sidecar` image packages a static darwin/arm64 `ffmpeg` into `$TOOLS_DIR` (on PATH via the entrypoint), with sha256 verification matching the existing `download_*` helpers. A stable `oz-macos-sidecar` image does NOT package ffmpeg.
9. No regression to Linux recording or to macOS screenshots/input: the Linux `x11grab` recorder, the macOS `screencapture` screenshot path, and macOS `CGEventPostToPid` input are all unchanged. The server-side macOS dispatch/provisioning code is unchanged.
