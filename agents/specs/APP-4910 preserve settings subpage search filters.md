*Proposed change: Preserve per-widget search filters when restoring settings subpages*

*Summary:* The Warp settings search flow correctly computes per-widget matches for the Code and AI subpages, but rebuilding the active subpage after page-level filtering resets its `PageType` filter to “all widgets,” so the Editor and Code Review page remains unfiltered. Preserve and reapply the active subpage’s query filter after every rebuild, including auto-selection of a different matching subpage.

*Key design choices:* Keep the existing `Uncategorized` widget structure (one header/setting widget per Editor and Code Review setting, matching the Account page) and fix the shared search-state restoration path instead of introducing new UI categories or duplicating full-page state. Apply the query again after a subpage is rebuilt so the existing `subpage_filter` match counts and rendered widget filter remain consistent.

*Design alternatives:*
- *Reapply the query after rebuilding the active subpage (selected):* Small, localized change that preserves the current `PageType`/`SettingsWidget` architecture and works for both Code and AI subpages. It also handles the auto-selected destination after the current subpage has no matches.
- *Avoid resetting Code/AI to the full page while calculating page-level matches:* Would require a second backing-page representation or a separate filter state, duplicating page construction and increasing lifecycle complexity.
- *Persist `PageType` filter indices through `set_active_subpage`:* Avoids a second filter call but indices belong to different widget vectors and can become stale when feature/platform-gated widgets change; it is harder to test and easier to misapply.
- *Convert the active page to `Categorized` sections:* Could add visual grouping, but it changes the settings UI without addressing the state-loss defect. The current per-widget `Uncategorized` pattern already matches Account and is sufficient for searchable settings.

*Root cause / approach:* At `e3d4cd663f867cd1a924e608cf6da6207dfb65e9`, `SettingsView::handle_search_editor_event` (`app/src/settings_view/mod.rs:1468-1590`) first switches through each AI/Code subpage and calls `update_filter`, then switches both backing pages to `None` to compute the page-level `pages_filter`. Its restore block calls `set_active_subpage(Some(...))` without another `update_filter` call. `CodeSettingsPageView::set_active_subpage` (`app/src/settings_view/code_page.rs:419-473`) and the equivalent AI builder create a fresh `PageType::new_uncategorized`, whose constructor initializes every widget index as visible. `PageType::get_filtered` (`app/src/settings_view/settings_page.rs:1490-1545`) only reads the stored indices and does not re-run the query during render, so the visible active subpage shows every widget even though `subpage_filter` contains the correct sidebar count. The same restore sequence applies to AI subpages and must use the same fix.

The implementation should centralize the “restore active subpage and apply the current query” operation (or an equivalent helper), call it after the backing-page filter pass, and call it again after search auto-selects a different subpage. The helper must leave `subpage_filter` as the source of sidebar/umbrella visibility and use the same query against the rebuilt `PageType` so `get_filtered()` renders only matching widgets.

*Affected files:* `app/src/settings_view/mod.rs` (search orchestration and restore/auto-select path), `app/src/settings_view/code_page.rs` (only if a small restore/filter helper or test seam is needed), `app/src/settings_view/ai_page.rs` (only if the shared helper needs an AI-side hook), and `app/src/settings_view/mod_tests.rs` and/or `app/src/settings_view/code_page_tests.rs` for regression coverage.

*Open questions resolved:*
- *Is the Editor and Code Review page actually missing widgets?* No. The current revision already creates `CodeSubpageHeaderWidget` plus individual `SettingsWidget` implementations for External Editor, Auto Open Code Review Pane, Code Review Panel, Diff Stats, Project Explorer, Global Search, Show Hidden Files, Format on Save, and Auto Save (`app/src/settings_view/code_page.rs:2470-3012`). The reported “not broken into widgets” symptom is caused by the rebuilt page’s all-visible filter.
- *Which organization pattern should be used?* Keep `PageType::new_uncategorized` with one widget per setting and the existing subpage header, matching the Account page’s searchable structure; no new visual category taxonomy is required for this fix.
- *Do AI subpages need the same treatment?* Yes. They use the same search sequence and rebuild a fresh `PageType` during restore; the generic restoration path and a focused AI regression assertion must prevent the same stale/all-visible state.
- *What happens when the current subpage has no matches?* Existing auto-selection remains responsible for moving to the first visible matching section. The restored destination must also have its query reapplied before rendering.

*Risks / blast radius:* The search handler is shared by all settings pages, and a restoration change can affect Code, AI, umbrella expansion, and auto-selection. Limit the change to subpage rebuild/filter sequencing; preserve the existing page-level `pages_filter`, `subpage_filter`, empty-query behavior, and feature/platform `should_render` checks. Regression tests must cover both a matching current subpage and auto-selection away from a non-matching current subpage.
