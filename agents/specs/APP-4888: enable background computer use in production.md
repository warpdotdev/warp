# APP-4888 — Enable background computer use in production

*Spec: staged production rollout for background computer use*

== PRODUCT ==
*Summary:* Promote the existing per-window/background computer-use capability from internal Dogfood builds to production client builds. A production client may advertise background computer use only when the feature is enabled in that build and the current operating-system/display environment can actually perform it; all other environments must retain the existing whole-screen behavior.

*Key design choices:* Enable the existing Cargo-to-runtime bridge by default rather than duplicating the launch in `RELEASE_FLAGS`; preserve the `FeatureFlag::BackgroundComputerUse` enum and the capability probe as rollback/fallback controls; keep unsupported Linux Wayland and Windows clients on the legacy screen-only path.

*Behavior* (numbered, testable invariants):
1. Stable and Preview production client builds initialize `FeatureFlag::BackgroundComputerUse` through the existing `background_computer_use` Cargo feature, while Dogfood no longer has a separate, competing-only enablement entry.
2. The client sends `supports_background_computer_use=true` only when both `FeatureFlag::BackgroundComputerUse.is_enabled()` and `computer_use::background_supported()` are true. If either term is false, the request advertises `false` and the server receives the legacy screen-only contract.
3. macOS clients report capability and can target a specific window without requiring that window to be frontmost; Linux X11 clients report capability only after the XI2/MPX probe succeeds; Linux Wayland and Windows clients report capability as false.
4. A request with capability false remains behavior-compatible with the existing screen-targeted computer-use flow: no window target is advertised or emitted, no window metadata is required, and existing approval/cancellation behavior remains intact.
5. A request with capability true can enumerate/select a window and execute a window-targeted action plus a window-targeted screenshot. The model-visible coordinates and returned window metadata remain consistent with the existing background-computer-use contract.
6. Rollout is staged by release rings: Dogfood validation first, then Preview, then a monitored Stable canary and progressively larger Stable cohorts. Promotion pauses or rolls back if capability-advertisement, action-error, crash, latency, or approval/cancellation signals regress.

*Design alternatives*:
- *Enable only `PREVIEW_FLAGS` and `RELEASE_FLAGS`*: rejected. This duplicates the channel enablement that the Cargo bridge is designed to provide, and the existing release-list comment explicitly recommends adding a promoted feature to `app/Cargo.toml` defaults instead. It also leaves the production build's declared feature set inconsistent with the runtime flag.
- *Enable `background_computer_use` in `app/Cargo.toml` `default`, selected*: this makes the existing cfg bridge deterministic for normal client builds, avoids a stable-only override in `app/src/bin/stable.rs`, and leaves the runtime enum/probe available for rollback and platform fallback. Remove the Dogfood-only list entry so there is one source of launch truth.
- *Enable unconditionally on every OS*: rejected. Wayland has no per-window portal contract and Windows currently only supports foreground/screen input; advertising the capability there would cause the server/model to emit actions the client cannot honor.
- *Add a server-side experiment or proto change*: rejected for this ticket. The server already consumes the boolean request capability and preserves the legacy path when it is false; changing server behavior would expand the blast radius without solving the client build omission.

*Proposed changes:*
1. Add `background_computer_use` to the `default` feature list in `app/Cargo.toml`. Do not add a stable-only override in `app/src/bin/stable.rs`; the existing `app/src/features.rs` bridge will enable the runtime enum for all normal production client builds.
2. Remove `FeatureFlag::BackgroundComputerUse` from `DOGFOOD_FLAGS` in `crates/warp_features/src/lib.rs`. Do not delete the enum variant, the app bridge, the capability probe, or the request/action gates; those are required for rollback and unsupported-platform behavior. Do not add duplicate `PREVIEW_FLAGS` or `RELEASE_FLAGS` entries: the selected Cargo-default path is the single production enablement mechanism.
3. Keep the request capability expression as the single advertisement gate: `FeatureFlag::BackgroundComputerUse.is_enabled() && computer_use::background_supported()`. If it is refactored into a helper for testing, both request construction and any future capability reporting must call that helper; do not create a second OS/platform decision table.
4. Keep action execution's `background_enabled` input tied to `FeatureFlag::BackgroundComputerUse.is_enabled()`. When false, preserve the actor's legacy screen path exactly as documented by `computer_use::Options`.
5. Use existing `TelemetryEvent::ComputerUseApproved` and `TelemetryEvent::ComputerUseCancelled` signals, plus server request/tool metrics, for rollout monitoring. Dashboard the capability boolean and action errors by client channel, OS/display backend (macOS, Linux X11, Linux Wayland, Windows), and release version; do not include window IDs, screenshots, coordinates, prompts, or other user content in new monitoring fields. If the current metrics cannot distinguish capability advertisements from legacy requests, add a privacy-safe counter at the existing request boundary rather than a new user-content telemetry payload.
6. No `warp-server` production code or protocol change is in scope. Run its existing focused computer-use tests as a compatibility gate and add a server test only if the current suite lacks an assertion that capability false omits window-target behavior.

*Open questions resolved:* The target repository is `warpdotdev/warp`; server support is already shipped. Production enablement is selected through the app Cargo default because the current codebase's feature-flag guidance favors that path for launch. The feature flag remains in the enum after promotion so a revert or channel-specific hotfix can disable it without deleting the capability implementation. Linux X11 real-WM behavior is a rollout validation requirement, not an assumption that Xvfb validation is sufficient.

*Rollout and rollback:*
1. *Dogfood:* land the change behind the existing internal validation process; verify macOS and X11 behavior, the false-capability matrix, and no regression in screen-only computer use.
2. *Preview:* publish the first artifact containing the default Cargo feature; hold for a complete Preview observation window while comparing capability-advertisement rate, computer-use approval/cancellation, action errors, crash-free rate, and latency with the previous build.
3. *Stable canary:* release to the smallest existing Stable ring. Expand to the next rings only when the same signals remain within the pre-release baseline and no platform-specific fallback or crash cluster appears. Record the observed version/ring and decision in the release change record.
4. *Rollback:* stop the next ring and ship the previous client or a revert that removes `background_computer_use` from `app/Cargo.toml` `default`; leave the enum, probe, server gate, and legacy path intact. A rolled-back client advertises false, so no server rollback or protocol migration is required. If only one OS is affected, ship a channel/platform-specific build with the default disabled while retaining the implementation for Dogfood investigation.
