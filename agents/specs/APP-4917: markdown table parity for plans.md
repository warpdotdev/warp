# Spec: Markdown table parity for plan and notebook documents (APP-4917)

== PRODUCT ==
*Summary:* Make valid GitHub Flavored Markdown (GFM) tables render consistently in Warp plan/notebook documents without collapsing the distinct notebook and general Markdown-file editing experiences. The first phase is a table-parity and rollout change that reuses the existing shared Markdown parser, buffer, layout, and table renderer; full editor unification remains a documented follow-up.

*Key design choices:* Keep `NotebooksEditorModel`/`RichTextEditorView` for AI plans, Warp Drive notebooks, and rendered file notebooks; keep `CodeEditorView`/`LocalCodeEditorView` for general Markdown source editing. Use the existing `FeatureFlag::MarkdownTables`-controlled GFM path and one shared table appearance/interaction implementation rather than adding a plan-only renderer. Treat raw Markdown, document versioning, cloud/file persistence, LSP/Vim/diff/autosave, and notebook-only blocks as separate contracts.

*Behavior* (numbered, testable invariants from the user's/consumer's view):
1. A valid GFM table in an AI plan document renders as a table block when the supported product channel has Markdown table support enabled. Header cells, body cells, left/center/right column alignment, inline emphasis/code/links, escaped pipes, and multiple rows remain readable.
2. The same valid GFM table renders with the same shared table treatment in Warp Drive notebooks and the rendered view of file-backed Markdown notebooks. Surface-specific typography may differ, but table structure, alignment, row/column hierarchy, separators, padding, and overflow behavior must not drift.
3. AI plan streaming, agent diff application, document-version selection/restoration, and user edits continue to operate on the notebook model. Re-rendering updated Markdown must not drop a table, duplicate rows, lose inline formatting, or change the selected document version.
4. Warp Drive notebook persistence and notebook-only content (embedded objects, workflows, Mermaid, and child notebook models) remain supported. A table must not be silently converted into a plain paragraph when a notebook is saved, reopened, synced, or restored.
5. File-backed Markdown notebooks keep their current two-mode contract: Rendered mode uses the selectable notebook renderer, while Raw mode opens the source in the existing code-editor/file-buffer path. Toggling modes must preserve the source Markdown, including table delimiters and alignment markers.
6. General `.md` editing remains a code-editor experience. LSP, Vim/keymap behavior, tabs, global buffers, dirty state, save/save-as, autosave, diff UI, remote-conflict handling, and plain-text source semantics are not replaced by notebook editing as part of this work.
7. In editable notebook contexts, users can place the cursor in table cells, navigate, select within/across cells, type/delete/paste, copy Markdown, follow links, and scroll horizontally through wide tables without losing selection or cursor visibility. In read-only contexts, selection and copy remain available as today.
8. Table-like text that is malformed, unsupported, or inside a fenced code block remains deterministic and readable using the existing non-table fallback; it is not promoted to a table merely because it contains pipes.
9. Raw/source Markdown remains recoverable: notebook serialization/export and the file Raw mode preserve table content sufficiently to reopen the same table, including alignments and inline cell markup. No lossy HTML-only representation becomes the persistence source of truth.
10. The `MarkdownTables` rollout is explicit and observable. Every channel/build that advertises these plan/notebook surfaces has the flag enabled (unless an intentional user/team override is documented), and a disabled/unsupported state has the existing predictable paragraph fallback rather than a partially interactive table.
11. TUI plan documents are treated as a separate read-only renderer contract. Their existing GFM parser/table renderer continues to render the same valid table content, but GUI flag or model changes must not be assumed to update TUI behavior without a TUI test.

*Scope:* Phase 1 includes parser/flag verification, shared editor-backed table parity, persistence/round-trip guarantees, and regression coverage across the four GUI surfaces plus the independent TUI path. It does not merge the notebook and code-editor models, redesign non-table Markdown, or add a new Markdown table syntax.

*Success metrics:* The same fixture table is correctly parsed, rendered, interacted with, serialized, and reopened in AI plans, Warp Drive notebooks, and rendered file notebooks; the same fixture remains ordinary source text in the general Markdown editor; no existing save/version/sync/editor behavior regresses; and the feature is proven in automated tests, presubmit, and a running GUI with screenshot evidence.

*Design alternatives*:
- *A. Full editor unification now* — Replace general Markdown `CodeEditorView` with `RichTextEditorView`/`NotebooksEditorModel`, or create one model that conditionally supplies both behaviors. This could make rendered/editable tables feel identical, but it would couple notebook block semantics and child models to LSP/Vim/diff/autosave/global-buffer persistence, complicate Raw/Rendered transitions, threaten source fidelity, and require migration of established code-editor behavior. This is materially larger than the L estimate and is rejected for Phase 1.
- *B. Shared document facade with two renderers* — Introduce a common document/persistence abstraction while retaining notebook and code-editor renderers. This reduces future duplication but still requires new model ownership, synchronization, source-of-truth rules, and cross-renderer selection semantics before it provides user value. It is a possible follow-up after table parity is stable, not a prerequisite.
- *C. Selected: table parity and explicit rollout* — Keep the existing model boundaries, make all supported notebook/plan reload paths use the existing GFM table path, keep the shared table appearance/interaction implementation as the only GUI table renderer, make channel/override behavior explicit, and add cross-surface regression/UI coverage. This is the smallest change that fulfills the useful part of the request while preserving existing contracts.

*Proposed changes:*
1. Define one canonical fixture set for valid GFM tables (all alignments, inline formatting, escaped pipes, links, wide cells, multiple rows), malformed table-like text, and fenced-code table-like text. Use it in parser, buffer, layout, surface, and TUI tests.
2. Audit and, where necessary, correct every notebook/plan Markdown ingestion path so it calls `reset_with_markdown`/`Buffer::from_markdown` with the same GFM-enabled path. Do not introduce a second parser or convert general Markdown files to notebook blocks.
3. Keep `FeatureFlag::MarkdownTables` as the rollout gate for the existing implementation. Make supported channel initialization and any user/team override behavior testable and documented; if the flag is disabled, assert the existing deterministic paragraph fallback rather than permitting a half-enabled table UI.
4. Preserve the existing shared `markdown_table_appearance`/`markdown_table_style` source and editor `RenderableTable` implementation. Any needed fixes should be limited to parity, feature-state wiring, or regressions found by the fixture; do not duplicate table painting in `ai_document_view`, file notebooks, or the code editor.
5. Add model/surface coverage for AI agent diff updates, document version restoration, Warp Drive persistence/sync, and file notebook Rendered/Raw transitions. Assert both rendered structure and serialized Markdown/source recovery.
6. Add a separate TUI regression test using the same fixture, because `warp_tui` parses and paints tables independently. Keep GUI and TUI implementation ownership separate while aligning their supported input behavior.
7. Do not change the general Markdown editor's model or code-specific features. If a product later wants a unified editor, open a follow-up design/spec with explicit decisions for LSP/Vim, diff/autosave, global buffers, notebook-only blocks, source persistence, and migration.

*Persistence and data flow:* Source Markdown enters a notebook through `reset_with_markdown` -> `Buffer::from_markdown` -> GFM `FormattedTextLine::Table` -> internal table block/cache -> shared table layout/rendering. User and agent edits serialize through the existing Markdown export path. File Raw mode remains backed by the global code buffer, and Warp Drive/AI document persistence remains owned by their existing models. No new table storage format is introduced.

*Open questions resolved:*
- *Should Phase 1 unify editors or add table support?* Add table parity first. The current stacks share rendering infrastructure but differ in model semantics, persistence, raw/rendered modes, LSP/Vim/diff/autosave, and notebook-only blocks; unification is not a safe L-sized change.
- *Which GUI surfaces are in scope?* AI plan documents, Warp Drive notebooks, and file-backed Markdown notebooks are in scope for rendered-table parity. General `.md` code editing is explicitly protected, with only source-preservation regression checks.
- *Is TUI included automatically?* No. TUI plans use a separate parser/renderer; they require their own regression test and are not changed implicitly by GUI work.
- *Is a new table renderer or storage format needed?* No. The shared parser, internal table representation, appearance helper, layout, interaction, serialization, and existing tests are the implementation baseline.
- *What happens when the feature is off?* Preserve the current flag-gated paragraph fallback and test it. Supported production channels should explicitly enable the flag; an intentional override may disable it but must not leave interactive table affordances active.
- *Can visual parity require identical fonts everywhere?* No. Share table structure/chrome and interaction semantics; retain surface-specific typography where the current renderer requires it.

*Risks / mitigations:*
- *Persistence/version regression:* Reparse/export can lose table structure or selected document versions. Add round-trip tests around `apply_diffs`, restore, sync, and raw file toggles.
- *Editor boundary regression:* Replacing code editor behavior would break LSP/Vim/diff/save semantics. Keep CodeEditorView untouched and add non-regression checks.
- *Flag drift:* Compile-time availability and runtime/channel state can diverge. Test startup initialization and enabled/disabled behavior in the same build configuration used by the app.
- *Interaction regression:* Table geometry changes can affect cursor, selection, links, or horizontal scrolling. Retain and extend geometry/interaction tests and verify them in the running GUI.
- *Renderer drift:* A surface-specific table implementation could diverge. Route all GUI notebook tables through the existing shared appearance/RenderableTable path and add a fixture comparison.
- *Environment limitations:* If a local GUI cannot be launched in CI, record the environment mismatch, retain deterministic automated coverage, and require a reviewer-run computer-use session before merge.
