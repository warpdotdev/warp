*Spec: TUI shows the input editor with a `Starting shell...` indicator during session bootstrap (mirror the GUI)*

== PRODUCT ==
*Summary:* While a TUI session is bootstrapping (the shell has not yet reached its first post-bootstrap precmd), the Warp TUI renders the prompt input editor with a single status line `Starting shell...` immediately above it. Prompt submission is disabled during this state; the user can prepare a draft that survives into the normal prompt once bootstrap completes. Silent shell startup-script execution remains in this presentation so the input does not flicker away during normal startup. If a startup script produces visible output that may require interaction, the terminal takes input until bootstrap resumes. This mirrors the GUI desktop client, which already shows `Starting shell...` above an editable input while gating execution and focuses the terminal when a visible bootstrap block appears.
*Scope boundary — bootstrap vs. PTY-owned work (IMPORTANT):* This change applies **only** to the initial session bootstrap phase before `is_bootstrapping_precmd_done()` becomes true, except while visible startup output temporarily owns the PTY. It does **NOT** apply to regular long-running commands (LRCs) or alt-screen apps, which select `TuiInputTarget::Pty`, hide the editor, and forward input to the foreground process. Silent shell startup-script execution remains `Disabled`; a visible startup-script block selects `Pty` so interactive startup prompts such as update confirmations remain usable. The `Starting shell...` line is never shown during a regular LRC or while visible startup output owns the PTY. See behavior invariant #9 and validation criteria #3 and #7 for the explicit routing guards.
*Key design choices:*
1. Render the input editor + the `Starting shell...` line during the existing `TuiInputTarget::Disabled` (bootstrap) state — and **only** that state — rather than introducing a new enum variant. Silent startup-script execution remains `Disabled`; `Pty` remains reserved for LRCs, alt-screen apps, and visible startup-script interactions.
2. Keep prompt submission gated by the *existing* `handle_submitted` guard (`input_target().agent_editor_owns_input()`), which already returns early for `Disabled` (and for `Pty`); no new submission-blocking code is needed.
3. Do not forward ordinary keystrokes to the PTY during bootstrap (unchanged: `with_pty_input` is only attached when `pty_owns_input()`), so showing the editor does not risk leaking draft text into the bootstrapping shell.
4. *Scope discipline:* the new render/focus branch is keyed on `input_target == Disabled` only; it must never fire for `Pty`. This keeps LRC, alt-screen, and visible startup-script input behavior intact and is enforced by routing and render regression tests.

*Behavior* (numbered, testable invariants from the user's view):
1. While the active session is not yet bootstrapped (the TUI's `Disabled` input-target state), the session frame visibly renders the prompt input editor.
2. While in that bootstrap state, a single line containing exactly `Starting shell...` is rendered immediately above the input editor.
3. While in that bootstrap state, pressing `Enter` (the submit key) does not submit the prompt, execute a shell command, or send an agent query; the draft text remains in the editor unchanged.
4. The user can type and edit a draft in the input editor during bootstrap, and that draft persists without loss when bootstrap completes (the same input model/view is rendered before and after; no clear on the `BootstrapPrecmdDone` transition).
5. When bootstrap completes (`BootstrapPrecmdDone`), the `Starting shell...` line disappears, normal prompt rendering resumes (input box border, footer, inline menus, warping indicator all return to their `AgentEditor` behavior), and `Enter` submits as usual.
6. Silent shell startup-script execution keeps showing the bootstrap indicator and editor. If startup output becomes visible, the TUI switches to `Pty`, hides the editor, focuses the session, and forwards input until startup completes. Alt-screen apps, blockers, orchestration tabs, and conversation restore retain their existing behavior.
7. The GUI desktop client is unchanged (this change is TUI-only).
8. Reserved bindings (`ctrl-c`, `ctrl-d`, plan toggle, tab navigation) behave consistently with the normal prompt mode while the input is focused during bootstrap, because the input view's keymap context is active. Specifically `ctrl-c` clears the draft and arms the exit-confirmation window (same as normal prompt mode with no active conversation), and a second `ctrl-c` within the window exits the TUI.
9. *Long-running commands (LRCs) are explicitly excluded.* While a user-controlled long-running command is active (`TuiInputTarget::Pty` selected via `inline_process_owns_input`, e.g. `cat`/`sleep`/`tail -f` running in the foreground of an already-bootstrapped shell), the input editor remains **hidden** — no editor, no `Starting shell...` line, no footer input box — exactly as the TUI renders today. PTY input is forwarded to the foreground process as today. The `Starting shell...` indicator and the draft-editor-during-bootstrap behavior never appear during an LRC. (This is the requester-clarified scope boundary: bootstrap-only, not LRCs.)

*Design alternatives* (per decision point with more than one reasonable approach):
- *Input-target modeling* — (a) **Repurpose `Disabled`** (chosen): change `render` to emit the `Starting shell...` line + the input box + footer during `Disabled`, and change `focus_current_owner` to focus the input view during `Disabled`; update the `Disabled` doc to "submission disabled and PTY input not forwarded, but the editor is rendered with a `Starting shell...` indicator so the user can prepare a draft." Pros: minimal churn (enum, state fn, and the existing `shell_startup_routes_input_by_bootstrap_stage` test all stay valid); reuses the existing `handle_submitted` submission guard; the `BootstrapPrecmdDone` re-render already transitions out. Cons: the name `Disabled` becomes slightly softer ("disabled submission" rather than "hidden editor"), mitigated by the doc update. (b) *Add a new `TuiInputTarget::Bootstrapping` variant*: more self-documenting; `agent_editor_owns_input()` and `pty_owns_input()` both return `false` for it (so submission + PTY gating are automatic), and a new `renders_editor()`/`bootstrapping()` predicate selects the new render+focus branch. Cons: more churn — the enum, `tui_input_target_for_state`, `focus_current_owner`, `render`, `keymap_context`, and the `shell_startup_routes_input_by_bootstrap_stage` test (which asserts `Disabled` after `reinit_shell`) all change. *Open questions resolved:*
- *Does this change apply to long-running commands (LRCs) too, or only to the initial session bootstrap?* Resolved (requester-clarified): **bootstrap only.** The `Starting shell...` indicator, the shown input editor, and the draft-during-bootstrap behavior apply ONLY to the `TuiInputTarget::Disabled` state (before `is_bootstrapping_precmd_done()`). During a regular LRC (`TuiInputTarget::Pty` via `inline_process_owns_input`), the input editor stays hidden exactly as today, PTY input is forwarded to the foreground process, and `Starting shell...` is never shown. The `Pty` conditions are checked before the bootstrap check in `tui_input_target_for_state`, so an LRC always wins; this spec changes only the `Disabled` branch. Enforced by behavior invariant #9 and validation criterion #7.
- *What happens while shell startup scripts execute?* Silent startup-script execution remains in `Disabled`, preserving the `Starting shell...` editor continuously through normal startup. Once the active startup block becomes visible under the current transcript scope, `tui_input_target_for_state` selects `Pty`; the view hides the editor and forwards input so interactive startup prompts remain usable. `ModelEvent::VisibleBootstrapBlock` updates focus immediately, matching the GUI's terminal-focus policy.
- *Should the TUI mirror the GUI's remote-server-setup stage strings (`Installing Warp SSH Extension... (N%)`, `Initializing...`, etc.)?* Resolved: no, out of scope. The TUI does not import `RemoteServerSetupState` and has no SSH remote-server-setup UI surface today. This ticket ships the common `"Starting shell..."` string, which is identical to the GUI's common-path wording. Mirroring the remote-server ladder is a follow-up tracked under *Risks / blast radius*, to be done when/if the TUI gains a remote-server-setup surface.
- *Should the user be able to type a draft during bootstrap, or is the editor read-only?* Resolved: editable (focus the input view during `Disabled`), per the request ("show the input", "the draft remains available for editing after bootstrap") and GUI parity (editable input, gated execution).
- *Should the footer render during bootstrap?* Resolved: yes — render the normal footer (model/cwd/usage row) via the same `render_footer` path used for `AgentEditor`, so the bootstrap frame is visually consistent with the normal prompt frame and the `ctrl-c` exit-confirmation hint surface stays available. The cwd falls back to the process cwd until shell metadata arrives (existing `current_working_directory` fallback), which is acceptable.
- *Does showing the editor risk forwarding draft text to the bootstrapping shell's PTY?* Resolved: no. `with_pty_input` is attached only when `pty_owns_input()` (line 2876); `Disabled` does not own PTY input, so keystrokes go to the focused editor, not the PTY. This is verified by the input-policy test and the visual proof.

*Risks / blast radius:*
- *Bootstrap vs. LRC scope bleed (requester-flagged)*: `tui_input_target_for_state` checks `alt_screen_active`, `script_execution && startup_script_visible`, and `inline_process_owns_input` before the bootstrap-ready check. A regular LRC (`inline_process_owns_input` true, e.g. `cat`/`sleep`/`tail -f` in an already-bootstrapped shell) therefore resolves to `Pty`, never `Disabled`. The render/focus branch is guarded on `Disabled`, so it cannot fire during an LRC. The existing `ordinary_long_running_command_owns_inline_input` test and criterion #7 render test prove this.
- *Shell-startup PTY routing*: silent startup-script execution stays `Disabled`, avoiding a transient hidden-input frame during normal startup. A startup block with visible content resolves to `Pty`, and `ModelEvent::VisibleBootstrapBlock` updates focus and rerenders immediately. `with_pty_input` remains attached only for `Pty`.
- *Focus shift during bootstrap*: moving focus from `focus_self` to the input view during `Disabled` changes which keymap context is active. Reserved bindings (`ctrl-c`, `ctrl-d`, plan toggle, tab nav) are registered for the `TuiInputView`/`TuiTerminalSessionView` contexts (see `init` at line 329-377) and behave as in normal prompt mode. `ctrl-c` clears the draft + arms exit (consistent with normal mode); `ctrl-d` exits on an empty draft / deletes otherwise (consistent with normal mode). The visual proof criterion must confirm these during a real bootstrap; if a reviewer finds a ctrl-d-during-bootstrap exit surprising, a follow-up can gate `Eof`'s exit on `is_bootstrapping_precmd_done()` — out of scope for this ticket unless the visual proof surfaces a problem.
- *Blocker precedence during bootstrap*: blockers (e.g. a `RunAgents` card) already suppress the input box (line 2777-2784); the new bootstrap branch must respect `!blocker_active` so a blocker still takes precedence. Covered by mirroring the existing `!blocker_active` guard.
- *Conversation restore*: `ConversationRestoreState::Loading { origin: Startup, .. }` returns early (line 2714-2718) with a restoring screen, so the bootstrap branch is not reached during startup restore. `ConversationList`-origin loading proceeds and the bootstrap branch renders underneath; this is the same overlaying as today and is not a regression.
- *Remote-server-setup wording divergence*: documented above as a follow-up, not a regression.
- *Zero-state coexistence*: during a fresh bootstrap the transcript is empty, so the zero state fills the transcript slot above the `Starting shell...` line. The visual proof confirms this stacks correctly.
