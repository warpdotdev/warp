== PRODUCT ==
*Summary:* Add a "Speech Recognition Language" dropdown to Warp's Voice settings (Settings → AI → Voice) so a user can pin the language Wispr Flow transcribes in. The selected language flows from the client, through the warp-server transcribe API, into the Wispr Flow request body. "Auto-detect" (the default) sends no `language` and preserves today's behavior.

*Key design choices:*
1. **Setting is an enum, `AutoDetect` default, not `Option<String>.`** Mirrors the existing `VoiceInputToggleKey` enum-backed setting (type-safe, fixed dropdown, serializes cleanly, auto-populates the dropdown from `iter()`). A free-form string would invite typos and offer no validation. `AutoDetect` is `#[default]` and means "send no language".
2. **Top-level `language` in the Wispr body, string value, only when explicitly set.** The server adds `"language": "<iso-639-1>"` to the Wispr Flow request body only when the client sent a non-empty value; otherwise the key is omitted (Wispr auto-detects). This matches the requester's explicit playground example (`"language": "nl"` alongside `audio`). See the **Wispr `language` shape** risk note — if Wispr rejects a bare string, the fallback is a single-element list `["nl"]` (a one-line change), and the end-to-end validation criterion below catches a rejection.
3. **Cloud-synced globally, not device-local.** A language preference is user-level, not hardware-specific (unlike the toggle key, which varies by keyboard). So `SyncToCloud::Globally(RespectUserSyncSetting::Yes)`, like `voice_input_enabled` — not `SyncToCloud::Never` like the toggle key.
4. **Extend the `Transcriber` trait to carry the language.** `Transcriber::transcribe` gains a `language: Option<String>` parameter; `ServerVoiceTranscriber` sets `TranscribeRequest.language` from it. There is exactly one production implementor (`ServerVoiceTranscriber`), so this is low-blast-radius. The call site reads the setting from `ctx` before spawning the transcribe future.

*Behavior* (numbered, testable invariants from the user's view):
1. A "Speech Recognition Language" dropdown renders in Settings → AI → Voice, immediately below the "Key for Activating Voice Input" dropdown, **only while Voice Input is enabled** (same `is_voice_input_enabled(app)` gate as the key dropdown). When Voice Input is disabled, the dropdown is hidden (not just greyed out).
2. The dropdown's first/selected-by-default option is **"Auto-detect"**. A freshly-init user (no saved setting) sees "Auto-detect" selected.
3. Selecting a language from the dropdown **persists** it (like other settings) and updates the dropdown's displayed selection immediately; on the next voice request that language is used.
4. Selecting "Auto-detect" again reverts to sending no language (current behavior).
5. The setting **syncs across devices** (global sync), so a language chosen on one machine applies on another, subject to the user's cloud-sync preference.
6. **End-to-end:** when a language is selected, the client sends `language: "<code>"` in the transcribe request; the server forwards it to Wispr Flow; Wispr returns a 200 with transcribed text (proving it accepted the language). When "Auto-detect" is selected, no `language` is sent anywhere and behavior is unchanged from today.
7. **No collateral damage:** the existing toggle-key dropdown, the Voice Input enable toggle, transcription without a language, and the `properties` context Wispr already receives all continue to work unchanged.

*Risks / blast radius:*
- **Wispr `language` shape (string vs list).** The requester's playground example uses a top-level string `"language": "nl"`. Wispr's current public docs describe `language` as a **list** of ISO 639-1 codes (`["en"]`). This spec follows the requester's explicit string example; if Wispr's `/api` rejects a bare string, the fix is to send `[]string{request.Language}` instead — a one-line change in `buildWisprRequestBody`, caught by the end-to-end validation criterion (#1 below) before merge. Flagged for the requester.
- **Trait signature change** touches one production implementor and any internal callers of `transcribe(...)`. Grep confirms only `ServerVoiceTranscriber` implements the trait and the call site is the single `editor/view/voice.rs` spawn; `./script/presubmit` (clippy, `-D warnings`) will catch any missed call site.
- **Cloud sync** means a language set on one device propagates to others; this is intended but worth noting. Respects the user's sync opt-out.
- **Server has no existing transcribe test** and `transcribeWispr` makes a live HTTP call, so the regression test targets the extracted pure `buildWisprRequestBody` helper rather than the full handler (no network in CI).
