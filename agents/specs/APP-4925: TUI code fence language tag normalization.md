*Spec: TUI code-fence language tag normalization (APP-4925)*

Source ticket: https://linear.app/warpdotdev/issue/APP-4925/tui-weirdbroken-syntax-highlighting-in-code-blocks-language-tag-shown

== PRODUCT ==
*Summary:* When streamed AI Markdown puts a code-fence language identifier on the line immediately after an otherwise unlabeled opening fence (for example, ```` ``` ```` followed by `go` and then `package main`), the identifier must be treated as code-block metadata instead of rendered as the first code line. Canonical adjacent-fence syntax remains unchanged, and the TUI must preserve the code body's line breaks and indentation.

*Key design choices:* Normalize only at the existing AI-output section parser boundary; require an exact standalone, supported language token plus a following body line before promoting it; and leave incomplete streams, ordinary unlabeled snippets, and metadata-bearing fences untouched until the evidence is unambiguous.

*Behavior* (numbered, testable invariants):
1. For a canonical fence such as ```` ```go\npackage main\n...\n``` ```` the parser returns one code section with language `go` and code beginning with `package main`; the behavior is unchanged.
2. For a malformed/streamed fence such as ```` ```\ngo\npackage main\n...\n``` ```` the parser promotes the standalone `go` line to the language field and removes only that line from the code payload. The TUI therefore shows `go` in its language header and starts the code body at `package main`, never with a duplicate `go` code row.
3. Promotion is conservative: it applies only when the opening fence has no adjacent language and no fence parameters, the candidate is the first code line and is a standalone recognized programming/shell language alias, and a subsequent non-empty, non-closing-fence body line is present. Unknown or ordinary first lines remain unlabeled code content.
4. Streaming input is not destructively normalized before enough context exists. A partial payload ending after the candidate token (including an optional trailing newline) remains unchanged; when the accumulated payload later contains a qualifying body line, a fresh parse promotes the token exactly once without dropping or duplicating any body line.
5. Every code-body line after the promoted token retains its original order, line breaks, and indentation. The normalization does not trim or rewrite the body.
6. Existing special/metadata fences retain their current semantics. In particular, adjacent `mermaid`, `warp-runnable-command`, `warp-embedded-object`, table, and `path`/`start` parameter forms are not reinterpreted by this heuristic; canonical Mermaid rendering and source-link metadata remain intact.
7. A candidate that is the only content before the closing fence is not promoted. This preserves a legitimate unlabeled snippet whose first line happens to equal a supported language name when there is no body evidence.

*Root cause:* The regex recognizes a language only when it is adjacent to the opening fence. The malformed stream therefore enters an unlabeled code state, and the next line is appended as code before the TUI receives it. The TUI is behaving as designed; changing it would hide, rather than repair, the malformed section model.

*Proposed changes:*
1. Keep the existing opening-fence and parameter parsing intact. In the unlabeled `CurrentSection::Code` path, inspect the first code line with a bounded lookahead rather than globally changing the regex or rewriting the renderer.
2. Add a small normalization helper/state transition that considers promotion only when the opening fence had an empty language and an empty parameter string (there is no `CodeSource` metadata to preserve). Compare the first line after trimming surrounding whitespace as a single token, normalize aliases case-insensitively for recognition, and convert it through `ProgrammingLanguage`.
3. Accept a candidate only if it is a supported syntax language (`to_extension().is_some()`) or a supported shell (`is_shell()`), is not a reserved metadata token (`mermaid`, `warp-runnable-command`, `warp-embedded-object`, or table metadata), and the immediately following line is non-empty and is not a closing fence. On acceptance, set the language/language token and skip only the candidate line; append all later lines verbatim. If lookahead is unavailable, blank, or a closing fence, append the candidate as ordinary code.
4. Do not persist a separate streaming flag: the parser already re-runs over the accumulated message on each update. The bounded lookahead makes partial updates non-destructive, and the later complete parse naturally changes the payload once the body line arrives.
5. Add parser regression coverage in `app/src/ai/agent/util_tests.rs` and deterministic TUI coverage in `crates/warp_tui/src/tui_code_block_view_tests.rs` (or an equivalent existing TUI agent-block fixture) so both the section contract and the rendered rows are protected.

*Design alternatives*:
- *Broaden `CODE_START_REGEX` across a newline:* This is compact but couples a regular expression to streaming lookahead, makes parameter/source handling harder to reason about, and risks consuming ordinary first code lines. Rejected in favor of an explicit state transition.
- *Replace the app parser with `markdown_parser` for all agent output:* The shared parser already models a language line after a fence, but replacing this parser would change image/table/source-link/Mermaid handling and the existing section API in a much larger blast radius. Rejected; the fix belongs at the narrow boundary that owns AI-output sections.
- *Normalize every first line that resembles a language:* This would fix the screenshot but corrupt legitimate unlabeled snippets and metadata. Rejected; require a supported token plus a following body line, and preserve incomplete/ambiguous input.
- *Repair the TUI payload/rendering layer:* The renderer has no reliable way to distinguish an accidental language line from intentional code and would duplicate logic across GUI/TUI surfaces. Rejected; keep the TUI a faithful consumer of normalized sections.

*Open questions resolved:* The task's client-repository request overrides the stale `foreman/config.json` server target; implementation and this shared PR belong to `warpdotdev/warp`. The reported indentation-loss symptom is not reproduced by the minimal parser payload, so this spec treats body preservation as a required invariant and adds an end-to-end TUI check rather than assuming a separate layout race. No user-visible language promotion is attempted for reserved metadata tokens or a candidate without a body line.

*Risks / blast radius:* The heuristic touches every AI code block parsed by `app/src/ai/agent/util.rs`; a false positive could remove a legitimate first line or change syntax highlighting. Guards (unlabeled/no-parameters only, exact standalone supported token, immediate non-empty body evidence, reserved-token exclusions) limit that risk. Streaming reparses can change a code block from unlabeled to labeled once enough context arrives, so regression tests must cover both partial and complete payloads. Canonical fences, source links, Mermaid, runnable commands, embedded blocks, and body indentation must remain covered.
