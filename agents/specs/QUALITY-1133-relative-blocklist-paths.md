*Spec: Render blocklist tool paths relative to the invocation cwd*

== PRODUCT ==
*Summary:* File paths in file-oriented agent tool-call rows should be concise without becoming ambiguous. Each row formats its execution target against the cwd captured for the exchange that produced the tool call: targets in that cwd are shown as child-relative paths, nearby targets may use at most two leading parent traversals, and farther or incompatible targets stay absolute. Display formatting must not change which file the tool executes against or opens.

*Key design choices:*
1. Use the persisted `AIAgentExchange.working_directory` / `AIBlock.current_working_directory` as the invocation-cwd snapshot for every action rendered in that exchange. Do not add a per-action cwd field or read the terminal's later live cwd while rendering.
2. Add one shell-native display-path formatter next to `shell_native_absolute_path`, then make every in-scope renderer consume display-ready values instead of re-absolutizing them.
3. Compute the permitted relative forms by walking zero, one, or two ancestors of the captured cwd with `typed_path`; fall back to the normalized shell-native absolute target when no match exists. This preserves Unix, Windows, WSL, and MSYS2 syntax without interpreting a shell path through the host OS's `std::path` rules.
4. Keep display text and execution/open targets separate. Existing execution inputs stay unchanged; clickable rows continue resolving their displayed relative text with the same captured cwd into an absolute `DetectedLinkType::FilePath`.

*Behavior* (numbered, testable invariants from the user's view):
1. A target below the invocation cwd renders relative to that cwd with no leading `./`. From cwd `/foo/bar`, `/foo/bar/bazz/buzz` renders as `bazz/buzz`.
2. A target equal to the invocation cwd renders as `.`.
3. A target outside the invocation cwd renders relatively when reaching it requires one or two leading parent components. From cwd `/foo/bar/buzz`, `/foo/bazz` renders as `../../bazz`.
4. A target that requires three or more leading parent components renders as its normalized shell-native absolute path.
5. A target on an incompatible root, drive, host path space, or other path encoding for which a safe relative form cannot be produced renders as its shell-native absolute path.
6. If no invocation cwd is available, the formatter does not invent a base: an absolute input remains absolute, while a relative input retains the existing expanded/normalized fallback produced by `shell_native_absolute_path`.
7. The same rule applies in every state of these blocklist surfaces:
   - read-files request, success, grouped-result, and partial-failure rows;
   - grep search paths;
   - file-glob and file-glob-v2 search directories;
   - search-codebase repo roots and result file rows, with both `SearchCodebaseUI` enabled and disabled;
   - upload-artifact request paths and successful-result file paths.
8. Line-range suffixes remain unchanged and attached to the formatted path, including grouped ranges such as `src/lib.rs (10-20, 40-45)`.
9. Shell-native separators and roots are preserved: native Windows rows use `\`, Unix/WSL/MSYS2 rows use `/`, and absolute fallback retains the corresponding drive, UNC, or Unix root syntax.
10. Rows that are currently clickable remain clickable. Clicking a relative read-files or search-codebase result opens the same absolute target and line range as before; this ticket does not add click behavior to rows that are currently plain text.
11. A completed row never changes merely because the terminal later changes cwd. Restoring the conversation also produces the same text because the exchange's persisted cwd is reused.
12. Path formatting changes presentation only. Permission checks, filesystem access, search scope, upload source, skill detection, telemetry target paths, and action results continue using their existing absolute execution targets.
13. Skill references (`ReadSkill`), edit-diff file labels (`RequestFileEdits`), code-block source annotations, MCP resource URIs, and file paths in arbitrary assistant prose are out of scope because they use distinct reference/display contracts rather than the file-tool target rows above.

*Design alternatives:*
- *Cwd ownership* — (a) **Persisted exchange / AIBlock snapshot (chosen):** it already represents the cwd sent with the request that produced the tool calls, is immutable for the row, and survives restore. Pros: no schema migration; live and restored rendering share one source; later `cd` cannot rewrite history. Cons: actions within one exchange share a cwd rather than carrying independent values. This matches the existing exchange execution/display model and the ticket's stability requirement. (b) Add `invocation_cwd` to every `AIAgentAction`: finer theoretical granularity, but it requires API/conversion/persistence changes and backfill behavior while providing no additional known user-visible correctness for this flow. Rejected.
- *Relative-path algorithm* — (a) Call `warp_util::path::to_relative_path` and count leading `ParentDir` components: small and already tested on host-native Unix/Windows. Rejected as the canonical implementation because blocklist paths may be Unix-encoded on a Windows host under WSL/MSYS2; the helper explicitly declines WSL and `std::path` can emit host rather than shell separators. (b) **Use the same `typed_path` encoding as `shell_native_absolute_path` and walk at most two cwd ancestors (chosen):** normalize the target/cwd, try `strip_prefix(cwd)`, then `strip_prefix(cwd.parent())` and `strip_prefix(cwd.parent().parent())`, prefixing the remainder with the corresponding `..` components. If none matches, keep the absolute target. Pros: directly encodes the threshold, remains lexical/no-I/O, preserves shell syntax, and naturally rejects incompatible roots. Cons: introduces a small helper rather than reusing `to_relative_path` directly.
- *Formatting ownership* — (a) Format independently in each renderer: locally simple, but duplicates threshold/root behavior and risks mismatched link character ranges. Rejected. (b) **Add a canonical formatter in `crates/ai/src/paths.rs`, then use it from `FileLocations` and the remaining scalar-path renderers (chosen):** all surfaces share one policy while line-range composition stays in `FileLocations`. `render_read_files_text` becomes explicitly display-only. Pros: one policy and test matrix; existing execution-path helpers remain untouched. Cons: callers that currently pass raw paths must be made explicit.
- *Click targets* — (a) Replace string link detection with an explicit `{ display, absolute }` model across every row: strongest separation, but broad UI/API churn for a presentation fix. Rejected. (b) **Keep existing `DetectedLinkType::FilePath { absolute_path, ... }` and resolve the new relative display string with the snapshotted cwd (chosen):** the detector already accepts cwd and stores an absolute target. Require renderer/detector tests to prove their strings and character ranges stay aligned.

*Open questions resolved:*
- *What exactly is “cwd at invocation time”?* The cwd snapshot stored on the exchange that produced the action and copied into its `AIBlock`. Live creation captures it with the request; restore reads the same `exchange.working_directory`. It is not the terminal's cwd at repaint/click time.
- *Do individual actions need their own persisted cwd?* No. All actions in one model output belong to the same exchange/request context, and the existing UI/persistence boundary is per exchange. Adding action schema solely for this display change is unnecessary.
- *How is the two-level threshold counted?* Count only leading parent components in the candidate relative path. Descendant depth is unrestricted. `../sibling`, `../../sibling`, `..`, and `../..` are allowed; any candidate beginning with three `..` components is rejected in favor of absolute.
- *What is shown for the cwd itself?* `.`. This is the shell-native relative representation and is already expected by existing path-helper tests.
- *What happens when cwd is absent or paths are incompatible?* Preserve the safe fallback from the absolute resolver; never invent a relative path. An already absolute input stays absolute. Incompatible drives/roots remain absolute.
- *Are symlinks resolved before choosing display text?* No. Display formatting is lexical and preserves the path the tool used. Link detection may retain its existing canonical/existence checks for opening the target, but those checks do not rewrite display text.
- *Which extra path surfaces are included?* Upload-artifact request and successful-result paths are included because they are local file-tool targets rendered in the same blocklist action area. `ReadSkill`, edit diffs, code blocks, MCP URIs, and arbitrary prose are explicitly excluded under behavior invariant #13.
- *Does this ticket make every path clickable?* No. It preserves click behavior on rows that already support it and verifies their absolute targets; it does not add interactions to plain-text grep/glob/upload rows.

*Risks / blast radius:*
- *Double formatting:* a relative result passed through `shell_native_absolute_path` becomes absolute again. Making `render_read_files_text` display-only and testing its exact text is mandatory.
- *Renderer/link mismatch:* link handlers index by displayed character ranges. Search-codebase's feature-enabled view and the block's completion-time detector must derive identical formatted strings in identical order.
- *Cross-platform roots:* Windows drives/UNC prefixes and Unix paths on Windows-hosted WSL/MSYS2 sessions must not be compared with the wrong encoding. The shared helper must use the same `use_unix_paths` decision as the absolute resolver.
- *No-cwd and restored legacy data:* older exchanges may lack `working_directory`; those rows must keep their existing path string rather than panic or adopt the viewer's current cwd.
- *Search result grouping:* retain the current grouping/order behavior in each feature-flag path; path shortening must not merge, split, reorder, or change line ranges beyond existing behavior.
- *Upload result provenance:* format only a successful result filepath that represents the same local upload source. Artifact identifiers, descriptions, errors, and remote artifact URLs remain unchanged.
