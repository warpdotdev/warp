*Spec: TUI input focus during agent tool-call drafting*

== PRODUCT ==
*Summary:* When the TUI agent is still streaming/preparing a shell command, file edit, question, or similar tool call, the session input remains visible but currently stops accepting keystrokes because a newly created option selector takes focus before a permission request is active. The input must remain visible, focused, and usable until a permission card is actually rendered for a blocked front-of-queue action.

*Key design choices:* Remove implicit focus from generic option-page initialization; keep focus transitions explicit at the points where a selector becomes an intentional interactive surface; preserve the existing blocked-card layout and focus handoff instead of redesigning the session layout.

*Behavior* (numbered, testable invariants):
1. During the pre-Blocked drafting/streaming phase, the session input is rendered and remains the focused keyboard responder even after a stateful shell-command, file-edit, or ask-question child view is materialized.
2. A keystroke delivered during drafting updates the input buffer exactly as it does when no tool child exists; initializing a hidden permission selector must not consume or reroute that keystroke.
3. A permission selector may be constructed while drafting, but it must not acquire focus merely because its option page was initialized.
4. Once the front-of-queue action is actually `AIActionStatus::Blocked` and its approval UI is rendered, existing behavior is retained: the input may be suppressed, the approval card is visible, and its interactive surface owns focus and responds to its keybindings. This includes the shell-command approval card's inline command editor, the file-edit approval card, and an `AskUserQuestion` card while it is actively awaiting answers.
5. Active approval focus is not weakened by the fix: the shell-command card's selector initially owns focus and can transfer focus to its inline command editor for editing before approval; the file-edit card remains interactive; and the AskUserQuestion selector remains focused and usable while answers are awaited.
6. When a blocked permission action resolves or is dismissed, focus returns to the session input through the existing blocker synchronization path, with the input draft/cursor/selection preserved.
7. Ask-question and orchestration configuration flows retain intentional interaction: an ask-question selector is focused when the action is waiting for answers, and opening an orchestration configuration page explicitly focuses that page's selector.

== TECH ==
*Context:* At commit `c353a2a4121521058bb73e8417c3a1b357c2e860`, `TuiOptionSelector::reset_interaction_for_page` resets page state and unconditionally calls `ctx.focus_self()` (`crates/warp_tui/src/option_selector.rs:386-398`), and `set_page` invokes that reset (`crates/warp_tui/src/option_selector.rs:400-410`). `TuiAIBlock::sync_action_views` creates stateful children as streamed action output appears (`crates/warp_tui/src/agent_block.rs:391-590`): shell-command and file-edit children eagerly create `TuiPermissionPrompt` instances (`crates/warp_tui/src/tui_shell_command_view.rs:145-185`, `crates/warp_tui/src/tui_file_edits_view.rs:219-270`), while ask-question children call `selector.set_page` during `show_current_question` (`crates/warp_tui/src/tui_ask_question_view.rs:109-145,204-230`). Each of those initialization paths therefore focuses a selector before a visible blocker exists. `TuiAIBlock::active_blocking_child` only returns a blocker when the action is front-of-queue, `AIActionStatus::Blocked`, and the child has an active prompt (`crates/warp_tui/src/agent_block.rs:790-840`); `TuiTerminalSessionView` suppresses input only for that derived blocker (`crates/warp_tui/src/terminal_session_view.rs:1451-1479,2849-2960`). The session's blocker focus synchronization (`crates/warp_tui/src/terminal_session_view.rs:458-496,1474-1482`) is therefore not the cause of the pre-Blocked defect. `TuiPermissionPrompt` explicitly focuses on `ActionBlockedOnUserConfirmation` (`crates/warp_tui/src/tui_permission_prompt.rs:128-185`), and `TuiAskQuestionView` explicitly focuses when the action is blocked (`crates/warp_tui/src/tui_ask_question_view.rs:109-145`); those paths are intentional and must remain. `BlocklistAIActionModel::get_action_status` exposes the queue/status distinction used by these conditions (`app/src/ai/blocklist/action_model.rs:614-660`), while actual confirmation events are emitted from the action execution path (`app/src/ai/blocklist/action_model.rs:971-1030`).

*Root cause:* Constructor-time selector page setup has a side effect (focus) that is valid only when the selector is intentionally entering an interactive mode. Streaming a tool-call child creates the selector before the action reaches the visible blocked state, so the focus side effect targets a selector that is not rendered as the active blocker. The input is still present because `active_blocking_child` is `None`, producing a visible-but-unusable input. The fix boundary is exactly this pre-Blocked `ctx.focus_self()` side effect; no focus logic that runs at or after the `Blocked` transition may be removed or changed.

*Design alternatives*:
- *Selected — remove only the implicit focus side effect from `reset_interaction_for_page` and add explicit focus at intentional interaction entry points.* This is the smallest change, centralizes page-state reset without hidden focus mutation, and preserves every focus transition at or after `Blocked`, including the shell-command selector/inline editor, file-edit approval card, and AskUserQuestion selector. Callers that enter orchestration configuration must explicitly focus their selector after `open_page`; blocked permission and ask-question callers already have explicit focus paths.
- *Add a `focus` argument or separate focused-page API to `set_page`.* This makes intent visible at every call site, but expands the API and requires auditing every page transition for a boolean choice. It is unnecessary for the narrow defect if explicit focus is added only where a caller actually enters interaction.
- *Keep selector focus and make `TuiTerminalSessionView` repeatedly refocus the input until a blocker is active.* This couples the session to hidden child construction, risks fighting legitimate editor/configuration focus, and leaves other hosts vulnerable to the same side effect.
- *Redesign the layout so input and blocker are always rendered together.* This is a larger UX change, does not address the hidden-selector focus race directly, and is outside APP-4907's pre-Blocked scope.

*Proposed changes:* Change `TuiOptionSelector::reset_interaction_for_page` so it resets selection, search/custom-text state, and layout without the single pre-Blocked `ctx.focus_self()` call. Preserve all existing explicit focus calls used by selection actions, custom-text editing, search navigation, acceptance-card return, `TuiPermissionPrompt`'s `ActionBlockedOnUserConfirmation` event, its shell-command inline-editor handoff, and `TuiAskQuestionView`'s blocked/waiting path. Add an explicit selector focus in `TuiOrchestrationBlock::open_page` because it is a user-driven transition into configuration and currently relies on the removed implicit focus. Do not change `TuiPermissionPrompt::focus`, shell-command editor focus, file-edit card focus, AskUserQuestion focus, `active_blocking_child`, `sync_blocker_focus`, input suppression, permission-card rendering, or action status semantics.

*Affected files:* `crates/warp_tui/src/option_selector.rs`; `crates/warp_tui/src/orchestration_block.rs`; focused call sites and regression tests in `crates/warp_tui/src/tui_permission_prompt.rs`, `crates/warp_tui/src/tui_ask_question_view.rs`, `crates/warp_tui/src/option_selector_tests.rs`, `crates/warp_tui/src/terminal_session_view_tests.rs`, `crates/warp_tui/src/tui_shell_command_view_tests.rs`, `crates/warp_tui/src/tui_permission_prompt_tests.rs`, `crates/warp_tui/src/tui_ask_question_view_tests.rs`, and `crates/warp_tui/src/orchestration_block_tests.rs`.

*Open questions resolved:* The scope is only pre-Blocked drafting focus loss, not a permission-card redesign; visible blocked prompts may continue hiding the input. The only focus operation removed is the constructor-time `ctx.focus_self()` from generic page initialization; focus logic at or after the `Blocked` transition is explicitly preserved. Shell-command approval cards (including their inline command editor), file-edit approval cards, and the AskUserQuestion selector while awaiting answers must remain focused and usable. Shell-command and file-edit eager prompt construction and the shared ask-question selector path are all in scope because they share the faulty initialization side effect. Generic tool calls that lazily create prompts only once blocked are a preserved comparison path. The requester, Harry Albert (`harryalbert`), is the reviewer for the shared implementation PR.

*Risks / blast radius:* `TuiOptionSelector::set_page` is shared by permission prompts, questions, orchestration configuration, search pages, and custom-text flows. Removing its one pre-Blocked implicit focus could leave a legitimately interactive page unfocused if its caller lacks an explicit focus transition; the implementation must audit all call sites and add focus only at user-driven entry points. Conversely, retaining any constructor-time focus in streamed child creation would preserve the bug. Active shell-command selector/editor focus, file-edit approval focus, and AskUserQuestion focus are protected behaviors and must have explicit regression coverage. No action queue, permission decision, input rendering condition, or blocker status logic should change.
