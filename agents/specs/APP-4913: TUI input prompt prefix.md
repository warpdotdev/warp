*Proposed change: Prefix the normal TUI input bar with `>`*

*Summary:* In normal agent-input mode, the live `TuiInputView` currently renders only the editable `TuiEditorElement`; add a cyan `>` at the beginning using the same two-cell gutter positioning as shell mode's `!` in Figma node 1170-21099. Shell-mode input keeps its existing `!` gutter and behavior.

*Key design choices:* Mirror `shell_element` exactly for the normal-mode gutter: a plain-cyan `>` in a two-cell gutter (glyph plus one right-padding cell) followed by the unchanged flex editor. Do not reuse the submitted-input background/bold token, change the live input's background/border/spacing, add a second prefix in shell mode, or gate this small parity change behind a feature flag.

*Design alternatives:*
- *Prefix shape/position* — (chosen) mirror the shell gutter: a one-cell `>` plus one right-padding cell followed by a flex editor; a one-cell marker with no gap would not match `!` positioning, while inserting `>` into the editor buffer would corrupt submitted text, cursor offsets, selection, and shell-mode detection.
- *Styling/background* — (chosen) use the plain cyan `accent_text_style()` and leave the live input's current background, border, and spacing unchanged; using `input_prefix_style()` would add the explicitly rejected bold/tinted treatment, and applying `input_background()` to the editor would be an unrelated visual change.
- *Shell-mode interaction* — (chosen) shell mode remains `!` only; the normal branch is not composed around `shell_element`; showing `> !` would make the mode ambiguous and change an already-tested affordance.
- *Gutter interaction* — (chosen) make the two-cell gutter a `TuiHoverable` using the existing `prefix_mouse_state`, and dispatch `SetCursor { offset: 1 }` on click as the shell gutter does; making it click-through would leave a visible prompt area that cannot place the cursor at the beginning.
- *Feature rollout* — (chosen) always render in normal mode; this is a localized visual parity change with no independent rollout or migration requirement.

*Root cause / approach:* `TuiInputView::render` currently returns `render_input(ctx)` in the non-shell branch (`crates/warp_tui/src/input/view.rs:424-435 @ 19512e441b6a1e4ca06fbf85582bbdfbf0108778`), whereas `shell_element` already composes a two-column gutter (glyph plus one right-padding cell) and flex editor (`crates/warp_tui/src/input/view.rs:404-421 @ 19512e441b6a1e4ca06fbf85582bbdfbf0108778`). Add the analogous normal-mode composition with `TuiText::new(">")`, `accent_text_style()`, the same right padding, hover/click handler, and flex child. Keep the editor as a separate flex child so its own layout/hit-testing starts after the two-cell gutter; the existing session border, horizontal padding, background, and editor styling remain unchanged (`crates/warp_tui/src/terminal_session_view.rs:3078-3087 @ 19512e441b6a1e4ca06fbf85582bbdfbf0108778`).

*Affected files:* `crates/warp_tui/src/input/view.rs`; `crates/warp_tui/src/input/view_tests.rs`. No submitted-input renderer, GUI code, server code, persistence, style-token, or feature-flag files change.

*Open questions resolved:*
- The glyph is ASCII `>` in the normal-mode accent/cyan color. It occupies the first cell of the existing shell-style two-cell gutter; the second cell is the existing right-padding gap, not a new input-bar spacing rule.
- The prefix uses plain `accent_text_style()` only. The live input container/editor keeps its current background, border, spacing, and editor styling; `input_background()` and the bold `input_prefix_style()` are not used.
- Normal mode always shows the prefix, including an empty buffer. Shell mode continues to render `!` only, including autodetected and explicitly locked shell mode.
- The gutter is interactive and places the cursor at the editor's beginning without starting a drag selection, matching the existing shell gutter contract. Editor clicks are still measured relative to the flex child after the two-cell gutter.
- Only the first visual row contains `>`. The editor's continuation/wrapped rows start at the same two-column offset under the text; no repeated `>` is rendered.
- No feature flag is required.

*Risks / blast radius:* The flex gutter reduces the editor's offered width by two columns and shifts the normal-mode cursor and mouse hit-test coordinates; the change must use the same element composition and offset semantics as the existing shell gutter. Existing `!` entry/exit, selection, scrolling, submission, inline menus, and terminal-session border layout must remain unchanged. The submitted-input renderer and shared style tokens must not change.
