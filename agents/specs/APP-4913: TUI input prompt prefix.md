*Proposed change: Prefix the normal TUI input bar with `> `*

*Summary:* In normal agent-input mode, the live `TuiInputView` currently renders only the editable `TuiEditorElement`; add a two-cell prompt gutter containing `> ` before that editor, matching the submitted-input marker in Figma node 1170-21099. Shell-mode input keeps its existing `! ` gutter and behavior.

*Key design choices:* Use the existing `INPUT_PREFIX` token (`> `) and `input_prefix_style()` so the live prompt marker uses the same glyph, spacing, accent, bold treatment, and prefix-cell tint as submitted input. Compose the marker as a two-cell gutter plus a flex editor, just like `shell_element`, so continuation rows and cursor/mouse coordinates remain aligned. Do not add a second prefix in shell mode, tint the entire live input container, or gate this small parity change behind a feature flag.

*Design alternatives:*
- *Prefix shape* — (chosen) a two-cell `> ` gutter followed by a flex editor, reusing `INPUT_PREFIX`; (a) a one-cell `>` with no gap would put the cursor/text directly against the glyph and diverge from submitted input; (b) inserting `> ` into the editor buffer would corrupt submitted text, cursor offsets, selection, and shell-mode detection.
- *Styling/background* — (chosen) apply `TuiUiBuilder::input_prefix_style()` to the two prefix cells while preserving the live input's current unfilled bordered container; (a) `accent_text_style()` would lose the existing submitted-marker bold/tint token; (b) applying `input_background()` to every live editor cell would broaden the request and change the current input surface beyond the prefix.
- *Shell-mode interaction* — (chosen) shell mode remains `! ` only; the normal branch is not composed around `shell_element`; showing `> !` would make the mode ambiguous and change an already-tested affordance.
- *Gutter interaction* — (chosen) make the two-cell gutter a `TuiHoverable` using the existing `prefix_mouse_state`, and dispatch `SetCursor { offset: 1 }` on click as the shell gutter does; making it click-through would leave a visible prompt area that cannot place the cursor at the beginning.
- *Feature rollout* — (chosen) always render in normal mode; this is a localized visual parity change with no independent rollout or migration requirement.

*Root cause / approach:* `TuiInputView::render` currently returns `render_input(ctx)` in the non-shell branch (`crates/warp_tui/src/input/view.rs:404-435 @ 19512e441b6a1e4ca06fbf85582bbdfbf0108778`), whereas `shell_element` already composes a two-column gutter and flex editor (`crates/warp_tui/src/input/view.rs:404-421 @ 19512e441b6a1e4ca06fbf85582bbdfbf0108778`). Add the analogous normal-mode composition, using the shared `INPUT_PREFIX` marker currently rendered for submitted input (`crates/warp_tui/src/agent_block_sections.rs:24-71 @ 19512e441b6a1e4ca06fbf85582bbdfbf0108778`) and `input_prefix_style()` (`crates/warp_tui/src/tui_builder.rs:181-205 @ 19512e441b6a1e4ca06fbf85582bbdfbf0108778`). Keep the editor as a separate flex child so its own layout/hit-testing starts after the two-cell gutter; the existing session border and one-cell horizontal padding remain unchanged (`crates/warp_tui/src/terminal_session_view.rs:3078-3087 @ 19512e441b6a1e4ca06fbf85582bbdfbf0108778`).

*Affected files:* `crates/warp_tui/src/input/view.rs`; `crates/warp_tui/src/input/view_tests.rs`; if needed to share the canonical marker without duplication, make the existing `INPUT_PREFIX` crate-visible or move it to the TUI input-style/token location used by both renderers. No GUI code, server code, persistence, or feature-flag files change.

*Open questions resolved:*
- The glyph is ASCII `>` followed by exactly one space (`> `), two terminal cells total, matching the submitted-input marker rather than a Unicode prompt variant.
- The prefix uses `input_prefix_style()` (accent foreground, bold, existing `input_background` on the prefix cells). The live input container/editor keeps its current background behavior; this request does not add a full-row tint.
- Normal mode always shows the prefix, including an empty buffer. Shell mode continues to render `! ` only, including autodetected and explicitly locked shell mode.
- The gutter is interactive and places the cursor at the editor's beginning without starting a drag selection, matching the existing shell gutter contract. Editor clicks are still measured relative to the flex child after the two-cell gutter.
- Only the first visual row contains `> `. The editor's continuation/wrapped rows start at the same two-column offset under the text; no repeated `>` is rendered.
- No feature flag is required.

*Risks / blast radius:* The flex gutter reduces the editor's offered width by two columns and shifts the normal-mode cursor and mouse hit-test coordinates; the change must use the same element composition and offset semantics as the existing shell gutter. Existing `!` entry/exit, selection, scrolling, submission, inline menus, and terminal-session border layout must remain unchanged. A shared-token visibility change must not alter submitted-input rendering.
