# CODE-1884: Voice input V0 — TUI `/voice` slash command

*Spec: Voice input V0 — start recording from the headless TUI `/voice` command, show an animated listening outline and footer status, and stop/transcribe with `Esc` or `Enter`*

== PRODUCT ==
*Summary:* Add a no-argument `/voice` slash command to the headless TUI composer in `crates/warp_tui`. Selecting the command or submitting an exact `/voice` immediately starts the existing microphone recorder. While the slash-originated session is listening, the input outline continuously transitions between the theme's `cyan_overlay_2` and `Lilac-600` colors and the bottom-left footer/status row says `listening to voice input... · esc or enter to stop`; the input gutter stays empty because the existing shell `!` gutter is suppressed and no voice glyph replaces it. `Esc` or `Enter` stops recording for transcription and the footer says `Transcribing...` until the result is inserted into the composer. Recording uses the existing cross-platform audio backend and is supported on macOS, Linux, and Windows; macOS-specific code is limited to detecting the operating system's microphone authorization state. This V0 does not add GUI behavior, CLI-agent rich-input behavior, or cloud-agent composition behavior.

*Key design choices:* Register `/voice` only when the shared command registry is built for `SettingsMode::Tui`; reuse the app's existing `voice_input::VoiceInput` and `VoiceTranscriber` singletons with the same generation-scoped `VoiceInputLifecycle` used by GUI voice surfaces; reject superseded callbacks by matching their generation and retain the transcription handle only to cancel an in-flight transcription request; render the listening state through an animated input outline plus a replacing footer status while suppressing (but not replacing) the input's shell `!` gutter; handle `Esc` and `Enter` in `TuiInputView` for the TUI slash-originated session, preserving inline-menu priority and normal submission while voice is idle.

*Behavior* (numbered, testable invariants):
1. In a logged-in, local TUI session with AI enabled, voice team policy enabled, the `voice_input` capability compiled, and voice requests available, `/voice` appears in the TUI slash menu as a no-argument command. It is absent from GUI command results because the command is registered only for `SettingsMode::Tui`, absent from cloud-agent composition, and absent from the CLI-agent rich-input allowlist.
2. The command accepts no argument. Submitting `/voice` or `/voice` followed only by whitespace starts recording; `/voice text` does not start a session and reports the TUI's transient invalid-command hint. Selecting the menu row executes immediately, clears the command text, and never submits `/voice` as an AI prompt.
3. A successful selection/submission starts exactly one `VoiceInput::start_listening` session using the existing recorder and transcriber infrastructure. The TUI controller enters `Listening` only after the recorder starts successfully and emits a redraw event; it does not create a second recorder or a TUI-specific transcription service.
4. While slash-originated state is `Listening`, the input gutter is blank (the existing shell `!` is suppressed and no voice glyph/status is rendered), the input outline continuously and smoothly transitions from `cyan_overlay_2` to `Lilac-600` and back, and the footer's first/only status text is `listening to voice input... · esc or enter to stop`. While state is `Transcribing`, the outline stops animating, the input gutter remains blank, and the footer's status text is exactly `Transcribing... · esc to cancel`. The footer status remains visible even when the composer buffer is non-empty.
5. Voice status has precedence over the existing shell affordances: a voice-active footer never renders `SHELL_MODE_HINT` (`shell mode · esc to exit`), and a voice-active input never renders the `!` shell gutter or any replacement glyph. With voice stopped, shell mode continues to render both existing affordances. Other existing footer priorities (for example, ctrl-c exit confirmation) remain unchanged unless the implementation documents and tests a direct conflict.
6. Pressing `Esc` or `Enter` while slash-originated `Listening` calls the successful `VoiceInput::stop_listening` path, not `abort_listening`, and transitions the TUI controller to `Transcribing` while WAV conversion/transcription is pending. The input consumes the key so `Esc` does not also exit shell mode and `Enter` does not submit the composer. `Esc` retains the transcription-cancellation behavior below.
7. Pressing `Esc` while `Transcribing` cancels any retained transcription request, returns the lifecycle to idle, and shows a concise cancellation hint. If WAV conversion is still owned by the shared recorder, the backend remains active until that internal conversion finishes and its result is ignored unless its generation still matches. An inline menu, if one is open due to a stale redraw, still gets the existing first-Escape dismissal priority.
8. On a successful transcription, the returned text is inserted into the existing TUI composer through `CodeEditorModel`/`user_insert`, the voice state returns to idle, the voice footer disappears, the normal shell gutter/footer behavior resumes, and normal agent-input rendering resumes without submitting the text automatically. The `/voice` command text is not retained in the buffer.
9. On microphone denial, recorder/WAV failure, missing transcriber, quota exhaustion, or server transcription failure, the controller aborts and clears any retained transcription handle, returns to idle, clears the voice affordance, and uses a concise `TransientHint` in the TUI footer.
10. Existing TUI slash-command selection, exact-match parsing, shell-mode entry/exit, inline-menu Escape priority, footer transient hints, and all non-voice GUI voice behavior remain unchanged. A voice session is not available while a CLI agent owns input or while the active routing is cloud-agent composition.
11. TUI voice recording is platform-neutral across macOS, Linux, and Windows. All platforms use the shared `cpal` recorder for microphone discovery, stream creation, and device errors. The headless platform delegate additionally queries AVFoundation authorization on macOS because a denied macOS capture stream can otherwise appear valid while returning silent audio; non-macOS platforms defer permission and device validation to `cpal`.

*Headless-TUI feasibility findings (verified against the source at the pinned commit):*
- The recorder is not coupled to a GUI `ViewContext`, `WindowHandle`, GPU scene, or GUI editor. `VoiceInput::start_listening` takes only `&mut ModelContext<Self>` plus `VoiceInputToggledFrom`; it calls native `cpal::default_host`, obtains the default input device/config, builds/plays a stream, feeds samples through `ModelContext::spawn_stream_local`, resamples with `rubato::SincFixedIn`, and stores a `VoiceSession` backed by a `futures::channel::oneshot`. `stop_listening` pauses cpal, converts the resampled samples to 16 kHz/16-bit mono WAV/base64 in a spawned future, and sends `VoiceSessionResult`; these operations are model/background-executor APIs that the headless event loop can drive.
- `VoiceInput` consults `ctx.microphone_access_state()` before creating the active stream, and the platform API remains frontend-neutral through `warpui_core::platform::Delegate`. The headless delegate queries `AVCaptureDevice::authorizationStatusForMediaType` on macOS so explicit denial or restriction is rejected before accepting a silent capture stream. On Linux and Windows, the delegate does not pre-reject recording: the shared `cpal` path remains responsible for discovering the default input device, reading its configuration, building the stream, and returning device/backend failures.
- The headless permission seam must remain cross-platform and must not require a GUI window or GUI event loop. macOS authorization detection is a platform-specific guard around the same shared recorder, not a macOS-only recording implementation. Linux and Windows must continue through the normal `cpal` recorder path, with unavailable microphones or audio backends reported through the existing TUI transient-error flow.
- `ModelContext::spawn_stream_local` and `ModelContext::spawn` deliver callbacks through the app's headless event loop, which is present in `warpui/src/platform/headless/event_loop.rs`; no GPU rendering or `WindowHandle` is required. The live path therefore needs a running `run_tui` event loop, but not a GUI display/event loop. Tests must inject a fake backend/session so they do not require cpal hardware.
- `VoiceTranscriber` is a singleton wrapper around `Arc<dyn Transcriber>`. `ServerVoiceTranscriber` implements that trait by building the existing `TranscribeRequest` and calling `ServerApi::transcribe` (the `/ai/transcribe` boundary); this code has no GUI/editor dependency. `initialize_app` registers `VoiceInput` only under `#[cfg(feature = "voice_input")]`, always registers `VoiceTranscriber`, and registers `AIRequestUsageModel`; `run_tui` invokes that same initialization before mounting the TUI.
- The current Cargo graph does not link the recorder into the TUI: `app`'s `tui` feature is only `warpui_core/tui`, `gui` is the feature that currently enables `voice_input`, and `crates/warp_tui/Cargo.toml` depends on `warp` with only `image_as_context,tui`. The implementation must explicitly enable `voice_input` in both the app TUI feature and the `warp_tui` `warp` dependency (or prove an equivalent unified feature path), then verify with Cargo metadata/build output that `cpal`, `rubato`, `hound`, and the app singleton registrations are present in the TUI binary.
- The production feasibility gate is therefore: feature wiring/build succeeds; the headless permission shim reports the correct state; cpal can open a real default input device on the target; the existing resampling/WAV/session pipeline completes; and `ServerVoiceTranscriber` can reach `/ai/transcribe`. Until all four are demonstrated, `/voice` must fail cleanly with the TUI transient hint and must not pretend that recording started.

*Design alternatives*:
- *Register `/voice` in the shared command list for every frontend.* Rejected: this is explicitly TUI-only and `SettingsMode::Tui` already provides a registry branch for TUI-only commands. The chosen placement prevents GUI menu exposure while still allowing the normal TUI data-source path to discover it.
- *Implement a second recorder/transcription client in `warp_tui`.* Rejected: it would duplicate microphone permissions, WAV conversion, authentication, quota handling, and server API behavior. The chosen controller adapts the already-registered app singletons.
- *Put all state in `TuiTerminalSessionView` and make the input view emit a generic Escape event.* Rejected: Escape precedence and render state would be split across two owners. The chosen `TuiVoiceInputModel` is shared by the session owner and `TuiInputView`, emits explicit state events, and lets the input consume contextual Escape before shell-mode exit.
- *Use the input `!` gutter as a voice indicator.* Rejected: the requester explicitly requires no voice indicator in the gutter. The listening outline and footer row provide the voice affordance; the existing shell `!` is suppressed during voice so the gutter is blank, with separate render-to-lines assertions.
- *Use the real microphone/network in unit tests.* Rejected: it is nondeterministic and unsuitable for CI. Lifecycle tests drive the production state primitive/model directly, while recorder availability and async surface behavior are separated behind existing app singletons, generation checks, and the retained transcription handle.
