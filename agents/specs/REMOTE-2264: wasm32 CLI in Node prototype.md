# Spec: wasm32 Oz CLI in a Node runtime

== PRODUCT ==
*Summary:* Build a narrow, headless prototype that compiles the Oz/ Warp CLI code to `wasm32-unknown-unknown`, loads the resulting module in Node, and exercises one authenticated agent request with a prompt such as `hello`. The prototype is an feasibility probe for a sandboxless orchestrator: it must demonstrate the useful networked MAA slice while making the missing filesystem, process, PTY, GUI, and shell capabilities explicit instead of hiding them behind browser polyfills.

*Key design choices:* Use a dedicated `#[wasm_bindgen]` headless entrypoint owned by the Warp app/agent code, not `warp::run()` or the browser GUI bootstrap; make Node host capabilities explicit (HTTP/SSE, timer, and stdout/stderr/event sinks); and keep the first slice intentionally constrained to prompt → server request → streamed response, with every filesystem/shell/process/PTY/MCP/indexing tool disabled or rejected deterministically. Treat `wasm32-unknown-unknown` with host bindings as the primary target. Evaluate WASI as an alternative, but do not switch the prototype to WASI or imply that WASI supplies browser-style networking.

*Behavior* (numbered, testable invariants):
1. A reproducible build produces a `wasm32-unknown-unknown` artifact and Node loader from the repository's documented command; the build does not depend on a browser page, DOM, canvas, clipboard, or GUI event loop.
2. A Node LTS harness imports the artifact, passes a prompt and API key/configuration explicitly to the exported entrypoint, and receives a structured completion or error result. The wasm module must not depend on `Args::from_env()` for argv/environment values.
3. The harness can run a `hello` prompt through the minimum MAA-style request/response loop. The loop sends the configured request to a mock HTTP/SSE endpoint (and, when credentials are supplied, to the configured Warp endpoint), consumes the response stream to a terminal event, and exposes the final text/status to Node.
4. Outbound HTTP uses an explicit Node host bridge. The bridge carries method, URL, headers, and body to Node's `fetch` implementation and returns status, headers, and streamed body chunks to wasm. API-key material is passed only in request/auth configuration and is never included in stdout, stderr, structured results, fixtures, or committed files.
5. stdout and stderr/log events are observable by the Node harness. The harness captures them separately and preserves event order; the prototype must not require browser-only `web_sys::console` or a global Warp web event bus to observe output.
6. Filesystem reads/writes, directory watchers, codebase indexing, subprocess/shell/PTY creation, third-party harnesses, MCP process startup, snapshots, and GUI initialization are not attempted in the supported slice. If the model requests one of those tools, the harness receives a stable `unsupported_capability` error/event and the run terminates or continues according to the documented policy; it must not silently execute a native fallback.
7. Missing/empty prompts, missing API keys, malformed host responses, non-2xx HTTP responses, SSE disconnects, and host/run timeouts produce deterministic structured errors and release all pending JS promises. A failed request cannot leave the Node process hanging.
8. The findings report records what compiled, what ran, every shim/stub required, the sufficiency limits of `wasm32-unknown-unknown`, the WASI comparison, and a recommendation for the next three engineering steps. It includes rough startup, memory, and performance observations gathered from the harness.

*Design alternatives*:
- *Full Warp binary plus DOM/browser polyfills in Node:* Rejected for the prototype. It would exercise GUI startup and browser-only `window`, `document`, canvas, clipboard, and event-listener code rather than the agent runtime, and would conceal the actual host capability boundary.
- *Dedicated wasm-bindgen entrypoint in the app/agent owner:* Selected. Add a target-specific `cdylib` export in the Warp app/agent ownership boundary, with a narrow function that accepts explicit config and a host object and returns a Promise/result. This reuses the existing MAA protocol/event types without invoking `warp::run()`, Clap argv parsing, or full `initialize_app`.
- *Direct Node HTTP calls from Rust through native APIs:* Rejected. `wasm32-unknown-unknown` has no syscalls and cannot use native sockets/processes; all network I/O must cross an explicit JS host boundary.
- *WASI as the primary target:* Rejected for this probe. WASI can standardize selected filesystem/stdio/time capabilities, but it does not automatically provide the browser-style fetch/SSE integration already assumed by the wasm HTTP path, nor does it make GUI/PTY/process features available. WASI may be built as a comparison artifact only if the target-specific code remains small and the report records exactly which capabilities it adds and which still require host APIs.
- *Reuse the complete `AgentDriver` immediately:* Rejected for the first slice. `AgentDriver::new` requires authenticated `AppContext` and creates a terminal driver; its setup includes cwd/FS, MCP, shell, snapshots, and sharing. The first slice instead uses the smallest existing MAA request/client loop and an allowlist with no local tools. A later phase can reintroduce driver services behind explicit host capabilities.

*Proposed changes:*
1. Add a target-specific wasm entrypoint in the Warp app/agent ownership boundary and a `cdylib` build target. Export one JS-callable async operation (name and generated binding may follow repository conventions) that accepts `{prompt, api_key, server_url, model/config, timeout}` plus a host capability object, validates the input, and returns structured terminal/error events.
2. Add a `wasm_node` target feature/module in the HTTP path. Define the host contract as `fetch(request) -> Promise<response>`, where request contains method, URL, headers, and bytes, and response exposes status, headers, and an async sequence of body chunks; include a host log/event sink for stdout/stderr. The Node harness implements this contract with Node LTS `fetch`, `AbortController`, and console/event capture. No browser DOM shim is permitted.
3. Build the minimum MAA request using the existing multi-agent API/client event model. The supported-tools list must contain no local filesystem, shell, PTY, process, indexing, snapshot, or third-party harness capability. Translate streamed server events into a stable JSON result and stable `unsupported_capability` errors.
4. Add `script/wasm/build-node` (or an equivalent repository-standard command with the same contract) that checks/builds `wasm32-unknown-unknown`, runs `wasm-bindgen --target nodejs`, and writes the wasm plus generated Node loader to a documented output directory. Add `script/wasm/node-harness.mjs` (or equivalent) that accepts `--prompt`, `--api-key`, `--server-root-url`, and timeout options, invokes the export, captures stdout/stderr, and exits nonzero on structured failure.
5. Add focused Rust/Node tests and a deterministic mock HTTP/SSE fixture. The fixture must cover a successful terminal event, unsupported tool request, non-2xx response, stream disconnect, malformed response, missing key, empty prompt, and timeout. The harness must redact API keys from captured output and test artifacts.
6. Produce a findings document or harness output artifact as part of the implementation run (not a second spec file) containing exact build/run commands, compile-failure categories, runtime shims, wasm32-versus-WASI comparison, feasibility outcome, and startup/memory/performance observations.

*Open questions resolved:*
- *Which target is primary?* `wasm32-unknown-unknown` with explicit Node host bindings is primary because it matches the existing browser wasm build and directly tests the constrained sandboxless model. WASI is a documented comparison, not an unvalidated substitution.
- *How are argv and API keys passed?* The JS export receives structured arguments; it does not call `Args::from_env()` and does not read `std::env`. The Node CLI may read `process.argv`/`WARP_API_KEY`, but only the harness may do so, and it passes the value into wasm without logging it.
- *How much agent functionality is required?* One prompt, one authenticated MAA request, streamed response consumption, terminal result, and deterministic unsupported-tool/error behavior are the minimum. Shell, real FS, PTY, watchers, indexing, MCP process startup, GUI, and snapshots are explicitly out of scope for the first slice.
- *Is a real Warp server required for tests?* No. The deterministic mock HTTP/SSE fixture is mandatory and proves the host boundary offline; a separately documented authenticated run against the configured Warp endpoint is an optional feasibility check when `WARP_API_KEY` is available.
- *What does “Node runtime” mean?* Use a supported Node LTS release with built-in `fetch` and `AbortController` (Node 20 or newer, matching the repository's existing Node-runtime minimum). No browser/DOM package is a required dependency.

*Risks / blast radius:* The app crate has broad GUI and platform dependencies, so an apparently small export can accidentally pull browser-only initialization or native-only code; keep all entrypoint and capability code behind target/feature gates and test import/run in a DOM-free Node process. The existing reqwest event-source path may require a Node-specific stream adapter. MAA protocol changes or auth setup may make a real-server run unavailable; preserve the mock fixture and report that limitation rather than weakening the success criteria. Single-threaded wasm execution can block the JS event loop if synchronous work or unbounded streams are introduced; all host I/O must be Promise-based with explicit cancellation/timeouts. The prototype is an experiment, not a security boundary: document that host bindings define the actual authority and that no local tool capability is exposed.

*Validation & verification criteria* (must ALL pass before merge):
1. *Target build:* From a clean checkout, `rustup target add wasm32-unknown-unknown` followed by the documented `./script/wasm/build-node` command succeeds and produces the wasm artifact plus Node loader; the command is recorded verbatim in the findings output. A direct `cargo check --target wasm32-unknown-unknown` using the prototype's feature set also passes.
2. *Node import smoke test:* `node script/wasm/node-harness.mjs --prompt hello --api-key "$WARP_API_KEY" --server-root-url <mock-url>` (or the exact documented equivalent) imports the generated loader in a DOM-free Node process, invokes the exported function, and exits deterministically rather than hanging.
3. *Explicit startup configuration:* A test invokes the export directly with a prompt/API key/config object and proves that changing the Node process argv/environment after invocation does not alter the wasm request; a missing API key and empty prompt each produce the specified structured input error. No API-key value appears in stdout, stderr, JSON results, fixtures, or logs.
4. *Host HTTP contract:* A Node test with a local mock server asserts the exact method, URL, headers, body, and abort behavior received from the wasm host bridge, and returns a streamed response in chunks. The wasm side consumes the chunks and emits a terminal result.
5. *MAA happy path:* The mock SSE fixture produces the minimal valid response sequence; the exported operation returns a success/terminal event containing the expected text, and stdout/stderr capture contains only the documented non-secret events. This verifies the in-process request/response loop, not merely a standalone HTTP request.
6. *HTTP and stream failures:* Tests cover non-2xx response, malformed response payload, stream disconnect, host rejection, and timeout/abort. Each returns the documented structured error, releases pending promises, and leaves Node able to exit cleanly.
7. *Capability boundary:* A fixture that asks for a filesystem read, filesystem write, shell/process command, PTY, indexing, MCP startup, snapshot, or third-party harness operation observes the stable `unsupported_capability` response. Instrumentation or the Node mock proves no filesystem API, child process, DOM API, browser event listener, GUI initializer, or native fallback was invoked.
8. *Logging/output boundary:* The harness captures stdout and stderr separately and preserves ordering. The Node path works without `window`, `document`, `canvas`, clipboard, `warpEmitEvent`, or browser `console` bindings; browser-only initialization is not reached.
9. *WASI comparison:* The findings output contains an executable comparison (or a recorded build/runtime limitation) for `wasm32-unknown-unknown` versus WASI, explicitly listing filesystem/stdio/network/stream/host-binding differences and why WASI does or does not improve this prototype. The recommendation remains checkable against observed results, not assumptions.
10. *Resource notes:* The harness records artifact size, cold-start time, peak/steady memory if available, request/stream duration, and any event-loop blocking. Measurements are labeled as prototype observations and include the Node version and build profile.
11. *Regression and adjacent behavior:* Focused Rust tests and Node harness tests pass, and the existing browser wasm build command remains valid (`./script/wasm/bundle --check-only` with its documented channel/features). No native CLI behavior is changed; the normal native `oz agent run --prompt hello --api-key "$WARP_API_KEY"` path remains covered by the relevant existing tests.
12. *Presubmit:* `./script/presubmit` passes from the repository root. If the full presubmit cannot run in the implementation environment, the PR must state the exact unavailable dependency and mark the result unverified rather than claiming success.
