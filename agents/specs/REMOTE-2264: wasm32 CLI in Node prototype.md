# Spec: wasm32 Oz CLI in a Node runtime

== PRODUCT ==
*Summary:* Build a narrow, headless proof-of-concept that compiles the Oz/Warp CLI agent path to `wasm32-unknown-unknown`, loads the resulting module in Node, and exercises one authenticated `AgentDriver` run with a prompt such as `hello`. This is an experiment for learning, not a production feature: the required proof is a working manual end-to-end run and a findings report, while tests or local fixtures are optional confidence aids rather than deliverables. The prototype's value is proving that a client-side driver can preserve the session-sharing boundary while making missing filesystem, process, PTY, GUI, and shell capabilities explicit instead of hiding them behind browser polyfills. A direct MAA request loop is an explicit fallback only if the AgentDriver path is blocked by a concrete, documented dependency.

*Key design choices:* Make the full `AgentDriver` path the primary experiment because a major reason to run the agent in wasm rather than on the server is client-side session sharing; use the existing headless `cli` feature/configuration instead of inventing a second CLI build mode; use a dedicated `#[wasm_bindgen]` entrypoint owned by the Warp app/agent code, not `warp::run()` or the browser GUI bootstrap; and keep application logic runtime-neutral by putting Node-specific HTTP/SSE behavior behind the existing `reqwest`/`http_client` transport boundary. Lean on existing wasm carve-outs (`local_fs`, `local_tty`, and target gates), adding only the narrow stubs/gates required by `AgentDriver`, `TerminalDriver`, MCP, snapshot, watcher, and indexing dependencies. Treat `wasm32-unknown-unknown` with host bindings as the primary target. Evaluate WASI as an alternative, but do not switch the prototype to WASI or imply that WASI supplies browser-style networking.

*Behavior* (numbered, testable invariants):
1. A reproducible build produces a `wasm32-unknown-unknown` artifact and Node loader from the repository's documented command; the build does not depend on a browser page, DOM, canvas, clipboard, or GUI event loop.
2. A Node LTS harness imports the artifact, passes a prompt and API key/configuration explicitly to the exported entrypoint, and receives a structured completion or error result. The wasm module must not depend on `Args::from_env()` for argv/environment values.
3. The harness first attempts a `hello` prompt through `AgentDriver`: it constructs the driver, reaches the in-process MAA loop, and exercises the session-sharing registration/transport boundary. The run consumes the response stream to a terminal event and exposes the final text/status to Node. If this path cannot run after the existing wasm carve-outs and narrowly-scoped stubs are applied, the findings must name the concrete blocker and then exercise the direct MAA fallback.
4. Outbound HTTP remains expressed through the existing runtime-neutral `reqwest`/`http_client` API. Its wasm transport adapter crosses an explicit host boundary, where the Node harness supplies `fetch` (method, URL, headers, body; status, headers, and streamed chunks) without exposing Node types or branching into Node-specific behavior in `AgentDriver` or other application logic. API-key material is passed only in request/auth configuration and is never included in stdout, stderr, structured results, local captures, or committed files.
5. stdout and stderr/log events are observable by the Node harness. The harness captures them separately and preserves event order; the prototype must not require browser-only `web_sys::console` or a global Warp web event bus to observe output.
6. Filesystem reads/writes, directory watchers, codebase indexing, subprocess/shell/PTY creation, third-party harnesses, MCP process startup, snapshots, and GUI initialization are not attempted in the supported slice. `AgentDriver` dependencies on `TerminalDriver`, MCP, snapshot writers, filesystem watchers, indexing, and session setup are individually wasm-gated, stubbed, or replaced by explicit host callbacks. If the model requests one of those tools, the harness receives a stable `unsupported_capability` error/event and the run terminates or continues according to the documented policy; it must not silently execute a native fallback.
7. Missing/empty prompts, missing API keys, malformed host responses, non-2xx HTTP responses, SSE disconnects, and host/run timeouts produce deterministic structured errors and release all pending JS promises. A failed request cannot leave the Node process hanging.
8. The findings report records what compiled, what ran, every shim/stub required, the sufficiency limits of `wasm32-unknown-unknown`, the WASI comparison, and a recommendation for the next three engineering steps. It includes rough startup, memory, and performance observations gathered from the harness.

*Design alternatives*:
- *Full Warp binary plus DOM/browser polyfills in Node:* Rejected for the prototype. It would exercise GUI startup and browser-only `window`, `document`, canvas, clipboard, and event-listener code rather than the agent runtime, and would conceal the actual host capability boundary.
- *Dedicated wasm-bindgen entrypoint in the app/agent owner:* Selected. Add a target-specific `cdylib` export in the Warp app/agent ownership boundary, with a narrow function that accepts explicit config and a host object and returns a Promise/result. This reuses the existing MAA protocol/event types without invoking `warp::run()`, Clap argv parsing, or full `initialize_app`.
- *Direct Node HTTP calls from Rust through native APIs:* Rejected. `wasm32-unknown-unknown` has no syscalls and cannot use native sockets/processes; all network I/O must cross an explicit JS host boundary.
- *WASI as the primary target:* Rejected for this probe. WASI can standardize selected filesystem/stdio/time capabilities, but it does not automatically provide the browser-style fetch/SSE integration already assumed by the wasm HTTP path, nor does it make GUI/PTY/process features available. WASI may be built as a comparison artifact only if the target-specific code remains small and the report records exactly which capabilities it adds and which still require host APIs.
- *Reuse the complete `AgentDriver` immediately:* Selected as the primary path despite its larger dependency surface. The client-side value of the prototype is session sharing, and session sharing is owned by the driver/conversation-consumer path. Existing `local_fs`, `local_tty`, and wasm target gates should make the work tractable; each remaining dependency must be gated, stubbed, or host-backed and reported. If a concrete blocker remains, use the direct MAA request/client loop as a fallback and record the lost session-sharing capability.

*Proposed changes:*
1. Add a target-specific wasm entrypoint in the Warp app/agent ownership boundary and a `cdylib` build target, using the existing headless `cli` feature/configuration as the starting point. Export one JS-callable async operation (name and generated binding may follow repository conventions) that accepts `{prompt, api_key, server_url, model/config, timeout}` plus host capabilities, validates the input, creates the `AgentDriver` with an explicit `AgentDriverOptions` profile, and returns structured driver/session/terminal/error events.
2. Make the AgentDriver setup wasm-safe by inventorying each dependency and using existing target carve-outs first: `local_fs` paths (cwd, file edits, snapshots, declarations, skill watchers, codebase indexing), `local_tty`/PTY and shell/process paths, MCP process startup, and any GUI/window/persistence bootstrap must be excluded or replaced with stable host-capability errors. Preserve the driver’s in-process MAA loop and session-sharing registration/consumer transport; provide only the minimal no-op or host-backed terminal/session implementation needed to prove the path.
3. Extend the wasm implementation of the existing HTTP/session-sharing transport boundary, without adding Node-specific concepts to `AgentDriver` or the MAA application logic. Define the host contract as `fetch(request) -> Promise<response>`, where request contains method, URL, headers, and bytes, and response exposes status, headers, and an async sequence of body chunks; add session-sharing connect/send/receive/close callbacks and a host log/event sink for stdout/stderr. The Node harness implements these contracts with Node LTS `fetch`, `AbortController`, and event capture. No browser DOM shim is permitted.
4. If the AgentDriver route is blocked by a concrete dependency that cannot be made wasm-safe within this prototype, implement a separate direct MAA fallback entrypoint using the existing multi-agent API/client event model. The fallback must be clearly labeled in code and findings, must retain the same host HTTP/output/error contract, and must state that it does not validate session sharing.
5. Add `script/wasm/build-node` (or an equivalent repository-standard command with the same contract) that checks/builds `wasm32-unknown-unknown`, runs `wasm-bindgen --target nodejs`, and writes the wasm plus generated Node loader to a documented output directory. Add `script/wasm/node-harness.mjs` (or equivalent) that accepts `--prompt`, `--api-key`, `--server-root-url`, and timeout options, invokes the AgentDriver-first export, captures stdout/stderr/session events, and exits nonzero on structured failure.
6. Run the manual proof against a real configured Warp endpoint. The required run must cover a successful AgentDriver terminal event with session registration and outbound streaming. Optional focused Rust/Node checks or an ephemeral local HTTP server may be added when they help isolate a transport or capability issue, but they are not required deliverables, must not be described as an MAA mock, and must not replace the real MAA run. If AgentDriver is blocked and fallback is used, record the concrete blocker and the missing session-sharing behavior. The harness must redact API keys from captured output and any local captures.
7. Produce a findings document or harness output artifact as part of the implementation run (not a second spec file) containing exact build/run commands, the AgentDriver dependency inventory and gating decisions, session-sharing observations, fallback/blocker evidence if applicable, compile-failure categories, runtime shims, wasm32-versus-WASI comparison, feasibility outcome, and startup/memory/performance observations.

*Open questions resolved:*
- *Which target is primary?* `wasm32-unknown-unknown` with explicit Node host bindings is primary because it matches the existing browser wasm build and directly tests the constrained sandboxless model. WASI is a documented comparison, not an unvalidated substitution.
- *How are argv and API keys passed?* The JS export receives structured arguments; it does not call `Args::from_env()` and does not read `std::env`. The Node CLI may read `process.argv`/`WARP_API_KEY`, but only the harness may do so, and it passes the value into wasm without logging it.
- *How much agent functionality is required?* The primary minimum is one prompt through `AgentDriver`, authenticated MAA response streaming, terminal result, session registration/transport, and deterministic unsupported-tool/dependency behavior. Shell, real FS, PTY, watchers, indexing, MCP process startup, GUI, and snapshots remain out of scope except for proving that the driver reaches stable capability errors. The direct MAA request/client loop is only the fallback and does not satisfy the session-sharing objective.
- *What validates the MAA path?* A real authenticated run against the configured Warp endpoint is the required proof. An ephemeral local HTTP server or captured response may be used only as an optional transport-debugging aid; it is not an MAA substitute, is not called a mock SSE fixture, and cannot establish AgentDriver/session-sharing viability on its own.
- *What does “Node runtime” mean?* Use a supported Node LTS release with built-in `fetch` and `AbortController` (Node 20 or newer, matching the repository's existing Node-runtime minimum). No browser/DOM package is a required dependency.

*Risks / blast radius:* The app crate has broad GUI and platform dependencies, so an apparently small export can accidentally pull browser-only initialization or native-only code; keep all entrypoint and capability code behind target/feature gates and test import/run in a DOM-free Node process. `AgentDriver` may still transit a native-only dependency that the existing `local_fs`/`local_tty` carve-outs do not cover; the implementation must identify that dependency and either add a narrow wasm gate/stub or record it as the concrete partial/negative outcome before using the fallback. Session-sharing transport may have browser/WebSocket assumptions and must be host-backed or explicitly blocked with evidence. The existing reqwest event-source path may require a Node-specific stream adapter. MAA protocol changes or auth setup may make a real-server run unavailable; report that limitation as a partial result rather than substituting an offline stream fixture. Single-threaded wasm execution can block the JS event loop if synchronous work or unbounded streams are introduced; all host I/O must be Promise-based with explicit cancellation/timeouts. The prototype is an experiment, not a security boundary: document that host bindings define the actual authority and that no local tool capability is exposed.
