*Spec: macOS video recording for computer use (REMOTE-2160)*

== PRODUCT ==
*Summary:* Add real video recording for computer use on macOS cloud runners (Namespace macOS VMs), mirroring the existing Linux `x11grab` recorder. Today `create_recorder()` on macOS returns a no-op (`RecordingError::Environment`), so computer-use recordings are unavailable on macOS. After this change, a macOS run with computer use enabled produces a real MP4 (H.264 / yuv420p) of the main display and publishes it through the same path Linux uses — provided the macOS sidecar image bundles ffmpeg (dev channel first, matching Linux).

*Key design choices:*
1. Mirror the Linux recorder, don't reinvent. The macOS recorder shells out to the same ffmpeg subprocess lifecycle (spawn → wait for first output → SIGINT to finalize → bounded by `-t`/`-fs`), swapping only the input device from `x11grab` (`$DISPLAY`) to `avfoundation` (`Capture screen 0:none`). Reuse the shared `RecordingConfig` / `RecordingOutput` / `RecordingError` types and the `RecordingHandle` lifecycle unchanged.
2. Select the avfoundation screen by NAME, not index. `"Capture screen 0"` is parsed directly by ffmpeg (`sscanf(ctx->video_filename, "Capture screen %d", ...)`) and is stable/English; the integer index shifts when the camera count changes (cameras precede screens in avfoundation's combined index space). Robust in headless Namespace VMs (one display, typically no cameras), and matches the macOS screenshot path's main-display-only behavior (`screencapture -m`).
3. Gate ffmpeg on the dev channel for BOTH platforms, consistently. macOS bundles a static darwin/arm64 ffmpeg into the `oz-macos-sidecar` application image only when `WARP_CHANNEL=dev`, mirroring `xvfb-sidecar/Dockerfile`'s existing dev-gate. Stable macOS ships without ffmpeg → recording stays unsupported (no-op) on stable, exactly as on stable Linux today. Promoting ffmpeg to stable is a separate, independent decision.
4. The committed spec lives in the `warp` repo; the change spans two repos — `warp` (the macOS recording substrate) and `warp-agent-docker` (bundling ffmpeg into the sidecar image). This PR carries the `warp` substrate plus this spec file; a coordinated PR on `warp-agent-docker` carries the packaging change. Both are described here as one contract.

*Behavior* (numbered, testable invariants from the consumer's view):
1. On macOS, `create_recorder()` returns a real recorder (not no-op) when ffmpeg is on PATH. `Recorder::start` launches an ffmpeg `avfoundation` capture of the main display and resolves only once capture is confirmed live (the output file has grown). `Recorder::stop` finalizes a playable MP4 and returns a `RecordingOutput` with a real path, duration, dimensions, and size.
2. On macOS, when ffmpeg is NOT on PATH (stable sidecar, or a dev image before bundling lands), `Recorder::start` fails fast with `RecordingError::Environment` naming ffmpeg — recording is reported unsupported, never silently produces nothing. Same shape as Linux without ffmpeg.
3. The macOS `WARP_MOCK_RECORDER` path is unchanged: setting it still returns the mock recorder (fixture-copy) for UI/publish-flow testing without real capture. The real recorder does not interfere with the mock path, and the mock path still takes precedence in `create_recorder()`.
4. The produced MP4 is playable and bounded: H.264 (`libx264`), `ultrafast` preset, `yuv420p` pixel format, `+faststart` movflag; dimensions forced even (the libx264/yuv420p requirement); duration ≤ `RecordingConfig.max_duration` and size ≤ `max_size_bytes`, enforced by ffmpeg `-t` / `-fs`. Encode settings identical to Linux recordings.
5. The recording captures the cursor and mouse clicks (`-capture_cursor 1 -capture_mouse_clicks 1`) so computer-use replay is intelligible.
6. Abandoned recordings are cleaned up: if a `RecordingHandle` is dropped without `stop`, the partial MP4 and its log are removed (ffmpeg is killed on drop). Same contract as Linux.
7. A recording ffmpeg finalizes produces `RecordingCompletionStatus::Completed`; one that exits before an explicit `stop` (limit reached or crash) produces `StoppedEarly`. `poll_exit` reports the kind. Same semantics as Linux.
8. Bundling: a dev `oz-macos-sidecar` image packages a static darwin/arm64 `ffmpeg` into `$TOOLS_DIR` (on PATH via the entrypoint), with sha256 verification matching the existing `download_*` helpers. A stable `oz-macos-sidecar` image does NOT package ffmpeg.
9. No regression to Linux recording or to macOS screenshots/input: the Linux `x11grab` recorder, the macOS `screencapture` screenshot path, and macOS `CGEventPostToPid` input are all unchanged. The server-side macOS dispatch/provisioning code is unchanged.

*Baseline:* This addendum extends the macOS substrate already shipped in this PR. It supersedes the baseline file-size readiness and fixed-`Capture screen 0` choices above for the follow-up fix; the existing encode settings, recording lifecycle, sidecar packaging contract, mock path, and Linux behavior remain in force.

== PRODUCT ==
*Summary:* A macOS recording start must not reject a capture merely because AVFoundation takes a while to deliver its first frame. The recorder must recognize real first-frame progress, retry only a transient startup failure a bounded number of times, and return an actionable non-retryable error for permission/device/selector failures. A successful capture must remain a playable recording of the intended display.

*Key design choices:*
1. Readiness is based on ffmpeg progress (`frame=`/`time=` after at least one encoded frame), not MP4 file size; the MP4 may remain zero bytes while AVFoundation is still opening the session.
2. Startup is supervised as three total attempts (the initial launch plus two retries), with 250 ms then 500 ms backoff and a 15 s per-attempt first-progress deadline. The total startup wall-clock budget is therefore bounded to 46 s, including backoff.
3. AVFoundation's `NSKVONotifying_AVCaptureScreenInput not linked into application` and `Configuration of video device failed, falling back to default` lines are benign warnings. They never trigger a retry or failure by themselves; permission, missing-device, and invalid-selector diagnostics fail fast.

*Behavior* (numbered, testable invariants):
1. `Recorder::start` resolves only after ffmpeg has reported its first encoded frame/progress, even if the output file is still zero-sized; the first-progress wait is capped at 15 s per attempt and 46 s across all attempts.
2. A live process that reaches the first-progress deadline without a frame is classified as a transient startup failure and retried at most twice. Each retry uses a new ffmpeg child and the same requested recording configuration.
3. A permanent diagnostic (Screen Recording/TCC denial, permission denied, no capture device, invalid `Capture screen N` selector, invalid display dimensions, or an unlaunchable ffmpeg) is surfaced immediately as a clear `RecordingError` with the relevant stderr tail and is never retried.
4. The two known AVFoundation warning lines may appear on both successful and failed attempts and do not by themselves change the outcome. If all three attempts fail transiently, `start` returns one bounded startup error that states the retry count and includes the final useful stderr tail.
5. Every failed attempt is stopped and reaped before the next attempt or return; its partial MP4 and progress/log files are removed. No ffmpeg child, zombie, or temporary output remains after success, permanent failure, exhausted retries, or `RecordingHandle` drop.
6. The capture selector and dimensions refer to the same intended display. For the existing main-display contract, derive the `CGMainDisplayID()` ordinal from the active CoreGraphics display list and use that ordinal in `Capture screen N:none`; do not silently fall back to `screen 0` when the main display cannot be mapped.
7. Successful recordings retain the existing Linux-parity contract: `libx264`, `ultrafast`, `yuv420p`, `+faststart`, cursor/click capture, `-t`/`-fs` limits, SIGINT finalization, `RecordingOutput` metadata, mock behavior, and unchanged Linux implementation.

*Design alternatives:*
- **Readiness:** retain file-size polling (rejected: MP4 size is a muxer side effect and caused the reported false timeout) vs. parse progress from a continuously drained stderr/progress stream (chosen: observes encoded frames without risking a filled stderr pipe; retain a bounded diagnostic tail/log).
- **Retry:** retry every non-zero ffmpeg exit (rejected: masks permission and selector errors) vs. classify stderr and retry only a live no-first-frame/session-start timeout (chosen: fail fast for permanent errors and make transient retries deterministic).
- **Display selection:** keep `Capture screen 0` (rejected: CoreGraphics main display is not guaranteed to be AVFoundation screen 0) vs. raw ffmpeg device indices (rejected: camera/device enumeration shifts) vs. derive the main display ordinal from the active CoreGraphics display list and use the matching dimensions (chosen; if mapping is unavailable, return an explicit environment error rather than capture the wrong screen).
- **Process sharing:** refactor Linux and macOS into a shared supervisor (deferred: broader Linux blast radius) vs. keep the supervisor local to `mac/recording.rs` with parity tests (chosen for this follow-up).

*Proposed changes:*
1. Refactor macOS startup into an attempt helper plus a bounded supervisor. Spawn ffmpeg with stderr/progress continuously drained (teeing to a temporary log or retaining a bounded tail), parse the first `frame=`/valid `time=` progress event as readiness, and continue retaining stderr until the child is stopped/reaped so classification and final errors are reliable. Keep all existing encode/input/output args unchanged except the derived screen selector and readiness plumbing.
2. Add an explicit startup classification enum/helper. Normalize stderr for matching; treat TCC/permission denial, no device, invalid selector, invalid dimensions, spawn failure, and unknown early exit as permanent. Treat a still-running child with no first progress at the per-attempt deadline as transient. Treat the two known AVFoundation warning strings as informational only; never classify either warning alone as failure.
3. On a transient result, send the same cleanup signal used by stop, await/reap the child (force-kill after the existing stop timeout), remove that attempt's MP4/log, sleep 250 ms or 500 ms, and relaunch. Stop after the third attempt and return a `RecordingError::Start` that names the bounded retry exhaustion and includes the final stderr tail. Permanent results clean up once and return without sleeping or relaunching.
4. Add a CoreGraphics helper that enumerates active display IDs, finds `CGMainDisplayID()`, returns both its ordinal and dimensions from that same display ID, and constructs `Capture screen {ordinal}:none`. If the main display is absent or the active list cannot be read, return a clear environment error; never substitute ordinal zero.
5. Add deterministic macOS-only tests using an injectable ffmpeg command/process or pure supervisor seam (do not depend on a real AVFoundation session): delayed first progress succeeds without file-size readiness; two transient attempts then success reaps the first children and leaves no temporary files; permanent stderr makes exactly one attempt; the benign warnings do not fail/retry; exhausted retries report the count/tail; selector/ordinal and dimensions come from the same display; and existing drop cleanup remains intact. Keep the existing Linux tests and implementation unchanged.

*Open questions resolved:*
- The timeout remains 15 s per attempt to preserve the existing operational budget while changing only the readiness signal; three attempts plus 250/500 ms backoff gives a hard 46 s startup cap.
- “Retry twice” means two retries after the initial launch (three total attempts), not three retries.
- A warning-only stderr stream is not a retry condition. A no-first-frame timeout is transient; a permanent marker or unknown early process exit is non-retryable.
- The follow-up changes only `warp`; ffmpeg packaging remains the coordinated `warp-agent-docker` PR already linked from this PR. Linux cannot reproduce the AVFoundation symptom and is covered only for regression.

*Risks / blast radius:* Progress parsing and child supervision can regress long recordings, stop/finalization, or cleanup if the reader is not continuously drained; mitigate with bounded log retention, explicit reap assertions, and the existing SIGINT/stop timeout. Display-list ordering may differ from an ffmpeg build; fail closed when the main display cannot be mapped rather than capturing another display. No Linux or encode-setting changes are permitted.
