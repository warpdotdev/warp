*Spec: Smart variable-speed/cut for computer-use video recordings (QUALITY-1112)*

== PRODUCT ==
*Summary:* On Linux, published computer-use recordings are captured at 1x, retain the moments around agent interactions at 1x, and remove blocked/thinking gaps with no on-screen delta. The transformation is client-side in the existing post-stop finalization path; no server-side segmentation and no player-side post-processing are introduced.

*Key design choices:* (1) use the existing action-dispatch timeline, with a deterministic 1.5 s action window and 500 ms margin on both ends, (2) cut with one post-stop ffmpeg trim/setpts/concat pass and remap ASS overlay timestamps through the same source-to-output map, and (3) keep the wire-level `playback_speed_multiplier` for compatibility but ignore it on the Linux smart-cut path (macOS retains its existing uniform-speed fallback).

*Behavior* (numbered, testable invariants from the user's/consumer's view):
1. A Linux recording with qualifying computer-use actions contains every action section at approximately real time (1x). The source interval for an action dispatched at offset `t` is `[t, min(t + 1.5 s, source_duration)]`; the retained interval expands by exactly 500 ms before and after, then clamps to `[0, source_duration]`. The 1.5 s window is the existing overlay `ACTION_GROUP_DURATION`, reused as the deterministic proxy because `ActionLogEntry` currently records dispatch offsets but not action end times.
2. A qualifying action is any `UseComputer` dispatch containing at least one real interaction (keyboard, typing, pointer, scroll, or other non-wait action). A dispatch containing only `Wait`/zero-duration no-op actions does not create a retained section. Pointer-only actions still create a timeline entry with an empty label list so their visual effects are kept even though no pill is rendered.
3. Offsets are sorted by source time before window construction. Duplicate, adjacent, or overlapping expanded windows are merged into one contiguous source segment. Actions at the same offset remain represented in that merged segment; no action is dropped because it has no overlay label.
4. The source gaps between merged segments are removed entirely. Each retained segment is reset to a zero-based PTS and concatenated in source order, producing an output timeline that starts at 0, advances monotonically, has no black frames or timestamp gaps, and preserves the source frame cadence and playback rate inside each segment.
5. Every rendered pill remains aligned with its source action after cuts. An overlay event uses its existing source interval (`[entry.offset, min(entry.offset + 1.5 s, next_entry.offset or source_duration)]`) and is remapped through the same source-to-output segment map before ASS generation. Events wholly in removed gaps are omitted; events that touch a cut boundary are clamped to the retained boundary. The remap is done at `Duration` precision before ASS's centisecond formatting.
6. No action timeline (no qualifying calls, including a click-free recording containing only waits) is a safe no-op: the original 1x capture is uploaded unchanged, with no empty/black output and no attempt to infer cuts. A single action produces one clamped segment. An action at time 0 clamps its leading margin to 0; an action at the recording end clamps its trailing margin to the source duration.
7. The Linux transformation is best-effort. If segment computation rejects a malformed timeline or ffmpeg cut/overlay processing fails, the original 1x capture is uploaded rather than a partial/corrupt artifact. Existing cleanup and terminal finalization behavior remains intact.
8. The Linux recorder no longer applies the server/default `playback_speed_multiplier` to the live capture or to the smart-cut output. The request field remains accepted for wire compatibility. On macOS, where `burn_in_action_log` is currently a no-op and capture is avfoundation, the existing uniform `setpts` behavior remains the documented fallback; no macOS cut implementation is part of this change. Other unsupported/no-op platforms remain unchanged.
9. `max_duration` remains an input wall-clock bound (`-t` before `-i`) and `max_size_bytes` remains the hard capture-file bound (`-fs`). Both apply to the new 1x Linux master, which can be larger than the old uniformly sped-up output and may therefore reach the size cap sooner in wall-clock time. No multiplier-based scaling of `-fs` or server policy change is introduced here; a size-limited capture still finalizes and cuts the prefix that exists.
10. Current capture metadata/API semantics are preserved: `RecordingOutput`/`StopRecordingResult` continue to report the recorder's wall-clock capture duration and capture-file size as they do today, while the uploaded video stream itself has the compact duration equal to the sum of retained segments. The no-audio capture contract remains video-only (`concat` uses `a=0`); adding audio is out of scope.

*Design alternatives*:
- *Live uniform speedup:* rejected because it speeds up the very action sections that must remain readable, cannot remove only blocked gaps, and would leave the existing overlay timestamps tied to the wrong playback clock.
- *Player/server post-processing:* rejected by the explicit client-side decision and because the artifact must be correct when downloaded or viewed without a specialized player; server code must not compute segments.
- *Physical per-segment files followed by an external concat:* rejected as extra temporary-file lifecycle and upload complexity. One post-stop `filter_complex` pass is selected.
- *`select`/`trim` without per-segment PTS reset:* rejected because concatenating source timestamps creates gaps or non-monotonic PTS. Each branch must use `trim=start=:end=,setpts=PTS-STARTPTS` before `concat=n=<N>:v=1:a=0`.
- *Action window ending at the next dispatch:* rejected because a long gap makes the previous action appear to own all intervening blocked time. The selected fixed 1.5 s window matches current pill duration and is deterministic; the 500 ms margin preserves visual context without inventing transitions.
- *250 ms or 1 s margins:* 250 ms is too tight at 15 fps and 1 s retains too much dead time. The selected 500 ms margin is exactly two 250 ms visual beats and is applied symmetrically, clamped, then merged.
- *Segment only labeled actions:* rejected because clicks, mouse movement, and drags can change the screen while intentionally producing no pill. The timeline records meaningful no-label interactions, while ASS generation skips empty label lists.
- *Scale `-fs` by the old multiplier:* rejected because it would weaken the configured hard disk bound and make resource use dependent on an optional server hint. The existing capture bound remains authoritative; server tuning can follow separately.
- *Deprecate `playback_speed_multiplier` immediately:* rejected because old servers and macOS still send/use it. Retain the field and document Linux smart-cut precedence; do not add a new server field in this PR.

*Proposed changes:*
1. Add a pure, unit-testable segment builder in the `computer_use` recording/overlay support. It accepts `&[ActionLogEntry]` plus source duration, sorts and validates offsets, constructs the fixed action windows, expands/clamps/merges them, and returns ordered `{source_start, source_end, output_start}` segments plus a source-to-output mapping function. Use saturating arithmetic and reject/ignore offsets at or beyond the source duration. Preserve nanosecond/microsecond precision internally.
2. Extend action collection so `UseComputer` calls with any non-no-op interaction append an `ActionLogEntry` even when `labels` is empty. Keep conversation ownership and active-state checks. Wait-only/no-op calls remain absent. Preserve the existing redaction-safe `overlay_labels_for` behavior.
3. Change the Linux capture command to capture at 1x by omitting the uniform `setpts` filter. Keep x11grab cursor behavior, input `-t`, codec settings, and output `-fs` unchanged. Do not change the macOS command.
4. Replace/extend `burn_in_action_log` with a post-stop Linux filtergraph that (a) emits one `trim` branch per retained interval, (b) applies `setpts=PTS-STARTPTS` to every branch, (c) concatenates branches with `concat=n=N:v=1:a=0`, and (d) applies the ASS subtitle filter after the cut. Map only the resulting video stream. A valid timeline with no labels still gets cut; an empty timeline returns the original path without re-encoding.
5. Generate ASS events from source offsets/windows and remap their start/end through the segment map before writing the ASS file. Keep sorting, group replacement/clamping, pill positions, escaping, and redaction unchanged. Drop an event whose remapped end is not greater than its remapped start.
6. Keep the existing finalize/upload fallback and cleanup: remove the source, cut/overlay sibling, and ASS temporary file after upload; on any cut or subtitle error upload the untouched source. Apply this to all existing finalization reasons, including size/duration limits and early ffmpeg exit.
7. Leave `RecordingOutput` and `StopRecordingResult` capture-duration/size metadata semantics unchanged, and document the difference between those fields and the compact uploaded stream duration in code comments/tests.

*Open questions resolved:*
- *Margin:* fixed at 500 ms on each side of every 1.5 s action window; no server-configurable margin is added.
- *Action duration:* fixed 1.5 s from each dispatch offset, matching `ACTION_GROUP_DURATION`; no guessed action-end timestamp is introduced.
- *Grouping:* sort by offset, expand/clamp, then merge intervals that overlap or touch; duplicate offsets are retained within the same merged segment.
- *Timeline source:* every meaningful interaction call, including pointer-only calls with empty labels; wait-only and zero-duration no-op calls do not create segments.
- *Playback multiplier:* retained for request compatibility and macOS fallback; ignored for Linux smart capture/cut. Linux no-action and failure fallbacks upload the unchanged 1x source, not a newly sped-up copy.
- *Platform:* Linux x11grab plus post-stop ASS/cut is the supported implementation. macOS remains uniform-speed avfoundation because the shared burn-in API is explicitly a no-op off Linux; Windows/no-op backends are unchanged.
- *Audio:* current capture has no audio input, so the graph is video-only and uses `concat ... a=0`; no audio synchronization policy is needed.
- *Metadata:* preserve current capture-time `duration` and capture-file `size_bytes` fields rather than changing the wire/API contract in this feature.
