*Spec: Change where TUI logs are stored (CODE-1902)*

== PRODUCT ==
*Summary:* Headless Warp TUI processes must write their logs to a frontend-specific location and use TUI-specific filenames. On macOS, the resulting files are `~/Library/Logs/warp-cli/warp_tui_dev.log`, `~/Library/Logs/warp-cli/warp_tui_preview.log`, or `~/Library/Logs/warp-cli/warp_tui.log`, matching the GUI's channel naming pattern without sharing the GUI directory. The existing `/view-logs` command must bundle and reveal the TUI logs from this resolved location.

*Key design choices:* Use an explicit `LogFrontend::Tui` configuration seam rather than inferring TUI from “file destination”/`is_cli`; resolve the TUI directory and filename centrally in `warp_logging`; retain `is_cli` only for the existing higher CLI/TUI rotation policy; make all path consumers (rotation, crash paths, bundles, prompt-chip sidecars, and `/view-logs`) read the resolved logging state so GUI behavior remains unchanged.

*Behavior*:
1. A TUI launch always writes file logs under the `warp-cli` log subdirectory. On macOS this is `~/Library/Logs/warp-cli/`; on Linux/FreeBSD and Windows, use the platform's existing base log directory and append `warp-cli` in the same way, so the frontend distinction is consistent across supported native platforms.
2. TUI filenames are deterministic by channel: `Channel::Dev` and `Channel::Local` use `warp_tui_dev.log`; `Channel::Preview` uses `warp_tui_preview.log`; `Channel::Stable`, `Channel::Oss`, and the non-shipped `Channel::Integration` fallback use `warp_tui.log`. The active file and all rotation suffixes use this same resolved base name.
3. GUI launches continue to use the current base directory, generated `ChannelState::logfile_name()` values, crash-recovery names, rotation limits, and bundle contents. No GUI path or filename changes are permitted.
4. CLI, remote proxy, and remote daemon launches continue to use the existing `oz` subdirectory and channel filenames, with their current rotation policy. The new TUI seam must not broaden or alter the CLI behavior.
5. `/view-logs` in the TUI bundles the active TUI log and its rotated/in-session companions from the resolved TUI directory/name, and its success hint contains the absolute zip path under that directory. It must not fall back to the GUI or legacy `oz` path. Reveal remains best-effort; path reporting remains unconditional.
6. Existing logs under `~/Library/Logs/oz/` are not migrated or deleted by this change. New TUI sessions use the new location; users can still access legacy files manually.

== TECH ==
*Context:* At `c353a2a4121521058bb73e8417c3a1b357c2e860`, `crates/warp_logging/src/native.rs:18-20` defines `CLI_LOG_SUBDIRECTORY = "oz"`, and `native.rs:575-579` appends it whenever `is_cli` is true. `app/src/lib.rs:623-632` currently derives `is_cli` from whether a launch mode has an explicit log destination; `app/src/lib.rs:965-984` gives `LaunchMode::Tui` a file destination, so TUI is incorrectly classified as CLI. The macOS base returned by `native.rs:648-655` is `~/Library/Logs/`, producing `~/Library/Logs/oz/`. `native.rs:431-474` derives bundle names from `ChannelState::logfile_name()`, and the path helpers/rotation code in `native.rs:292-342`, `native.rs:505-589`, and `native.rs:597-612` likewise read the shared channel filename. `crates/warp_logging/src/lib.rs:8-40` exposes `LogConfig`, `log_directory`, `log_file_path`, and `create_log_bundle_zip`. `crates/warp_tui/src/terminal_session_view.rs:2537-2565` invokes `create_log_bundle_zip()` and displays the returned absolute path; it has no intentional hard-coded directory, so changing the resolved logging state is the correct seam for `/view-logs`. `app/src/context_chips/logging.rs:126-141` derives its sidecar filename from `ChannelState::logfile_name()` and must consume the resolved logging filename for TUI correctness. TUI entry points are `crates/warp_tui/src/bin/dev.rs`, `local.rs`, `preview.rs`, `stable.rs`, and `oss.rs`; they set `ChannelState` before calling `warp_tui::run()`.

*Design alternatives*:
- *Keep `is_cli` as the directory switch and add a second `is_tui` boolean* — rejected because two booleans permit contradictory states and preserve the current overloaded meaning that caused the defect.
- *Infer frontend from `LaunchMode` inside `warp_logging`* — rejected because `warp_logging` should not depend on app launch-mode internals; an explicit frontend value keeps the crate boundary clear and is testable without booting the app.
- *Override `ChannelConfig.logfile_name` separately in each TUI binary* — rejected as the primary seam because it duplicates mapping across five entry points and leaves shared bundle/rotation/path helpers vulnerable to using the GUI name. The central logger resolves and stores one authoritative TUI name.
- *Use a single generic TUI filename for every channel* — rejected because it loses the GUI-compatible dev/preview/stable distinction and can mix logs from concurrently installed channels.
- *Migrate the old `oz` files into `warp-cli`* — rejected as out of scope and risky for concurrent GUI/CLI processes; leave legacy files untouched and make only new TUI sessions use the new location.

*Proposed changes*:
1. In `crates/warp_logging/src/lib.rs`, add a `LogFrontend` enum with `Gui` (default), `Tui`, and `Cli` variants, and add `frontend` to `LogConfig`. Keep `is_cli` as the rotation-policy input for compatibility; TUI callers set it true to retain the existing high rotation limit, while `frontend` exclusively selects directory/name behavior.
2. In `crates/warp_logging/src/native.rs`, add frontend-specific directory and filename helpers. `Gui` keeps the current base directory and `ChannelState::logfile_name()`; `Cli` keeps the `oz` subdirectory and the same channel name; `Tui` appends `warp-cli` and maps `ChannelState::channel()` to the names in Product behavior #2. Store the resolved filename in `LogState`, and refactor crash-recovery paths, startup/in-session rotation, `log_file_path`, `rotate_log_files`, `collect_log_paths_in`, and `create_log_bundle_zip` to use that stored value rather than re-reading `ChannelState::logfile_name()`. Expose a read-only resolved-name/path seam as needed by sidecar consumers.
3. In `app/src/lib.rs`, map launch modes to `LogFrontend`: `Tui` → `Tui`; GUI app/tests → `Gui`; command-line, remote proxy, and remote daemon → `Cli`. Continue passing `is_cli = true` for TUI/CLI modes so the current retention policy is preserved, but do not use `is_cli` to select the directory.
4. In `app/src/context_chips/logging.rs`, derive the prompt-chip sidecar path from the resolved logger filename/path so a TUI sidecar cannot be written under the GUI name or directory. GUI sidecar output must remain byte-for-byte/path-compatible.
5. In `crates/warp_tui/src/terminal_session_view.rs` and its tests, verify that the existing `/view-logs` execution consumes the resolved TUI bundle path, preserves unconditional absolute-path success output, and leaves best-effort reveal/error handling intact. If an API rename is needed while refactoring the logger, update this call site to the new resolved-state bundle API; do not introduce a hard-coded path in the TUI.
6. Add focused unit tests in `crates/warp_logging/src/native_tests.rs` for frontend directory/name resolution, all channel mappings, resolved `log_file_path`, and bundle collection using TUI names. Extend app/TUI tests to cover launch-mode frontend selection and `/view-logs` success output. Keep GUI and CLI assertions explicit so collateral behavior is detected.

*Open questions resolved*:
- *Which channels receive which TUI names?* Dev and Local are development channels and share `warp_tui_dev.log`; Preview uses `warp_tui_preview.log`; Stable, OSS, and the Integration fallback use `warp_tui.log`. This covers every `Channel` enum value without introducing an unrequested fourth shipped filename.
- *Should directory separation be macOS-only?* No. The user-visible requirement is macOS, but the seam is frontend-specific and must not create platform-dependent collisions; append `warp-cli` to each platform's existing base log directory while preserving each platform's current base-path logic.
- *Should TUI retain CLI rotation limits?* Yes. TUI currently enters the `is_cli` branch and gets `MAX_FILES_IN_CLI_ROTATION`; the fix changes only directory/name selection, not retention. The separate frontend field prevents that retention choice from controlling paths.
- *Does `/view-logs` need a new hard-coded path?* No. Its current command already delegates to `create_log_bundle_zip()` and displays the returned absolute path. Refactoring bundle/path consumers to use the resolved `LogState` makes the command point to the new location without duplicating platform logic; tests must prove this invariant.
- *What happens to existing `oz` logs?* Nothing. No migration, deletion, or GUI cleanup is part of CODE-1902.
- *Is a feature flag required?* No. This is a deterministic storage correction with no staged user behavior; the frontend seam and regression tests are sufficient.

*Risks / blast radius*:
- Refactoring shared path helpers could accidentally change GUI or CLI output. Mitigation: preserve the existing `Gui`/`Cli` branches verbatim where possible and add explicit path/name regression tests for both.
- A helper that still reads `ChannelState::logfile_name()` could make rotation or bundles silently miss TUI logs. Mitigation: make `LogState` the single source of truth and test active, `.old.N`, and `.in_session.N` files under each TUI name.
- Channel state is initialized before `warp::run_tui`; resolving the TUI name during logger initialization is deterministic. Test channel mappings without mutating global logger state by isolating the pure mapping helper.
- Concurrent GUI/TUI/CLI processes must not share active files. Mitigation: separate subdirectories and distinct names; no migration or cross-directory scanning.
- Prompt-chip sidecars and `/view-logs` are adjacent consumers. Mitigation: route both through the resolved logger path/name and include them in targeted tests.
