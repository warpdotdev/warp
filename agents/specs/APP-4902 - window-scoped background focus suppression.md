*Spec: Window-scoped background computer-use focus suppression (APP-4902)*

== PRODUCT ==
*Summary:* While a macOS background computer-use (CU) task is active, Warp must keep the target window controllable without raising it, but must not swallow the user's independent attempts to focus any Warp window. The current per-PID event-tap callback drops every raw focus event delivered to the previously frontmost process, so clicking Warp during an active CU task can leave Warp unable to come frontmost. The fix narrows suppression to the activation transition for the specific background target window.

*Key design choices:*
1. Keep the existing per-PID CGEvent taps, activation primer, and PR #13752 owner-scoped teardown; change only the callback's suppression predicate and the context data it needs. This preserves the working background-input mechanism and limits blast radius.
2. Identify the intended activation transition by the event's window-addressing identity, using the same private routing fields already stamped by `post_appkit_activation` / window-targeted mouse events. A focus event from another window, or one with no trustworthy window identity, passes through rather than being swallowed.
3. Make the rule session-local: each tap context carries its own activated window identity, so simultaneous CU sessions targeting different windows cannot suppress one another's focus transitions.

*Behavior* (numbered, testable invariants):
1. With an active CU task targeting a background window `T`, the user can click or otherwise focus every Warp window, including a Warp window different from the one that was frontmost when CU began; that Warp window becomes the normal frontmost/focused window and receives subsequent user input.
2. The active CU task can continue to deliver targeted mouse and keyboard events to `T` while `T` remains backgrounded; the activation primer still suppresses only the focus transition associated with activating `T`, and does not raise `T` or move the user's cursor.
3. Focus events for a different window, focus events without a valid/matching window identity, non-focus events, and all events observed by the target-process tap are passed through unchanged. Only raw focus event types `13`, `19`, or `20` observed by the previous-process tap, while suppression is enabled and the event addresses `T`, may be dropped.
4. Concurrent CU sessions targeting distinct windows remain isolated: session A drops only its own activation-window transition, session B drops only its own, and either session's teardown does not remove the other's activation state.
5. Existing session lifecycle behavior remains intact. On normal completion, cancellation, or teardown, `end_background_session` removes only the finishing owner's sessions, tears down taps before restoring focus, deactivates the activated window, and re-activates the window that was frontmost before that session. Repeated teardown remains harmless.
6. Screen-targeted / background-CU-disabled behavior remains the legacy path; no new window-scoped suppression is active when no background window activation session exists.

*Design alternatives:*
- *Remove the previous-process tap or replace it with app-level frontmost polling* — rejected. Polling cannot reliably intercept the activation transition before AppKit changes focus, and removing the tap breaks background activation for apps that require the primer. Keep the established tap ordering and narrow its predicate.
- *Continue suppressing every focus event from the previous PID* — rejected. This is the reported APP-4902 defect: it prevents a user click on any Warp window from reaching Warp.
- *Use one global “currently activating window” value* — rejected. A process-global value would race concurrent CU sessions and could cause one conversation to suppress another's transition. Carry the activation window in each `TapContext`/`ActiveSession` instead.
- *Fail closed when a focus event has no usable window identity* — rejected. Swallowing an event whose destination cannot be proven to be the CU activation would retain the focus-stuck failure mode. Fail open: pass through missing, invalid, or mismatched identity; the activation primer events already carry the required identity.
- *Match only the standard mouse window-under-pointer fields* — rejected as incomplete. Focus/AppKit-defined events may not populate those fields. Use the private routing pair already used by this module (`51` + validity `58`) as the primary activation identity, and preserve/use standard `91`/`92` fields when they are present on an event.

*Proposed changes:*
1. Extend the activation context carried to each tap callback with the session's activated window number (the `window_number` from the registry key / `ActiveSession`). The previous-process context must retain `is_previous`; the target-process context must retain the same window identity but never suppress.
2. Add a small, side-effect-free predicate in `activation.rs` for “focus event is the session's activation transition.” It must first recognize only raw types `13`, `19`, and `20`; require the previous-process context and an enabled suppression flag; read the event's window-addressing identity using the existing field-51/field-58 contract (and standard 91/92 values where available); and return true only when the event is validly addressed to the session's activated window. Missing, invalid, or different identities return false.
3. Keep `events_of_interest = u64::MAX`, tap placement, run-loop ownership, activation primer ordering, and the `ActiveSession::Drop` implementation unchanged unless a minimal context-construction adjustment is required. Do not change the registry key, owner matching, teardown order, or `post_appkit_activation` addressing writes.
4. Add focused macOS unit coverage in `activation_tests.rs` around the callback/predicate. Construct synthetic events with each relevant raw type and addressing state and assert the full matrix: matching activation window drops only on the previous tap while suppression is true; a different window passes; missing/invalid addressing passes; target tap passes; suppression disabled passes; non-focus events pass; and each of multiple contexts matches only its own window.
5. Preserve the existing `end_sessions_for_owner_clears_only_that_owners_entries` test and extend it only if needed to prove the new context/window state does not alter owner cleanup. Keep production call sites in `app/` unchanged unless compilation exposes a necessary signature adaptation.

*Open questions resolved:*
- The scope is the macOS client `warpdotdev/warp`; no `warp-server` or server/API change is required.
- The raw focus event set remains `13`, `19`, and `20`, matching the existing implementation and the reference activation mechanism; this change does not attempt to rename or reinterpret those undocumented event types.
- Window identity is the already-established activation address, not a new public API: field `51` carries the window number and field `58` marks the private routing field valid. Standard fields `91`/`92` may corroborate an identity when present but cannot be the sole discriminator for AppKit-defined focus events.
- When an event lacks a trustworthy matching identity, pass it through. This is the safe behavior for user input and avoids recreating the process-wide suppression bug.
- No UI copy, feature-flag, protocol, or persistence change is needed. The observable change is native macOS focus/input routing.
- The current child environment is Linux and cannot execute macOS CoreGraphics/AppKit taps. Triage supplied the confirmed macOS repro and root cause; Linux verification is an environment-mismatch skip, not evidence that the fix works. The implementation must perform the macOS regression and UI checks on a macOS runner/host.

*Risks / blast radius:* The code touches low-level, undocumented macOS event routing; an incorrect field interpretation could either reintroduce the focus-stuck bug or let the activation primer raise the target. Mitigate with the deterministic callback matrix, repeated real-macOS background-CU runs, explicit concurrent-session and teardown checks, and the full presubmit gate. Keep the change confined to `crates/computer_use/src/mac/activation.rs` and its tests unless compilation requires a narrow signature update.
