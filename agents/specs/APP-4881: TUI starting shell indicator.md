*Spec: TUI shows the input editor with a `Starting shell...` indicator during session bootstrap (mirror the GUI)*

== PRODUCT ==
*Summary:* While a TUI session is bootstrapping (the shell has not yet reached its first post-bootstrap precmd), the Warp TUI renders the prompt input editor with a single status line `Starting shell...` immediately above it. Prompt submission is disabled during this state; the user can prepare a draft that survives into the normal prompt once bootstrap completes. This mirrors the GUI desktop client, which already shows `Starting shell...` above an editable input while gating only execution.
*Key design choices:*
1. Render the input editor + the `Starting shell...` line during the existing `TuiInputTarget::Disabled` (bootstrap) state rather than introducing a new enum variant — the submission guard and state machine already exist, so this is the minimal, lowest-risk change.
2. Keep prompt submission gated by the *existing* `handle_submitted` guard (`input_target().agent_editor_owns_input()`), which already returns early for `Disabled`; no new submission-blocking code is needed.
3. Do not forward ordinary keystrokes to the PTY during bootstrap (unchanged: `with_pty_input` is only attached when `pty_owns_input()`), so showing the editor does not risk leaking draft text into the bootstrapping shell.

*Behavior* (numbered, testable invariants from the user's view):
1. While the active session is not yet bootstrapped (the TUI's `Disabled` input-target state), the session frame visibly renders the prompt input editor.
2. While in that bootstrap state, a single line containing exactly `Starting shell...` is rendered immediately above the input editor.
3. While in that bootstrap state, pressing `Enter` (the submit key) does not submit the prompt, execute a shell command, or send an agent query; the draft text remains in the editor unchanged.
4. The user can type and edit a draft in the input editor during bootstrap, and that draft persists without loss when bootstrap completes (the same input model/view is rendered before and after; no clear on the `BootstrapPrecmdDone` transition).
5. When bootstrap completes (`BootstrapPrecmdDone`), the `Starting shell...` line disappears, normal prompt rendering resumes (input box border, footer, inline menus, warping indicator all return to their `AgentEditor` behavior), and `Enter` submits as usual.
6. Non-bootstrap TUI states are unchanged: alt-screen apps, shell startup *script execution* (`Pty`), user-controlled long-running commands (`Pty`), blockers, orchestration tabs, and conversation restore all render and route input exactly as before.
7. The GUI desktop client is unchanged (this change is TUI-only).
8. Reserved bindings (`ctrl-c`, `ctrl-d`, plan toggle, tab navigation) behave consistently with the normal prompt mode while the input is focused during bootstrap, because the input view's keymap context is active. Specifically `ctrl-c` clears the draft and arms the exit-confirmation window (same as normal prompt mode with no active conversation), and a second `ctrl-c` within the window exits the TUI.

*Design alternatives* (per decision point with more than one reasonable approach):
- *Input-target modeling* — (a) **Repurpose `Disabled`** (chosen): change `render` to emit the `Starting shell...` line + the input box + footer during `Disabled`, and change `focus_current_owner` to focus the input view during `Disabled`; update the `Disabled` doc to "submission disabled and PTY input not forwarded, but the editor is rendered with a `Starting shell...` indicator so the user can prepare a draft." Pros: minimal churn (enum, state fn, and the existing `shell_startup_routes_input_by_bootstrap_stage` test all stay valid); reuses the existing `handle_submitted` submission guard; the `BootstrapPrecmdDone` re-render already transitions out. Cons: the name `Disabled` becomes slightly softer ("disabled submission" rather than "hidden editor"), mitigated by the doc update. (b) *Add a new `TuiInputTarget::Bootstrapping` variant*: more self-documenting; `agent_editor_owns_input()` and `pty_owns_input()` both return `false` for it (so submission + PTY gating are automatic), and a new `renders_editor()`/`bootstrapping()` predicate selects the new render+focus branch. Pros: clean separation, no overload of `Disabled`. Cons: more churn — the enum, `tui_input_target_for_state`, `focus_current_owner`, `render`, `keymap_context`, and the `shell_startup_routes_input_by_bootstrap_stage` test (which asserts `Disabled` after `reinit_shell`) all change. Rejected for this ticket as higher churn for the same behavior; the implementor may switch to (b) if the repurposed `Disabled` reading becomes confusing during review.
- *Source of the `Starting shell...` text* — (a) **TUI-local constant** (chosen): a `const STARTING_SHELL_HINT: &str = "Starting shell...";` in `terminal_session_view.rs`, rendered with a muted/accent `TuiText` style from `TuiUiBuilder`. Pros: matches the request exactly, zero new cross-crate coupling, no need to export the GUI's `bootstrapping_shell_message` ladder. Cons: diverges from the GUI's `bootstrapping_shell_message` which also produces remote-server-setup stage strings (`"Installing Warp SSH Extension... (N%)"`, `"Initializing..."`, etc.). (b) *Export and reuse the GUI's `bootstrapping_shell_message` via `tui_export`*: keeps wording identical to the GUI across all stages. Rejected for this ticket: the TUI does not currently import `RemoteServerSetupState` and has no SSH remote-server-setup UI surface today, so mirroring those stages is out of scope and would pull a new shared dependency for strings the TUI cannot yet produce. The remote-server stage ladder is recorded as a follow-up (see *Risks / blast radius*); the common `"Starting shell..."` string is identical to the GUI's common path, so user-visible parity holds for the bootstrap case this ticket targets.
- *Focus handling during bootstrap* — (a) **Focus the input view during `Disabled`** (chosen): the user can type a draft; `Enter` is intercepted by the editor and routed to `handle_submitted`, where the existing `agent_editor_owns_input()` guard drops it. Pros: matches the request ("show the input"; "the draft remains available for editing after bootstrap") and the GUI (editable input, gated execution); the draft persists because the same input model is rendered before and after. Cons: changes which view holds focus during bootstrap (was `focus_self`, now the input view), so reserved bindings now resolve against the input view's keymap context. (b) *Keep `focus_self` and render a non-focusable editor*: the editor would be visible but the user could not type. Rejected: does not satisfy "the draft remains available for editing." The keymap implications of (a) are handled by noting that the input view's keymap context already hosts the reserved `ctrl-c`/`ctrl-d`/plan-toggle bindings (see `crates/warp_tui/src/terminal_session_view.rs:330-377 @ 37c26a8`), so behavior stays consistent with normal prompt mode; this is verified by the visual proof criterion.

*Open questions resolved:*
- *Should the TUI mirror the GUI's remote-server-setup stage strings (`Installing Warp SSH Extension... (N%)`, `Initializing...`, etc.)?* Resolved: no, out of scope. The TUI does not import `RemoteServerSetupState` and has no SSH remote-server-setup UI surface today. This ticket ships the common `"Starting shell..."` string, which is identical to the GUI's common-path wording. Mirroring the remote-server ladder is a follow-up tracked under *Risks / blast radius*, to be done when/if the TUI gains a remote-server-setup surface.
- *Should the user be able to type a draft during bootstrap, or is the editor read-only?* Resolved: editable (focus the input view during `Disabled`), per the request ("show the input", "the draft remains available for editing after bootstrap") and GUI parity (editable input, gated execution).
- *Should the footer render during bootstrap?* Resolved: yes — render the normal footer (model/cwd/usage row) via the same `render_footer` path used for `AgentEditor`, so the bootstrap frame is visually consistent with the normal prompt frame and the `ctrl-c` exit-confirmation hint surface stays available. The cwd falls back to the process cwd until shell metadata arrives (existing `current_working_directory` fallback), which is acceptable.
- *Does showing the editor risk forwarding draft text to the bootstrapping shell's PTY?* Resolved: no. `with_pty_input` is attached only when `pty_owns_input()` (line 2876); `Disabled` does not own PTY input, so keystrokes go to the focused editor, not the PTY. This is verified by the input-policy test and the visual proof.

*Risks / blast radius:*
- *Shell-startup PTY routing*: the `Pty` state (alt-screen, script execution, long-running commands) is untouched — only `Disabled` changes. `with_pty_input` still attaches only for `Pty`. The render-to-lines test + the existing `shell_startup_routes_input_by_bootstrap_stage` test guard this.
- *Focus shift during bootstrap*: moving focus from `focus_self` to the input view during `Disabled` changes which keymap context is active. Reserved bindings (`ctrl-c`, `ctrl-d`, plan toggle, tab nav) are registered for the `TuiInputView`/`TuiTerminalSessionView` contexts (see `init` at line 329-377) and behave as in normal prompt mode. `ctrl-c` clears the draft + arms exit (consistent with normal mode); `ctrl-d` exits on an empty draft / deletes otherwise (consistent with normal mode). The visual proof criterion must confirm these during a real bootstrap; if a reviewer finds a ctrl-d-during-bootstrap exit surprising, a follow-up can gate `Eof`'s exit on `is_bootstrapping_precmd_done()` — out of scope for this ticket unless the visual proof surfaces a problem.
- *Blocker precedence during bootstrap*: blockers (e.g. a `RunAgents` card) already suppress the input box (line 2777-2784); the new bootstrap branch must respect `!blocker_active` so a blocker still takes precedence. Covered by mirroring the existing `!blocker_active` guard.
- *Conversation restore*: `ConversationRestoreState::Loading { origin: Startup, .. }` returns early (line 2714-2718) with a restoring screen, so the bootstrap branch is not reached during startup restore. `ConversationList`-origin loading proceeds and the bootstrap branch renders underneath; this is the same overlaying as today and is not a regression.
- *Remote-server-setup wording divergence*: documented above as a follow-up, not a regression.
- *Zero-state coexistence*: during a fresh bootstrap the transcript is empty, so the zero state fills the transcript slot above the `Starting shell...` line. The visual proof confirms this stacks correctly.
