*Spec: Audit and demote non-actionable `report_error!` calls in Warp AI-agent code*

== PRODUCT ==
*Summary:* Reduce Sentry noise in the Warp AI-agent client by retaining `report_error!` only for failures that indicate an engineering defect, a critical-subsystem impact, or an external behavior that may be our misuse. Handled network failures, forward-compatible data, stale resources, fallback timestamps, and best-effort cancellation should remain visible as local diagnostics without creating Sentry issues.
*Key design choices:* Perform an explicit call-by-call audit rather than a blanket replacement; preserve control flow and user-visible behavior; use `warn` for handled/degraded operations, `debug` for high-frequency per-message fallback diagnostics, and `safe_*` variants whenever error detail could contain sensitive response or user data.
*Behavior:*
1. All 26 `report_error!` calls in the six requested files have an explicit keep/demote decision grounded in `.agents/skills/logging-and-error-reporting/SKILL.md`.
2. Calls retained as `report_error!` continue to capture actionable malformed state, invariant violations, serialization/conversion defects, and thresholded actionable event-stream failures.
3. Demoted calls no longer create structured Sentry issues, but still emit an appropriate local breadcrumb/diagnostic and preserve the existing return values, retries, state updates, toasts, and fallback behavior.
4. No secrets, prompts, response bodies, or other user-generated content are introduced into release-channel logs; use `safe_warn!`/`safe_error!` when useful detail is sensitive.
5. Every demoted diff hunk has exactly one inline PR review comment explaining the applicable logging principle and why that hunk is non-actionable. No source-code comments are added for this audit.

== TECH ==
*Context:* The audit target is `warpdotdev/warp` at commit `e6aaaf9b8f84f786035cd797224a349abb56a4f5` (`master`). The logging policy is `.agents/skills/logging-and-error-reporting/SKILL.md` at that commit. The six target modules are `app/src/ai/agent/conversation.rs`, `app/src/ai/agent/task.rs`, `app/src/ai/agent/linearization.rs`, `app/src/ai/agent_conversations_model.rs`, `app/src/ai/agent_events/driver.rs`, and `app/src/ai/ambient_agents/task.rs`.
*Design alternatives:*
- Blanket-demote every `report_error!` in the listed files — rejected because task-tree cycles, missing parents, transaction mismatches, and missing pending-request mappings are malformed internal state and remain actionable.
- Keep every call and rely on `report_error!`'s typed `is_actionable()` classification — rejected because several sites pass ordinary strings or intentionally swallow handled external/fallback conditions, so they still create noisy Sentry issues or obscure the fact that the operation is expected to degrade.
- Replace all demotions with `log::error!` — rejected because retries, stale messages, compatibility fallbacks, and best-effort cancellation are expected/recoverable (`log::warn!`), while the timestamp fallback runs in a per-message path and should use `log::debug!` to avoid breadcrumb volume.
- Add a new typed error taxonomy solely for this audit — rejected as scope creep; the existing control flow already distinguishes the external/retryable paths from invariant paths, and the change is limited to the reporting sink.
*Proposed changes:* Audit the surrounding control flow at each site, replace only the demotions below, keep all other calls intact, and preserve static message prefixes plus non-sensitive structured context. For request/cancellation errors whose display chain may include response payloads, URLs, paths, or user content, use `safe_warn!` with a standalone safe arm and place detail only in the full arm. Do not alter retry strategy, task/conversation state transitions, error returns, or toast text.
