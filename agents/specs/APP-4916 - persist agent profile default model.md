*Spec: persist agent profile default model across restart/reconnect (file-backed migration + active-profile restore)*

Linear: APP-4916 (estimate L/5). Target repo: `warpdotdev/warp` (the client — see *Target repo* note; the affected code lives only in the client, not `warp-server`). All file references are commit-pinned at `f74bfd9` (`origin/master` HEAD when this spec was written).

== PRODUCT ==
*Summary:* A manually-selected default base model for an agent execution profile reverts to `auto (genius)` after app restart. Client logs (warp_dev.log, dev build v0.2026.07.21.09.02.dev_00) confirm the trigger is the **file-backed-profiles migration + active-profile restore failure on restart**: the session that lost the model initialized execution profiles as "Unsynced, file_backed_profiles: true" (a migration to the file-backed format), immediately logged 5x "Failed to restore active profile" for the Default profile's server sync id (`zhtjJcN6WZYl6r9nuFS7D1`), and then fell back to the Default profile whose `base_model` surfaces as `auto (genius)`. A definitive negative: `grep` across all 6 log files for `AdminDisabled|RequiresUpgrade|Unavailable|OutOfRequests|ProviderOutage|reconcile_disabled` returned **0 matches** — so the disabled-model reconcile path did NOT trigger this user. The selected profile (and its `base_model`) must persist and be correctly re-mapped across the file-backed migration and app restart.

*Key design choices:*
1. **Primary fix — restore the active profile (and thus its `base_model`) across the file-backed migration.** On restart, `PaneGroup` restores each terminal pane's active profile from the snapshot's `active_profile_id` (a legacy cloud `SyncId`), but `AIExecutionProfilesModel::get_profile_id_by_sync_id` returns `None` unconditionally in file-backed mode, so the restore fails and `active_profile` falls back to the Default profile (`auto (genius)`). Because file-backed profile ids are **deterministically derived** from the legacy server id (`ExecutionProfileId::from_legacy_server_id`), the restore path can re-map the snapshot's `ServerId` to the migrated file-backed id and verify it exists in the current collection before selecting it.
2. **Secondary hardening — stop the disabled-model reconcile from durably erasing a stored `base_model` on a transient unusable classification.** Not this user's trigger (0 matches in logs), but a real latent path that can cause the same symptom for other users: `reconcile_disabled_model_preferences` durably clears the stored preference when a recognized selected model is classified `Unavailable`/`AdminDisabled`/`RequiresUpgrade`-no-BYOK during a metadata refresh. Keep this as secondary hardening with the silent-clear caveat noted.
3. **Cloud/settings sync ordering** so a stale cloud execution-profile collection cannot overwrite a newer local selection during initial load/migration (the existing `force_local_wins_on_startup` hash guard; add a test).

*Behavior* (numbered, testable invariants from the user's view):
1. Default/happy path: selecting a concrete base model in Settings → Agents → Profiles persists it; the effective model used for Agent Mode requests is that model when it is usable.
2. **Persistence across restart + file-backed migration (PRIMARY):** after a restart that migrates legacy cloud profiles to the file-backed collection, a terminal pane whose snapshot recorded an active non-default profile (by its legacy `ServerId`) restores that profile — not the Default profile — so its `base_model` is preserved (not `auto (genius)`). The Default profile's own `base_model` is likewise preserved across the migration/restart.
3. **Active-profile restore must not fall back to Default when a stored selection exists:** if the snapshot's `active_profile_id` maps (via the deterministic legacy-server-id derivation) to a profile present in the current file-backed collection, that profile is restored. If it genuinely does not exist (deleted), restore falls back to Default and logs the reason — but never silently falls back when the profile is present under its migrated id.
4. The already-preserved transient disable states (`OutOfRequests`, `ProviderOutage`, `RequiresUpgrade`-with-BYOK) remain preserved (no regression). (Secondary hardening.)
5. A transient/reversible disable classification (`Unavailable`, `AdminDisabled`, `RequiresUpgrade`-without-BYOK) does **not** durably erase the stored `base_model`; the effective request model falls back non-destructively to a usable choice and reverts when the model becomes usable again. (Secondary hardening; not this user's trigger but same symptom class.)
6. Explicit user action is the only normal way the stored preference changes: selecting a different model sets it; selecting the default/auto normalizes it as today.
7. Cloud sync ordering: on initial load, a stale cloud execution-profile collection does not overwrite a newer local selection (the existing `force_local_wins_on_startup` hash guard).
8. UI: the profile editor's base-model dropdown reflects the *persisted* selection (not `auto (genius)`) after restart when the profile was active; when the selected model is currently unusable the UI indicates a fallback is in use without erasing the selection.

*Design alternatives* (per decision point with more than one reasonable approach):
- **Decision 1 — active-profile restore across the file-backed migration (PRIMARY).**
  - *Option A (selected):* Make `get_profile_id_by_sync_id` file-backed-aware: when `source.uses_settings_backend()` and the incoming `sync_id` is `SyncId::ServerId(s)`, derive `ExecutionProfileId::from_legacy_server_id(s)` and return it **only if** that id is present in the current `AISettings::execution_profiles` collection (so a genuinely-deleted profile still falls back to Default). This reuses the migration's deterministic id derivation and the existing collection lookup; no new state; the snapshot's `ServerId` is re-mapped exactly as the migration mapped it. *Pros:* minimal, deterministic, cross-device-stable, fixes the confirmed trigger, preserves the `base_model` that lives on the restored profile. *Cons:* couples the restore path to the legacy-id derivation (already a migration invariant); only applies to `ServerId` sync ids (the only kind a pre-migration synced profile produces).
  - *Option B: store the file-backed `ExecutionProfileId` (not the `SyncId`) in the snapshot going forward, and migrate old snapshots.* *Rejected as the primary fix:* doesn't help existing snapshots that already carry a `ServerId` (this user's case); would require a snapshot-format migration and still need the ServerId→file-id re-map for old snapshots. Can be a *future* hardening (persist the stable file-backed id in new snapshots) but is not necessary to fix the bug.
  - *Option C: eagerly re-run the legacy migration before restore so `profile_id_to_sync_id` is populated.* *Rejected:* file-backed mode intentionally does not maintain `profile_id_to_sync_id` (it's empty — `profiles.rs:232`); the collection is the source of truth, not a sync-id map. Option A uses the collection directly.
- **Decision 2 — restore-failure observability.** Today a failed restore logs only `"Failed to restore active profile"` with no reason. *Selected:* distinguish "sync id does not map to any collection profile (deleted)" from "sync id is not a ServerId / unexpected" in the log, so the new behavior is observable and support can trace it. *Option B (no extra logging):* rejected — the bug was found via logs, so logging is how we confirm the fix.
- **Decision 3 — disabled-model reconcile (SECONDARY hardening).** Same as the prior spec draft: stop durably clearing `base_model`/`coding_model`/`cli_agent_model`/`computer_use_model` for the recognized-but-transiently-unusable case; fall back non-destructively in `get_preferred_base_model`; clear only on explicit user action or confirmed permanent deletion. Kept as secondary because the logs prove it is not this user's trigger, but it is a real latent same-symptom path. Document the **silent-clear caveat**: today's durable clear is silent (no log at the clear site); the fix adds a `log::debug!` when a recognized selected model is unusable but the preference is being kept and a fallback used.
- **Decision 4 — cloud-sync ordering.** The existing `force_local_wins_on_startup` hash guard already forces local to win when the settings file has unsynced changes. *Selected:* Decision 1 removes the bad fallback at the source so the restored profile (not Default) is used; additionally add a test (criterion #10) that a stale cloud execution-profile collection does not overwrite a newer local selection when `force_local_wins_on_startup` is true. *Option B (per-field timestamps):* rejected as heavy and unnecessary.

*Risks / blast radius:*
- **Primary fix scope:** `get_profile_id_by_sync_id` is also called from any other restore path; making it file-backed-aware could change behavior for callers that expected `None`. Audit other callers (grep `get_profile_id_by_sync_id`) — the pane-group restore is the primary caller; confirm no caller relies on the `None` return in file-backed mode.
- **Migration timing:** `migrate_settings_profiles` runs on `CloudPreferencesSyncerEvent::InitialLoadCompleted` / `CloudModelEvent::ObjectSynced` (`profiles.rs:320-347`); pane restore runs during workspace snapshot restore. If restore runs before migration completes, the collection may not yet contain the migrated profile → the collection-lookup guard returns `None` → falls back to Default. Mitigation: confirm ordering (restore happens after the model is constructed; migration is eagerly run if the syncer already completed — `profiles.rs:402-410`); if a race exists, the test (criterion #2) must cover restore-before-migration and the fix must handle it (e.g. defer restore until migration completes, or re-attempt restore on `ExecutionProfiles` change).
- **Secondary behavior change:** `AdminDisabled`/`Unavailable`/`RequiresUpgrade`-no-BYOK no longer auto-clear (secondary hardening). Visible change for users who relied on the auto-clear; call it out in the PR description and changelog (`CHANGELOG-BUG-FIX`).
- A truly-permanently-deleted-but-still-recognized model leaves a stale `base_model` (secondary); mitigated: user can reselect; unrecognized ids already preserved.
