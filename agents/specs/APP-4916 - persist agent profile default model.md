*Spec: persist agent profile default model across restart/reconnect*

Linear: APP-4916 (estimate L/5). Target repo: `warpdotdev/warp` (the client — see *Target repo* note; the affected code lives only in the client, not `warp-server`). All file references are commit-pinned at `f74bfd9` (`origin/master` HEAD when this spec was written).

== PRODUCT ==
*Summary:* A manually-selected default base model for an agent execution profile must persist across app restart, reconnect, model-metadata refresh, and BYOK/admin/team changes. Today it intermittently reverts to `auto (genius)` because a metadata refresh durably erases the stored preference whenever the selected model is temporarily classified unusable, after which effective-model resolution falls back to the server default.

*Key design choices:*
1. **Separate durable preference from effective runtime fallback.** The stored `base_model` is the user's intent and is preserved. The model actually used at request time falls back to a usable choice when the selected model is currently unusable — *without mutating storage* — and reverts to the selected model when it becomes usable again.
2. **Clear the stored preference only by an explicit user action** (selecting a different model / resetting to default) **or a confirmed permanent/deleted-model condition** (the model id is absent from the server catalog AND local custom endpoints — the existing "unrecognized" preserve case). Transient or reversible disable classifications — `Unavailable`, `AdminDisabled`, `RequiresUpgrade` (no BYOK key), plus the already-preserved `OutOfRequests`, `ProviderOutage`, `RequiresUpgrade` (with BYOK) — do **not** durably clear.
3. **Cloud/settings sync must not let a stale execution-profile collection overwrite a newer local selection** during initial load/migration. The primary cross-device fix is choice #1 (no bad clear is emitted, so no bad `None` syncs), plus an ordering guard test.

*Behavior* (numbered, testable invariants from the user's view):
1. Default/happy path: selecting a concrete base model in Settings → Agents → Profiles persists it; the effective model used for Agent Mode requests is that model when it is usable.
2. Persistence across refresh: a model-metadata refresh (startup, reconnect, auth-complete, network-online, BYOK-key change) that classifies a *recognized* selected model as `Unavailable` / `AdminDisabled` / `RequiresUpgrade`-without-BYOK does **not** change the stored `base_model`. The profile still shows the selected model (disabled/greyed if unusable); the effective request model falls back to a usable choice; when the model becomes usable again the effective model reverts to the selection.
3. The already-preserved transient states (`OutOfRequests`, `ProviderOutage`, `RequiresUpgrade`-with-BYOK) remain preserved (no regression).
4. Explicit user action is the only normal way the stored preference changes: selecting a different model sets it; selecting the default/auto normalizes it as today.
5. Permanent removal: if the selected id is absent from the server catalog AND local custom endpoints (unrecognized on this device), the preference is preserved (cross-device safety, existing behavior) — not cleared by the reconcile path. (Actively clearing a confirmed-permanently-deleted-but-still-recognized model is out of scope; the user can reselect.)
6. Cloud sync ordering: on initial load, a stale cloud execution-profile collection does not overwrite a newer local selection. Cloud only wins when local has no newer unsynced change (the existing `force_local_wins_on_startup` hash guard).
7. Cross-device: because a durable clear is no longer emitted from a transient classification, the bad state no longer syncs to other devices.
8. UI: the profile editor's base-model dropdown reflects the *persisted* selection (not `auto (genius)`) after restart/refresh when the selected model is recognized; when the selected model is currently unusable the UI indicates a fallback is in use without erasing the selection.

*Design alternatives* (per decision point with more than one reasonable approach):
- **Decision 1 — durable clear on disable classification.**
  - *Option A (selected):* Stop durably clearing `base_model` / `coding_model` / `cli_agent_model` / `computer_use_model` in `reconcile_disabled_model_preferences` for **any** disable classification. Keep the stored preference; provide effective runtime fallback (Decision 2). Clear only via explicit user action or a confirmed permanent deletion. *Pros:* matches every acceptance criterion; reversible; survives restart/reconnect/BYOK-add/admin-toggle; stops cross-device propagation at the source. *Cons:* the profile UI must represent "selected but currently unusable, using fallback"; a truly-permanently-deleted-but-still-recognized model leaves a stale preference (mitigated: unrecognized ids are already preserved; the user can reselect; a future confirmed-deletion signal can be added).
  - *Option B: keep the durable-clear mechanism but narrow `should_clear_preference` to only truly-permanent conditions.* *Rejected:* cannot reliably distinguish permanent from transient for `Unavailable`/`RequiresUpgrade` (they can flip between refreshes); `AdminDisabled` is admin-controlled and reversible. Too fragile — would re-introduce the same class of bug.
  - *Option C: keep clearing but only guard cloud-sync ordering.* *Rejected:* does not fix the core durable-erase on restart/reconnect — the local selection is still destroyed on every refresh that classifies the model unusable.
- **Decision 2 — effective runtime fallback without storage mutation.**
  - *Option A (selected):* In `get_preferred_base_model` (and the `coding` / `cli_agent` / `computer_use` analogues), when the stored id is `Some` but currently not usable (`is_usable_llm` false), return the existing disable-aware fallback (`fallback_llm_info` → `usable_default_llm_info`) for the *effective* request model while keeping `base_model` set. Reuses existing fallback machinery; no new state.
  - *Option B: add a separate "effective model" resolver layer.* *Rejected:* duplicates `fallback_llm_info`/`usable_default_llm_info` and adds surface area for the lookup semantics to drift.
- **Decision 3 — `context_window_limit` reconcile.** Today `reconcile_disabled_model_preferences` also clears `context_window_limit` when the model is unusable *or* non-configurable (`llms.rs:1811-1816`). *Selected:* preserve `context_window_limit` for transient/unverified unusability (so the user's limit is not lost alongside the model); keep clearing it only for the confirmed **non-configurable** case (a stable property of the model, not a transient state), gated on recognized. Implementation to confirm the non-configurable branch is independent of the unusable branch.
- **Decision 4 — cloud-sync ordering.** The existing `force_local_wins_on_startup` hash guard (`cloud_preferences_syncer.rs:97,594`) already forces local to win when the settings file has unsynced changes; when the local hash matches the stored hash, cloud wins (`cloud_preferences_syncer.rs:644-649`) and could overwrite a newer local selection with a stale cloud collection. *Selected:* Decision 1 removes the bad clear at the source so cloud never receives a stale `None`; additionally add a test (criterion #9) that a stale cloud execution-profile collection does not overwrite a newer local selection when `force_local_wins_on_startup` is true. *Option B (per-field last-modified timestamps):* rejected as heavy and unnecessary once the source clear is fixed.

*Proposed changes:*
1. `reconcile_disabled_model_preferences` (`app/src/ai/llms.rs:1767`): remove the durable `set_base_model(None)` / `set_coding_model(None)` / `set_cli_agent_model(None)` / `set_computer_use_model(None)` calls for the recognized-but-currently-unusable case. Keep the recognized-vs-unrecognized guard (preserve unrecognized ids for cross-device safety). Keep the `context_window_limit` clear only for the confirmed non-configurable case (Decision 3). Net effect: the reconcile becomes **non-destructive** for hosted model preferences.
2. `get_preferred_base_model` (`app/src/ai/llms.rs:836`) and the coding (`llms.rs:902`) / cli (`llms.rs:990`) / computer-use analogues: when the stored id is `Some` but currently not usable, return the disable-aware fallback (`fallback_llm_info`) for the effective request model *without* mutating the stored preference. When the model becomes usable again, the effective model reverts to the selection.
3. *(Audit, likely no change)* `reconcile_stale_custom_router_selection` (`llms.rs:1271`) and `sanitize_disabled_custom_model_preferences` (`llms.rs:1330`): these clear `base_model` only for deleted local custom-router ids / removed custom-endpoint ids — legitimate permanent-removal cases. Confirm they still align with the "explicit action or confirmed permanent deletion" rule. If `sanitize_disabled_custom_model_preferences` can fire *transiently* (e.g. a `TeamsChanged` where custom inference is temporarily disabled), gate it on a confirmed permanent removal too; otherwise leave as-is.
4. Cloud-sync ordering (`app/src/settings/cloud_preferences_syncer.rs:582`): add/keep a test that a stale cloud execution-profile collection does not overwrite a newer local selection when `force_local_wins_on_startup` is true (criterion #9). The primary cross-device fix is change #1.
5. Telemetry/diagnostic logging: add a `log::debug!` at the (now-preserved) decision point when a recognized selected model is currently unusable but the preference is being kept and a fallback used, so the new "preserve + fallback" behavior is observable. No sensitive data.

*Open questions resolved:*
- "Is `Unavailable` always permanent?" No — a model can be transiently `Unavailable` (server-side hiccup, refresh race, recategorization). Treated as non-permanent; preference preserved.
- "Should `AdminDisabled` clear the preference?" No — an admin can re-enable; the preference is preserved and runtime falls back. When the admin re-enables, the user's selection returns. (Behavior change from today; matches the requested "reversible" requirement and is called out in risks.)
- "Should `RequiresUpgrade`-without-BYOK clear?" No — the user may add a BYOK key later; preserving lets the selection activate on BYOK add. (Behavior change; matches "reversible".)
- "What about a truly permanently deleted model?" The existing unrecognized-id guard already preserves it (cross-device safety). Actively clearing a confirmed-permanently-deleted-but-recognized model is out of scope; the user can reselect. A future confirmed-deletion signal (e.g. model absent across N refreshes / a server-deleted flag) can be added later.
- "Does the profile UI need changes?" The dropdown already filters `AdminDisabled` out of choices (`get_base_llm_choices_for_agent_mode`, `llms.rs:946`) and shows disabled models with tooltips. The change to verify is that the UI shows the *persisted* selection (not `auto (genius)`) after restart/refresh when the selected model is recognized-but-unusable, and indicates a fallback is in use. Implementation verifies via computer use; minimal/no copy changes expected.

*Risks / blast radius:*
- **Behavior change:** `AdminDisabled` / `Unavailable` / `RequiresUpgrade`-no-BYOK no longer auto-clear. Users who relied on the auto-clear (e.g. an admin disabled a model and expected the profile to drop it) will now see the selection preserved with a runtime fallback. This is the desired "reversible" behavior but is a visible change — call it out in the PR description and changelog (`CHANGELOG-BUG-FIX`).
- A truly-permanently-deleted-but-still-recognized model leaves a stale preference (mitigated: user can reselect; unrecognized ids already preserved). Acceptable per the acceptance criteria.
- Effective runtime fallback must not break the request path: ensure the fallback model is actually used for the request when the selected model is unusable (`get_preferred_base_model` returns the `LLMInfo` used by the request path). Verify existing tests (`active_models_fall_back_to_usable_choice_or_custom_endpoint_when_default_disabled`, `llms_tests.rs:506`) still pass and extend.
- Cloud sync: removing the clear stops the bad cross-device propagation. Ensure no new sync race from preserving.
