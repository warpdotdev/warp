*Proposed change: Demote expected GraphQL response-error logging*

*Summary:* `send_graphql_request` currently emits `safe_error!` for every GraphQL response with an `errors` array, flooding Sentry for an expected, handled protocol outcome. Change that log site to `safe_warn!` while preserving the separately actionable disabled-account report.

*Key design choices:* Keep the existing safe/full message content and response handling unchanged; change only the severity of the broad GraphQL response-error log; leave the `User not in context: Not found` `report_error!` and `AuthEvent::UserAccountDisabled` path untouched.

*Design alternatives:* Keeping `safe_error!` preserves maximum local severity but violates the logging guidance and caused the reported Sentry flood. Demoting to `safe_info!` or `safe_debug!` would reduce signal too far for a recoverable, non-ideal server response and would diverge from the established demotion pattern. Reworking GraphQL error handling or changing the separate `network_logging.rs` reports would broaden this focused fix without being required to address APP-4928. The selected `safe_warn!` demotion matches the logging guideline for expected/recoverable conditions and the existing `safe:`/`full:` macro contract.

*Root cause / approach:* At `crates/warp_server_client/src/graphql_helpers.rs:58-62` (commit `47234cc6d8dcd29e06a06162c95ad38fddfa929`), any `response.errors.is_some()` path logs with `warp_core::safe_error!`, even though GraphQL errors are a normal protocol response that the client continues to process. The actionable disabled/deleted-account case is separately recognized at lines 66-75 and reports through `report_error!` plus `AuthEvent::UserAccountDisabled`; that branch must remain unchanged. `safe_warn!` has the same safe/full signature, so the implementation is a one-macro substitution with no control-flow or API change.

*Affected files:* `crates/warp_server_client/src/graphql_helpers.rs` (change the broad response-error log only); `crates/warp_server_client/src/graphql_helpers_tests.rs` (existing error-path tests remain the regression coverage); no changes to `crates/warp_server_client/src/network_logging.rs` or `crates/warp_logging/src/native.rs`.

*Open questions resolved:* The optional `network_logging.rs:82` and `:97` `report_error!` sites are explicitly deferred; they are separate channel-teardown behavior and require their own actionability review. No UI or user-visible behavior changes are in scope. *Risks / blast radius:* The only runtime change is the emitted log severity for expected GraphQL responses. The actionable account-disabled path, authentication behavior, returned errors, and response-data handling must not change. `safe_warn!` retains the safe/full redaction boundary. On current main, `sentry_log_filter` maps both Error and Warn to breadcrumbs, so this aligns severity and prevents event creation on older release channels where the prior Error mapping produced Sentry events.
