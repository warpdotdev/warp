*Spec: fixed-height expandable toasts with an at-most-3 active-toast cap*

Linear: APP-4889 — Improve toast UI: fixed height with expandable long messages; cap active toasts.
Target repo: `warpdotdev/warp` (the Warp desktop client, Rust/GUI app).
Estimate: M (3). Originating thread: Slack channel C0BDQDW8V5E, message ts 1784571137.565609 (requester Andy Carlson).

== PRODUCT ==

*Summary:* The Warp desktop client's general-purpose toast stack (`DismissibleToastStack`) currently grows without bound in two ways: a single long message makes one toast arbitrarily tall, and an error flood stacks unlimited toasts — the "un-ending wall of errors" a user hit. This change gives every toast a fixed collapsed height with an accessible expand/collapse affordance for long messages, and caps the stack to at most 3 active toasts (evicting the oldest when a new one arrives), so the toast surface stays bounded and readable under error floods.

*Key design choices:*
1. Fix both problems in the shared `DismissibleToastStack` / `DismissibleToast` component (`app/src/view_components/dismissible_toast.rs`), so both instances the Workspace owns (`toast_stack` and `update_toast_stack`) get the fix at once.
2. Mirror the proven in-repo pattern from `AgentNotificationToastStack` (`app/src/ai/agent_management/notifications/toast_stack.rs`) for BOTH features: character-based truncation + a chevron/link affordance for expand/collapse, and a `while len > N { evict oldest }` cap. Reuse, don't reinvent.
3. Overflow policy = dismiss-oldest (evict the oldest toast when a new one exceeds the cap). It bounds the stack, surfaces the most recent (most relevant) errors, and matches the existing notification-toast behavior.

*Behavior* (numbered, testable invariants from the user's view):
1. Fixed collapsed height: a toast renders at a bounded collapsed height regardless of message length. Long messages are truncated (with an ellipsis) so the collapsed toast never exceeds ~2 wrapped lines at the toast font size; a `ConstrainedBox` max-height safety clamp guarantees the collapsed height is bounded even for edge-case wrapping.
2. Expandable long messages: when a toast's message is longer than the collapsed cap, an expand/collapse affordance is visible. Activating it toggles the toast between collapsed (truncated, fixed height) and expanded (full message, wraps naturally within the fixed toast width). When expanded, the affordance toggles back to "collapse".
3. Affordance only when needed: short messages (within the collapsed cap) show NO expand affordance and are not expandable.
4. Per-toast expand state: each toast remembers its own expanded/collapsed state independently. Expanding one toast does not expand others.
5. Accessible affordance: the expand/collapse control is keyboard-focusable and activatable, has an accessible role (button) and a dynamic accessible label ("Show more" when collapsed / "Show less" when expanded), so screen-reader users can perceive that a message is truncated and expand it.
6. At most 3 active toasts: the stack never displays more than 3 toasts at once, in any combination of ephemeral and persistent toasts.
7. Dismiss-oldest overflow: when a new toast is added that would make the stack exceed 3, the oldest toast is evicted before the new one is shown, so the stack shows the 3 most-recent toasts. The evicted toast's auto-dismiss timer (if ephemeral) is aborted so no late dismissal fires against an already-removed toast.
8. Existing per-object dedup preserved: adding a toast with an `object_id` still auto-dismisses older toasts sharing that `object_id` (the existing `dismiss_older_toasts` behavior) before the cap is applied. The cap and dedup compose without interfering.
9. Existing dismissal preserved: manual close (close button on hover/mobile) and ephemeral timeout dismissal continue to work unchanged.
10. Both toast stacks fixed: the workspace `toast_stack` (general errors/success/cloud-object messages) and `update_toast_stack` (app-update toasts) both gain the cap and fixed-height behavior because the change is in the shared `DismissibleToastStack`/`DismissibleToast` component.
11. No regression to adjacent toast surfaces: `AgentToastStack` (`app/src/view_components/agent_toast.rs`) and `AgentNotificationToastStack` (`app/src/ai/agent_management/notifications/toast_stack.rs`) are NOT modified by this change. The latter already has its own cap (2) and expand/collapse; the former is out of scope (see Open questions).

*Design alternatives:*
- Overflow policy — (a) **dismiss-oldest** (chosen): evict the oldest toast when a new one exceeds the cap. Pros: bounds the stack, surfaces the most-recent (most relevant) errors, matches the existing `AgentNotificationToastStack` pattern (`:150`), simple O(1)-amortized. Cons: a persistent error the user hasn't read is silently removed from the toast surface. Mitigation: the most common persistent-error floods (tab/model config errors) are also surfaced in Settings, and per-`object_id` dedup already collapses stale errors; the cap is a last-resort bound, not the only signal. (b) Queue (show 3, queue the rest, promote as older ones dismiss): Pros: no error is lost. Cons: significantly more complex (queue + promotion + ephemeral-timer coordination); for a true flood the user still sees a wall over time; ephemeral timeouts make queued items stale. (c) Drop-newest (ignore new toasts when at 3): Pros: simplest. Cons: shows stale errors and hides the actual current failure — wrong for an error-flood scenario. → Chose (a).
- "Fixed height" mechanism — (a) **Character-based truncation + optional max-height clamp** (chosen): truncate `main_text` to a collapsed char cap with `…`, render with wrap; add a `ConstrainedBox::with_max_height` safety clamp sized to ~2 lines so the collapsed height is bounded even at wrapping edge cases. Pros: deterministic, the truncation decision is a pure function that is unit-testable, and it reuses the exact proven pattern from `item_rendering.rs`. Cons: char cap is approximate vs. pixel width (a 140-char string of wide glyphs could still wrap to 3 lines — the max-height clamp catches that). (b) Pure pixel max-height clamp only (like `AgentToast`): Pros: truly fixed height. Cons: clips mid-line with no clean affordance trigger; "is this truncated?" needs text measurement that the retained-mode UI doesn't expose cheaply. (c) Strict single fixed pixel height for ALL toasts (short ones get whitespace): Pros: maximally uniform. Cons: short toasts look padded/odd; not what users expect. → Chose (a): truncation for the affordance trigger + a max-height safety clamp for the "fixed height" guarantee.
- Expand affordance — (a) **Explicit "Show more"/"Show less" link or chevron button** (chosen): mirrors `render_expand_chevron`; discoverable and accessible. (b) Click-whole-body-to-expand: conflicts with the existing `on_body_click` navigation behavior (`DismissibleToast::on_body_click` / `ClickBody`), so rejected. (c) Hover-to-expand: not discoverable, not accessible, conflicts with the close-button hover. → Chose (a).
- Where to enforce the cap — (a) **Inside `DismissibleToastStack`** (chosen): applies to BOTH `toast_stack` and `update_toast_stack` uniformly, and is the natural owner of the `toasts` vec. (b) Only at the workspace `toast_stack` call sites: would miss `update_toast_stack` and require repeating the logic everywhere. (c) In the global `ToastStack` model: would not cover direct callers that push to the view handle (e.g. `update_toast_stack`, log-bundle, CLI results). → Chose (a).
- Cap value — (a) **3** (chosen): the requester explicitly asked for "at most 3 active toasts." (b) 2 (matching `AgentNotificationToastStack`): rejected because the requester specified 3, and these are different surfaces. (c) A user-configurable setting: out of scope for this change; can be added later. → Chose (a). The constant is named `MAX_VISIBLE_TOASTS` so it is a one-line change to tune.
