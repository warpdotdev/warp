*Proposed change: Add a `/view-logs` TUI slash command that bundles the user's TUI logs into a timestamped zip, reveals it in the OS file manager when feasible, and always prints the absolute path.*

*Summary:* The Warp GUI already lets users zip their logs and reveal the bundle in Finder/Explorer via the Command Palette "View Logs" action. The headless TUI has no equivalent. Add a TUI-only `/view-logs` slash command that reuses `warp_logging::create_log_bundle_zip()`, attempts a best-effort reveal in the platform file manager, and always surfaces the absolute zip path so the command stays useful over SSH / headless / CI where no file manager can open.

*Key design choices:*
1. **TUI-only** — register the command only when `settings::settings_mode() == SettingsMode::Tui` (the exact `MCP` precedent at `app/src/search/slash_command_menu/static_commands/commands.rs:652`), so the shared registry stays clean and no GUI slash-command execution arm is needed (the GUI already has Command Palette "View Logs").
2. **TUI-local, best-effort reveal + always-print-path, with an SSH guard** — the TUI's `open_file_path_in_explorer` is a no-op in the headless delegate (`crates/warpui/src/platform/headless/delegate.rs:84`), so introduce a small TUI-local OS-shell helper (`open -R` / `explorer /select,` / `xdg-open <parent>`). The reveal is best-effort and is **skipped entirely when an SSH session is detected** (the `SSH_CONNECTION` or `SSH_TTY` env var is set), going straight to path-only — opening a local file manager over SSH is useless at best and misleading at worst. The absolute path is always shown via `show_success_hint`, guaranteeing usefulness when no file manager can open.
3. **Reuse `create_log_bundle_zip` with a test seam** — add a parameterized `create_log_bundle_zip_in(dir, name)` (mirroring the existing `collect_log_paths_in` "for testability" pattern at `crates/warp_logging/src/native.rs:347`) so the bundle behavior is deterministically unit-testable without depending on the one-shot global `LOG_STATE`.

*Design alternatives:*
- **Reveal mechanism:**
  - *Reuse `ctx.open_file_path_in_explorer`* — rejected: in the headless delegate (`crates/warpui/src/platform/headless/delegate.rs:84`) it is a no-op (`// Unsupported.`), verified, so the TUI would silently do nothing.
  - *Shared helper in `warp_util`* — cleaner and reusable across surfaces; could eventually back the GUI too. Rejected for this S-sized change to keep blast radius minimal and the GUI path untouched. Noted as a future refactor if a second caller appears.
  - *TUI-local OS-shell helper (chosen)* — minimal, touches only the TUI, easy to make best-effort with a path-only fallback. Split into a pure `build_reveal_command(path) -> Option<Command>` (unit-testable per platform via cfg), a pure `should_attempt_reveal() -> bool` that returns `false` when `SSH_CONNECTION` or `SSH_TTY` is set (unit-testable by setting/unsetting env in-process), and a best-effort `reveal_path_in_file_manager(path)` wrapper that returns early when `should_attempt_reveal()` is false.
  - *Always path-only (no reveal attempt)* — simplest, but the ticket explicitly asks to reveal when feasible; rejected in favor of best-effort reveal + always-print-path (with the SSH guard).
  - *Detect SSH by a richer heuristic (e.g. $TERM program, parent process)* — more accurate than env vars but fragile and out of scope for an S fix; `SSH_CONNECTION`/`SSH_TTY` is the conventional, low-false-positive signal and is what the requester approved.
- **Command scope (GUI vs TUI):**
  - *Both GUI and TUI* — would surface `/view-logs` in the GUI slash menu too, but requires a GUI execution arm in `Input::execute_slash_command` and is redundant with the existing Command Palette "View Logs" (`workspace:view_logs` / `WorkspaceAction::ViewLogs` at `app/src/workspace/view.rs:24295`). Rejected.
  - *TUI-only via `settings_mode == Tui` gate (chosen)* — exact precedent of `MCP` (`commands.rs:652`); zero GUI impact. The TUI menu already filters by `slash_command_is_supported_in_tui` (`app/src/terminal/input/slash_commands/mod.rs:170`), so adding the `TuiSlashCommand::ViewLogs` variant + `from_static_command` arm is sufficient for the TUI to show and execute it.

*Root cause / approach:*
Feature (no repro needed). The TUI slash-command pipeline is: shared `StaticCommand` registry (`all_commands()`) → `TuiSlashCommand::from_static_command` decides which the TUI shows → `execute_tui_slash_command` matches the variant. Add the command at all three layers plus a TUI-local reveal helper, reusing `warp_logging::create_log_bundle_zip()` (the same function the GUI uses at `app/src/workspace/view.rs:6525`) for the zip. The blocking zip task + callback pattern mirrors the GUI `Workspace::view_logs` (`app/src/workspace/view.rs:6523`) and the existing TUI `ExportToFile` arm (`crates/warp_tui/src/terminal_session_view.rs:2524`).

*Open questions resolved:*
- (a) Reveal vs path-only: reveal-when-feasible + **always** print the absolute path; the reveal is **skipped under SSH** (when `SSH_CONNECTION` or `SSH_TTY` is set) so SSH/headless/CI sessions go straight to path-only without a useless/misleading local file-manager open attempt. Settled from the ticket requirements, the headless-delegate no-op finding, and the requester's approval decision.
- (b) TUI-only vs also GUI: TUI-only via the `settings_mode == Tui` gate. Settled from the codebase — the GUI already has Command Palette "View Logs", and `MCP` is the established TUI-only precedent.
- Testability: the one-shot global `LOG_STATE` (`crates/warp_logging/src/native.rs:637`) makes `create_log_bundle_zip()` untestable in isolation, so mirror the existing `collect_log_paths_in(dir, name)` seam with `create_log_bundle_zip_in(dir, name)`. Settled from the codebase.
- Feature flag: none — a generally-useful, low-risk utility; the `settings_mode == Tui` gate plus the inherent non-WASM/`local_fs` TUI context is sufficient. Settled from the codebase (no flag needed for stable TUI utilities, per the feature-flag guidance).

*Risks / blast radius:*
- The shared `StaticCommand` registry is built once at `Registry::new()`; gating on `settings_mode()` at construction follows the proven `MCP` path — no GUI impact, and the GUI `view_logs` flow is untouched.
- Adding a `TuiSlashCommand` variant is exhaustive only over `execute_tui_slash_command` (verified — the only non-test match site), so the build fails closed if the arm is missing, which is desirable.
- The reveal helper shells out to OS commands; failures are contained (best-effort, `log::warn!`'d, never crashes) and the path is always printed, so a broken/absent opener never degrades the command below "print the path". The SSH guard (`should_attempt_reveal`) is read from the process env at call time and never spawns, so it adds no platform risk and cannot misfire into a bad spawn.
- Refactoring `create_log_bundle_zip()` to delegate to `create_log_bundle_zip_in` must preserve the exact existing output (timestamped `<stem>-YYYYMMDD-HHMMSS.zip` under the log dir, current + rotated logs) so the GUI `view_logs` behavior is unchanged — covered by reusing the same body and the existing `collect_log_paths_in` tests (criterion 5).
