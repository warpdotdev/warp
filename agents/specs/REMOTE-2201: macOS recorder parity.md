*Spec: macOS recorder parity — window-scoped recording and playback speed (REMOTE-2201)*

== PRODUCT ==
*Summary:* Bring the macOS avfoundation recorder in line with the Linux/X11 recorder for the two `RecordingConfig` capabilities that the existing macOS PR predates: playback-speed multipliers and window-scoped recording. The recommended first implementation is display capture plus a static crop, which keeps the existing ffmpeg sidecar and encode settings while making the macOS limitations explicit.

*Key design choices:*
1. Rebase the existing `factory/macos-video-recording` work onto current `master` first, then mirror the Linux `setpts` behavior and shared `RecordingConfig` fields; Linux and its encode settings remain untouched.
2. Implement `Target::Window` with avfoundation display capture plus a `crop=W:H:X:Y` video filter. Resolve the target's initial CoreGraphics bounds, select the display containing it, subtract that display's origin, and round the crop dimensions down to even values for `yuv420p`.
3. Treat crop capture as a visible-region snapshot: it does not follow window moves/resizes and an occluding window is recorded. ScreenCaptureKit is documented as the follow-up for true per-window, occlusion-safe tracking, not part of this M-sized parity change.

*Behavior* (numbered, testable invariants):
1. Before any macOS recorder code is changed, the implementation branch is rebased onto current `master` so `RecordingConfig` includes `playback_speed_multiplier` and `target` from `eb1c691a` and `ea24a1a5`; the existing macOS recorder lifecycle still compiles.
2. `Target::Screen` (including the default target) continues to capture the whole main display using the current avfoundation path and `Capture screen 0:none`, with the current frame rate, cursor/click capture, codec, pixel format, fast-start flag, output-size cap, and cleanup/finalization behavior.
3. For `Target::Window { window_id, .. }`, start resolves `mac::window::window_by_id(window_id)`, identifies the active display that fully contains the initial window bounds, and captures that display with `Capture screen N:none`. The crop origin is display-relative (`window_origin - display_origin`) in the top-left coordinate system; no y-axis inversion is introduced. The encoded crop width and height are positive even integers and represent the initial window bounds.
4. A missing window, zero/negative dimensions, a window spanning displays, or a crop outside the selected display fails closed with a clear `RecordingError` before returning a live recording; it never silently falls back to the full screen or another display.
5. If `playback_speed_multiplier > 1.0`, macOS adds `-vf setpts=(1/N)*PTS` using the same six-decimal formatting and reciprocal as Linux. For a window target, the single filter is ordered `crop=W:H:X:Y,setpts=...`; for a screen target it is only `setpts=...`. Values `<= 1.0` omit the filter.
6. `-t <max_duration>` is an avfoundation input option and appears before `-i`; this caps wall-clock capture independently of `setpts`. `-fs <max_size_bytes>` remains an output option. A stopped recording remains playable and has the same `RecordingOutput`/cleanup semantics as the current macOS recorder.
7. The crop is static for the lifetime of one recording. If the window moves or resizes after `start`, the original rectangle remains captured; if another window covers it, the occluder appears in the crop. These are documented macOS limitations relative to Linux `x11grab -window_id`, not hidden behavior.
8. Linux `Target::Screen`, `Target::Window`, playback speed, duration limiting, and all unrelated screenshot/input paths remain behaviorally unchanged.

*Design alternatives:*
- *Window capture:* avfoundation display + static `crop` (chosen): S–M, reuses the current ffmpeg process and preserves the existing MP4 lifecycle; it is visible-region capture and cannot track moves/resizes. ScreenCaptureKit `SCContentFilter(windowID:)` (deferred): true per-window, occlusion-safe, and move/resize tracking, but requires macOS 12.3+ handling and replacing the ffmpeg avfoundation input with a custom CMSampleBuffer/ffmpeg pipeline (L-sized and a materially larger blast radius).
- *Display selection:* always use `Capture screen 0` (rejected for window targets: wrong on a non-main display) vs. resolve the active display containing the window and use its `Capture screen N` ordinal plus display-relative crop (chosen). If active-display mapping or full containment cannot be established, fail closed rather than capture a different surface.
- *Filter construction:* append independent `-vf` options (rejected: ffmpeg accepts one effective video filter and the crop/speed behavior would be order-dependent) vs. build one ordered filter string (`crop` then `setpts`, or `setpts` alone) (chosen). A pure argument/geometry builder should make this contract unit-testable without spawning ffmpeg.
- *Landing:* if PR #13742 is still open, the preferred operational landing is to rebase its branch and add these commits there; if it has merged or is being finalized as-is, use this follow-up branch/PR against the merged recorder substrate. In either case, the rebase and the behavior contract above are mandatory.

*Proposed changes:*
1. Rebase `factory/macos-video-recording` (or this follow-up branch when #13742 is finalized) onto `master`, retaining the current macOS recorder and resolving any shared-config/`RecordingHandle` conflicts in favor of the Linux fields and cfg widening.
2. Refactor the macOS command construction into a pure, testable helper (for example, a `Vec<OsString>`/argument builder plus the process launcher). Keep the current avfoundation input, cursor/click capture, dimensions, codec, preset, `yuv420p`, `+faststart`, ffmpeg supervision, SIGINT finalization, and `-fs` behavior. Move `-t` before `-i`.
3. Add a macOS geometry/display resolver. Resolve `window_by_id`; enumerate active displays with CoreGraphics; choose the display whose bounds fully contain the window; compute `x = window.x - display.x`, `y = window.y - display.y`, `w = floor(window.width) & !1`, and `h = floor(window.height) & !1`; reject non-positive or out-of-bounds results. Use the chosen display ordinal in `Capture screen N:none`. Keep the `Target::Screen` path on the current main-display selector.
4. Build the output filter from the target crop and multiplier: `crop={w}:{h}:{x}:{y}` for a window, optionally followed by `,setpts={1/N:.6}*PTS`; `setpts={1/N:.6}*PTS` alone for a screen with a multiplier; no `-vf` at or below real-time speed. Do not change Linux command construction.
5. Add focused macOS argument/geometry tests in the existing `recording_tests.rs` or a macOS recorder test module. The tests must exercise the pure seams, not a real AVFoundation session, so they run deterministically on a macOS runner. Preserve the existing ffmpeg-absent/drop tests and resolve the open bot review thread at `mac/recording.rs:93`.
6. Document the static-crop, occlusion, multi-display, and move/resize contract in code comments or the implementation PR description. Do not add ScreenCaptureKit or a new dependency in this change.

*Open questions resolved:*
- *Which window approach?* Crop is selected for parity-now and the existing ffmpeg architecture; ScreenCaptureKit is a follow-up.
- *What happens on a second display?* Use the active display containing the window, the matching `Capture screen N`, and coordinates relative to that display. A spanning/unmappable window fails closed.
- *What happens when the window moves or is covered?* The initial rectangle is intentionally static; subsequent pixels are whatever lies in that rectangle. This is explicit behavior, not a promise of Linux-equivalent tracking.
- *What happens when the target is absent?* `Target::Screen` remains the unchanged whole-main-display path.
- *Where should the fix land?* Rebase first; add parity to #13742 if it remains open, otherwise use the follow-up branch represented by this draft PR.

*Risks / blast radius:* Crop coordinates or display ordinals could capture the wrong display; mitigate with pure mapping tests, fail-closed validation, and a multi-display macOS manual run. Static crop is a known parity gap; document it and keep ScreenCaptureKit out of scope. Filter ordering could accidentally stretch the duration cap; assert `-t` before `-i` and preserve `-fs` after the input. Rebase conflicts could regress Linux or the existing macOS lifecycle; require Linux presubmit plus macOS compile/tests before marking the shared PR ready.
