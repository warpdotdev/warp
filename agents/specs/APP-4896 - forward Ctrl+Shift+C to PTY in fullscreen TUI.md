*Spec: Forward `Ctrl+Shift+C` to the foreground TUI on Linux when Warp has no copyable selection (fullscreen/alt-screen apps like Claude Code)*

== PRODUCT ==
*Summary:* When a fullscreen TUI application (e.g. Claude Code in fullscreen rendering mode) is running on Linux and manages its own internal text selection — a selection Warp cannot see because it lives in the TUI process, not in Warp's grid — pressing `Ctrl+Shift+C` (Warp's native Copy binding on Linux) currently does nothing: Warp intercepts the keystroke, finds no Warp-visible selection to copy, and swallows it. The foreground TUI never sees the key, so its internal selection is never copied. This spec makes Warp, in that exact situation, forward the copy intent to the foreground TUI by writing the `Ctrl+C` byte (`ETX` / `0x03`) to the PTY — the same byte Warp already writes for `Ctrl+C`, which Claude Code (and other fullscreen TUIs) interpret as "copy the active selection." This restores parity with Konsole and with Warp's own `Ctrl+C` behavior, while leaving every existing Warp-native copy path untouched.
*Key design choices:*
1. *Forward only when Warp has nothing to copy AND the alt screen is active.* The fallback is appended at the end of `copy()` and fires only when no Warp-owned selection (CLI subagent, cloud error screen, block-list/alt-screen grid selection, input editor, selected blocks) was copied AND the terminal is on the alt screen (a fullscreen TUI is in control). On the normal screen, `Ctrl+Shift+C` with no selection stays a silent no-op — we never send `0x03` to a plain shell prompt (that would be SIGINT).
2. *Forward `0x03` (ETX), the `Ctrl+C` byte — not a `Ctrl+Shift+C` escape sequence.* This reuses the proven `user_write_ctrl_c_to_pty` path (the exact mechanism `Ctrl+C` already uses to reach the TUI), is interpreted as "copy" by Claude Code and any TUI that binds `Ctrl+C` to copy, and requires no extended-keyboard-protocol negotiation. It is a deliberate, documented semantic substitution (the user pressed `Ctrl+Shift+C`; the TUI receives `Ctrl+C`), chosen for compatibility and consistency with existing `Ctrl+C` behavior.
3. *Linux only.* The bug is Linux-specific (`os:linux`, Manjaro report) and `Ctrl+Shift+C` is the Linux Copy binding. macOS uses `Cmd+Shift+C` and has different clipboard/selection conventions; Windows uses `Ctrl+C` for copy via `WindowsCopy`. Scoping the fix to Linux (via a `cfg!(target_os = "linux")` runtime check) limits blast radius and avoids changing unreported macOS/Windows behavior; it can be widened later if a macOS report arrives.

*Behavior* (numbered, testable invariants from the user's view):
1. On Linux, while a fullscreen TUI is active (alt screen on) and Warp has no copyable selection of its own, pressing `Ctrl+Shift+C` writes exactly one byte `0x03` (`ETX`) to the PTY so the foreground TUI can copy its own selection. (This is the fix.)
2. On Linux, while a fullscreen TUI is active and Warp *does* have a copyable selection (e.g. the user made a Warp alt-screen selection by click-dragging in Warp), `Ctrl+Shift+C` copies that Warp selection to the clipboard and writes *nothing* to the PTY. (Existing behavior, unchanged — the fallback must not fire.)
3. On Linux, while on the normal screen (no alt screen) with no selection, `Ctrl+Shift+C` remains a silent no-op: nothing is copied and nothing is written to the PTY. (Existing behavior, unchanged — the fallback must not fire.)
4. On Linux, `Ctrl+Shift+C` still copies CLI-subagent selected text, cloud-mode error-screen selected text, input-editor selected text, and selected blocks exactly as today; the fallback fires only when none of those produced a copy. (Existing behavior, unchanged.)
5. The `Ctrl+C` path is unchanged on every platform: it still clears Warp selections and forwards `0x03` to the active block / PTY exactly as today. (Regression guard.)
6. macOS and Windows behavior is unchanged: the fallback does not fire on those platforms. (Platform-scope guard.)
7. When an AI agent is in control of the active block, the fallback writes nothing to the PTY (inherited from the existing `is_agent_in_control()` early-return inside `write_user_bytes_to_pty`). (Safety guard, unchanged.)

*Design alternatives* (per decision point with more than one reasonable approach):
- *Guard condition — when should the fallback fire?* (a) **Alt-screen active AND no Warp selection copied** (chosen): append the fallback at the end of `copy()` and gate it on `cfg!(target_os = "linux") && self.selected_blocks.is_empty() && self.model.lock().is_alt_screen_active()`. Because every earlier branch in `copy()` `return`s on a successful copy, reaching the end means no CLI-subagent / error-screen / `selection_to_string` / input-editor selection was copied; the only remaining Warp-selection source is `selected_blocks`, which the `.is_empty()` check excludes. Pros: minimal, surgical change (one new branch at the end of `copy()`); preserves every existing copy path verbatim; never sends `0x03` to a normal-screen shell prompt (SIGINT risk avoided). Cons: the guard relies on the fact that earlier branches `return` early — a future change that adds a non-returning selection branch would need to keep `selected_blocks.is_empty()` (or an explicit `copied` flag) accurate; mitigated by a code comment and by the regression test that asserts no PTY write when a Warp alt-screen selection exists. (b) *Restructure `copy()` to return `bool copied` and forward iff `!copied && alt_screen_active`*: more explicit about "did we copy anything," but larger churn (signature change + all call sites) for the same behavior. Rejected for this ticket as higher churn; revisitable if `copy()` grows more selection sources. (c) *Fire on any "no Warp selection" regardless of alt screen*: rejected — it would send `0x03` (SIGINT) to a normal-screen shell prompt when the user presses `Ctrl+Shift+C` with nothing selected, a behavior change with real downside and no upside for the reported bug.
- *Bytes to write — `0x03` (ETX) vs a `Ctrl+Shift+C` escape sequence?* (a) **Forward `0x03` (ETX), reusing `user_write_ctrl_c_to_pty`** (chosen): the byte Warp already sends for `Ctrl+C`; Claude Code and any fullscreen TUI that binds `Ctrl+C` to copy will copy; no extended-keyboard-protocol negotiation needed; one-line reuse of an existing, tested helper. Pros: maximal compatibility (works with TUIs that only bind `Ctrl+C`), consistency with the existing `Ctrl+C` path (same byte, same `write_user_bytes_to_pty` guards), minimal code. Cons: a semantic substitution — the user pressed `Ctrl+Shift+C` but the TUI receives `Ctrl+C`; a TUI that binds `Ctrl+Shift+C` and `Ctrl+C` to *different* actions would run the `Ctrl+C` action. This is acceptable because (i) the reporter confirms `Ctrl+C` already copies in Claude Code fullscreen (so the `Ctrl+C` action *is* the desired copy), (ii) the alternative is today's broken no-op, and (iii) it is no worse than `Ctrl+C`'s existing behavior. (b) *Forward a real `Ctrl+Shift+C` escape sequence (CSI … ;6u via Kitty protocol, or `ESC c` under modifyOtherKeys)*: semantically faithful — the TUI sees the actual key. Rejected for this ticket: it only works if the TUI has enabled an extended keyboard protocol *and* explicitly bound `Ctrl+Shift+C`; many TUIs (including the reporter's Claude Code setup, which copies on `Ctrl+C`) do not, so it would no-op for them and fail to fix the bug. Constructing the correct sequence also depends on the negotiated protocol, adding complexity and fragility. It remains a viable future enhancement if a TUI is found that distinguishes the two keys and binds `Ctrl+Shift+C` specifically.
- *Platform scope — Linux only, or all non-Windows / all platforms?* (a) **Linux only, via `cfg!(target_os = "linux")`** (chosen): matches the `os:linux` report and triage direction; `Ctrl+Shift+C` is the Linux Copy binding; avoids changing macOS (`Cmd+Shift+C`) and Windows (`Ctrl+C`/`WindowsCopy`) behavior, which is unreported and out of scope. Using a runtime `cfg!` check (not `#[cfg]`) keeps the code compiled and tested on all platforms while activating the behavior only on Linux. Pros: smallest blast radius; can be widened later by changing one predicate. Cons: a macOS user running a fullscreen TUI with an internal selection would still see `Cmd+Shift+C` no-op until a follow-up widens the guard — acceptable since no such report exists and macOS alt-screen selection/clipboard conventions differ. (b) *All non-Windows (`#[cfg(not(windows))]`)*: would also change macOS. Rejected: unrequested, and macOS users expect `Cmd+C`/`Cmd+Shift+C` to interact with the system clipboard, not to forward `0x03` to the PTY. (c) *All platforms*: rejected for the same reason plus Windows already has its own copy semantics.

*Open questions resolved:*
- *Guard condition (alt-screen only?)* Resolved: **alt-screen active AND no Warp selection copied.** The fallback fires only at the end of `copy()` when `selected_blocks.is_empty()` (and no earlier branch returned) AND `is_alt_screen_active()`. It never fires on the normal screen, so a plain shell prompt never receives `0x03`. See design alternative 1(a) and behavior invariants #2, #3.
- *Bytes to write (0x03 vs escape sequence)?* Resolved: **`0x03` (ETX), the `Ctrl+C` byte**, reusing `user_write_ctrl_c_to_pty`. This is the byte Warp already sends for `Ctrl+C`, which the reporter confirms copies in Claude Code fullscreen; it maximizes compatibility and consistency. A real `Ctrl+Shift+C` escape sequence is rejected as incompatible with TUIs that only bind `Ctrl+C` (see design alternative 2). See behavior invariant #1.
- *Platform scope (Linux only)?* Resolved: **Linux only**, via `cfg!(target_os = "linux")` runtime check. Matches the `os:linux` report and triage; leaves macOS/Windows untouched. See design alternative 3(a) and behavior invariant #6. Can be widened later if a macOS report arrives.

*Risks / blast radius:*
- *Sending `0x03` to a TUI that doesn't bind `Ctrl+C` to copy:* `0x03` is SIGINT/Ctrl+C. A fullscreen TUI that binds `Ctrl+C` to "cancel/interrupt" rather than "copy" would, on `Ctrl+Shift+C` with no Warp selection, receive an interrupt. This is exactly what `Ctrl+C` already does today (and the reporter confirms `Ctrl+C` copies in Claude Code, i.e. Claude Code binds it to copy), so the fallback introduces no behavior the user couldn't already trigger with `Ctrl+C`. The guard (alt-screen + no Warp selection + Linux) ensures it fires only in the fullscreen-TUI-with-its-own-selection scenario the bug describes, never at a normal shell prompt. If a TUI is found where this is undesirable, a feature flag (see *Proposed changes* #3) can gate it in a follow-up.
- *`copy()` early-return invariant:* the guard depends on every selection branch before the fallback `return`ing on a successful copy (so reaching the end means "nothing was copied"). Today's branches all do (`selected_blocks` is the only non-returning one, and it's guarded by `.is_empty()`). A code comment and the `copy_does_not_forward_when_alt_screen_has_warp_selection` regression test pin this invariant.
- *`write_user_bytes_to_pty` side effects:* the helper clears selected blocks (a no-op here, since the guard already ensures `selected_blocks.is_empty()`), updates scroll-position locking, and emits a non-editor typed event — all identical to today's `Ctrl+C` path and safe on the alt screen.
- *Agent-in-control sessions:* `write_user_bytes_to_pty` early-returns when the active block `is_agent_in_control()` (line 9286-9288), so the fallback writes nothing during an agent-controlled command. This is the desired safety behavior (behavior invariant #7) and is inherited for free.
- *macOS/Windows unchanged:* the `cfg!(target_os = "linux")` guard means the new branch compiles on all platforms (so CI exercises it) but activates only on Linux; macOS/Windows copy behavior is byte-for-byte unchanged (behavior invariant #6).
- *Existing copy tests:* `test_copy_on_select` (line 2851, normal screen) and `test_alt_screen_copy_on_select` (line 2893, alt screen *with* a Warp selection) are unaffected — the first never enters the alt screen, the second has a Warp selection so `copy()`/`maybe_copy_selection_to_clipboard` copies and the fallback does not fire.
