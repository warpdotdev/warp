# Spec: TUI `/logout` slash command (APP-4919)

*Target repo: `warpdotdev/warp` — headless TUI in `crates/warp_tui` (NOT the GUI `app/` surfaces, NOT `warp-server`). Base branch: `master`. All file references are commit-pinned to `8d2759cffb578dc05d5a2160786212ba1221dbd7` (HEAD of `master` at investigation time).*

== PRODUCT ==
*Summary:* Add a `/logout` slash command to the headless Warp TUI that logs the user out (clears credentials and user-dependent state, tears down the live terminal sessions) and returns the TUI to the existing sign-in/auth page — the `login_placeholder` that shows the device-authorization verification URI + user code — so the user can re-authenticate without restarting the process.

*Key design choices:*
1. **Reuse the GUI's canonical `auth::log_out` for user-state cleanup**, layered with TUI-specific session teardown + root-view flip + device-auth re-arm. Do not reinvent cleanup (prevents cross-account data leakage) and do not wipe-and-restart the process (the app stays alive and re-arms device auth in place).
2. **Immediate logout, no confirm dialog.** Consistent with the TUI's existing `/exit` (exits immediately). Building a TUI confirm modal is out of scope for an M-sized change; a "confirm when a command is running" enhancement is explicitly deferred.
3. **Re-arm device auth by calling `AuthManager::authorize_device` again.** The existing `AuthManager` subscription set up in `app/src/tui/mod.rs::init` already handles the whole flow (`ReceivedDeviceAuthorizationCode` → show URI/code, `AuthComplete` → `LoggedIn`), so re-arming is one call, not a re-init.

*Behavior* (numbered, testable invariants from the user's view):
1. When the user is logged in and in a terminal session, typing `/` and then `logout` (or `log`) surfaces `/logout` in the slash-command menu, with the description "Log out and return to the sign-in page".
2. Accepting/executing `/logout` (enter or menu selection) logs the user out: the terminal session(s) disappear and the screen returns to the existing auth page (`login_placeholder`) showing "Sign in to continue" plus, once the new device code is received, the verification URI ("Open {uri} in your browser") and user code ("and enter code: {code}") — the same surface shown at first launch when not logged in.
3. From that auth page, the user can complete device auth again (open the URI, enter the code) and is returned to a fresh terminal session, logged in as the (possibly different) account. No process restart is required.
4. `/logout` is only available when logged in (i.e. while `RootTuiState::Terminal`). On the auth page there is no input/menu, so `/logout` cannot be invoked there. It does not appear in the GUI's slash menu (registered only in TUI mode, mirroring `/exit` and `/view-logs`).
5. `/logout` takes no arguments; accepting it from the menu executes it immediately (no argument entry step).
6. Edge — a long-running shell command in the terminal when `/logout` runs: the session is torn down (PTY killed), same as `/exit` force-terminates. No panic, no hang.
7. Edge — `/logout` while an in-flight device-auth poll from a prior session is still running: logout clears credentials and starts a fresh device-code request; a stale token-exchange completing must not silently re-authenticate the old user (see Risks).
8. Edge — `/logout` when the user is somehow already logged out (unreachable via the menu per invariant 4, but defensively): the cleanup path is idempotent (clearing already-empty state) and re-arming `authorize_device` is safe.

*Design alternatives* (per decision point with more than one reasonable approach):
- **Logout cleanup scope** — (a) *Reuse `auth::log_out` (full reset incl. DB wipe) + TUI session teardown* [CHOSEN] vs (b) *narrow `AuthManager::log_out` (clear creds only) + TUI session teardown* vs (c) *tear everything down and re-exec the process*.
  - (a) pros: canonical path, no cross-account data leakage (DB wipe + model resets + polling stopped), no reinvented cleanup; cons: wipes local sqlite3 (sessions/cloud objects gone — the GUI's existing "Logout v0" behavior, correct for account-switching), references GUI-oriented dispatch (`root_view:log_out` action) that is a no-op for the TUI root view, and touches singleton models that must all exist in the TUI app context (see Risks).
  - (b) pros: smallest blast radius, no DB wipe; cons: the old user's cached cloud objects/conversations and polling loops would linger with stale context until the new user's data loads — data-leakage risk the `auth::log_out` comment explicitly calls out.
  - (c) pros: cleanest slate; cons: heavy, restarts the process (loses the in-process device-auth subscription wiring), worse UX than in-place re-arm. Rejected.
  - Selected: (a). It matches the GUI's logout semantics, prevents leakage, and the DB-wipe is the documented intentional behavior for account switching. The implementer must confirm every singleton referenced in `auth::log_out` exists in the TUI app context (the TUI runs the full headless app core via `warp::run_tui`, so they should — but verify; if any is GUI-only and absent, gate that reset behind `cfg`/existence or extract a TUI-safe subset, and record which).
- **Confirm dialog** — (a) *Immediate logout, no confirm* [CHOSEN] vs (b) *confirm when a command is running / unsaved state exists* (mirroring `auth::maybe_log_out`).
  - (a) pros: consistent with TUI `/exit` (immediate), simple, fits M scope; cons: a running command is killed without warning.
  - (b) pros: matches GUI safety; cons: requires a TUI confirm modal primitive (no native modal in headless TUI; would need a new `TuiElement` dialog + key handling), well beyond M scope.
  - Selected: (a). A "confirm when a long-running command is active" follow-up is explicitly deferred (note in the PR/task). The `/exit` precedent (force-terminate, no confirm) makes this consistent.
- **Re-arm device auth** — (a) *call `AuthManager::authorize_device` again* [CHOSEN] vs (b) *drop & re-create the `AuthManager`/`TuiLoginModel`* vs (c) *re-exec the process*.
  - (a) pros: one call; the existing `AuthManager` subscription in `init` already handles `ReceivedDeviceAuthorizationCode`/`AuthComplete`/`AuthFailed` for the whole app lifetime, so the auth page + browser-open + login-completion all just work; cons: a stale in-flight device-code poll from before logout could complete and re-auth the old user (see Risks).
  - (b) pros: fully fresh state; cons: re-creating singletons mid-run is fragile and would require re-subscribing; unnecessary.
  - (c) rejected (same as above).
  - Selected: (a).
- **Where the logout orchestration lives** — (a) *a new `pub fn log_out` in `app/src/tui/mod.rs` (next to `init`/`set_login_phase`), re-exported via `tui_export`* [CHOSEN] vs (b) *inline in `TuiTerminalSessionView::execute_tui_slash_command`*.
  - (a) pros: app-level logout logic lives at the app layer (mirrors `init`/`set_login_phase`), the per-session view just calls it; the session-teardown/root-flip is driven by the existing `TuiLoginModel` subscription in `session.rs` (extend it for `LoggedOut`), so the per-session view never tears down itself synchronously; cons: a small new public API surface.
  - (b) pros: localized; cons: puts app-level auth/teardown logic in the per-session view and would tear down the dispatching view from within its own handler (panic risk).
  - Selected: (a). The per-session view calls the re-exported `log_out`; the actual session teardown + root flip is performed by the `TuiLoginModel` subscription handler in `session.rs` on `TuiLoginEvent::LoggedOut`, **deferred** so the dispatching view's handler completes first (mirror how `TuiLoginEvent::LoggedIn` creates sessions from a model-event handler).

*Risks / blast radius:*
- **`auth::log_out` in the TUI app context (TOP RISK):** `auth::log_out` calls `.handle(ctx)` on many singleton models and dispatches `root_view:log_out` to GUI root views. The implementer MUST verify every referenced singleton exists when running under `warp::run_tui` (the TUI runs the full headless app core, so they should). If any is GUI-only/absent, `.handle(ctx)` panics. Mitigation: confirm by building+running the TUI with the `tui` feature; gate or extract a TUI-safe subset for any absent model and record which. The `root_view:log_out` action dispatch is a no-op for `RootTuiView` (it handles only `RootTuiAction::ExitApp`) — confirm it's harmlessly logged, not panicked.
- **Self-teardown from within the dispatching session view:** if `TuiLoginEvent::LoggedOut` is delivered synchronously inside `execute_tui_slash_command`, the current `TuiTerminalSessionView` would be dropped mid-handler → panic. Mitigation (required): the logout path must defer the session teardown + root flip so the dispatching handler completes first — e.g. emit the event and let the framework deliver it on a later turn, or use `ctx.spawn`/a deferred typed action. The implementer must verify the teardown is non-synchronous and add a test that dispatches `/logout` from a live session view without panicking.
- **Stale in-flight device-auth poll:** if a prior `exchange_device_access_token` completes after logout, `on_user_fetched` could re-authenticate the old user. `authorize_device` clears stale credentials first (`auth_manager.rs:272`), but the old poll holds its own `details`. Mitigation: verify whether `AuthManager` cancels/drops the in-flight poll on `log_out`/re-`authorize_device`; if not, document the residual risk or cancel it. Low likelihood (user-initiated logout makes the prior code stale) but must be reasoned about.
- **DB wipe on every logout:** `persistence::remove` deletes the local sqlite3 DB (sessions + cloud objects). This is the GUI's documented "Logout v0" behavior and is correct for account-switching, but a user re-authenticating as the *same* account will find local history gone. Document as known behavior in the PR; changing it is out of scope.
- **Blast radius:** the change is additive to the TUI slash surface and the TUI login state machine. Existing TUI commands (`/exit`, `/view-logs`, `/mcp`, `/agent`, `/new`, etc.) and the existing login → terminal flow are untouched (the `LoggedIn` branch is unchanged; only a new `LoggedOut` branch is added). The shared `auth::log_out` is reused unmodified.
