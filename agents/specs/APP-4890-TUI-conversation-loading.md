*Proposed change: Prevent stuck TUI conversation restores and left-align the loading indicator*

*Summary:* A server conversation restore can leave the TUI in `ConversationRestoreState::Loading` forever when the HTTP future never resolves, and the startup loading placeholder is horizontally centered instead of starting at the left edge of the TUI content area.
*Key design choices:* Bound the restore future at the TUI spawn boundary with the shared WarpUI async timeout (30 seconds), map a timeout to `None` so the existing restore-result failure path owns the state transition and message, and replace horizontal-plus-vertical centering with vertical-only centering whose content uses `CrossAxisAlignment::Start`.
*Design alternatives:* Keeping the current unbounded request was rejected because an accepted-but-never-completing server request has no callback or state transition. Adding the timeout inside the shared `load_conversation_from_server`/history loader was rejected because the TUI restore boundary is the smallest scope that covers both local-metadata and direct server-token restore paths without changing shared loader semantics. Keeping horizontal centering or removing all centering was rejected because the requested behavior is left alignment while retaining the current vertical placement. There are no meaningful alternatives to updating the existing render-to-lines assertion; it must encode the corrected alignment and preserve the spinner repaint check.
*Root cause / approach:* `TuiTerminalSessionView::restore_conversation` (`crates/warp_tui/src/terminal_session_view.rs:1615 @ 730a4acc00cc95ed4b6a12726d197ae2a9d93e91`) sets `ConversationRestoreState::Loading` and spawns the future returned by `load_conversation_by_server_token`. The direct server path (`app/src/ai/blocklist/history_model/conversation_loader.rs:297 @ 730a4acc00cc95ed4b6a12726d197ae2a9d93e91`) awaits `AIClient::get_ai_conversation` without a deadline; if that future remains pending, the `ctx.spawn` callback never calls `handle_conversation_restore_result`, so loading never leaves `Loading`. Wrap the future immediately before `ctx.spawn` with the shared `warpui::r#async::FutureExt::with_timeout` and a 30-second constant, converting timeout errors to `None`. The existing `handle_conversation_restore_result`/`fail_conversation_restore` path must then show the existing clear failure message for startup and preserve transient-hint behavior for conversation-list restores. In `crates/warp_tui/src/ui.rs:58 @ 730a4acc00cc95ed4b6a12726d197ae2a9d93e91`, render the loading column with `vertically_centered` and `CrossAxisAlignment::Start` instead of `centered_in_viewport`; remove the now-unused horizontal-centering helper. Update `crates/warp_tui/src/ui_tests.rs:24 @ 730a4acc00cc95ed4b6a12726d197ae2a9d93e91` to assert the loading label starts at column zero while retaining the hint and animation assertions.
*Affected files:* `crates/warp_tui/src/terminal_session_view.rs`, `crates/warp_tui/src/ui.rs`, `crates/warp_tui/src/ui_tests.rs`; add or extend the nearest TUI test module (likely `crates/warp_tui/src/terminal_session_view_tests.rs`) for the timeout-to-failure regression.
*Open questions resolved:* The timeout is 30 seconds, as directed by triage, and is applied to the future at the TUI restore spawn boundary rather than changing the shared history loader. A timeout maps to `None`, so the existing message `The conversation could not be loaded.` remains the single generic restore-failure message; startup transitions to `Failed`, while conversation-list restores return to `Idle` and show their transient hint. The timeout regression test must use a short injectable/test-only duration or equivalent helper seam so it does not sleep for 30 seconds, while production behavior remains fixed at 30 seconds. Existing request IDs and explicit Esc/Ctrl-C cancellation remain authoritative for stale-result suppression and cancellation.
*Risks / blast radius:* A timeout wrapper must preserve the current `SpawnedFutureHandle` ownership and abort behavior, and stale timed-out/completed requests must still be ignored by the request-id guard. Only restore operations should be bounded; do not alter unrelated server API calls or local database restore behavior. The loading column must remain vertically centered, keep its max-width/truncation behavior, and remain within the TUI viewport at narrow widths.
