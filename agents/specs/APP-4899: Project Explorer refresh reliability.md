*Spec: Project Explorer refresh reliability (APP-4899)*

== PRODUCT ==
*Summary:* Project Explorer can remain stale after directories are created, removed, or renamed during a session. This work provides an immediate, discoverable recovery path and makes filesystem synchronization resilient when native watcher delivery is incomplete, including WSL2. The ticket covers both phases below; Phase 1 is the user-visible safety net and Phase 2 improves the default automatic behavior.

*Key design choices:* (1) Treat watcher notifications as hints and reconcile the indexed tree from the filesystem, while retaining incremental updates for normal operation. (2) Expose the same explicit `Refresh Project Explorer` action through the Command Palette and a visible Project Explorer button. (3) Select a polling backend only for roots whose native watcher cannot observe the filesystem (WSL2) or cannot be registered (for example, inotify watch exhaustion), rather than polling every root.

*Behavior* (numbered, testable invariants from the user's view):
1. With Project Explorer open and a local directory displayed, the user can invoke `Refresh Project Explorer` from the Command Palette and from a visible refresh button in the Project Explorer surface. Both entry points perform the same operation and have an accessible label and tooltip.
2. Refresh reconciles every displayed local root, including roots that are git repositories and standalone lazy-loaded directories. Newly created files/directories appear, removed files/directories disappear, and renames do not leave either the old or new path stale. The user does not need to close the panel, switch directories, open a new shell, or restart Warp.
3. While a refresh is running, the existing tree remains usable and the refresh control communicates that work is in progress; after completion it returns to the idle state. A failed root does not erase successfully reconciled sibling roots.
4. Refresh preserves the user's expanded/collapsed folders, selected item when that path still exists, scroll position where possible, hidden-file preference, and the existing local/remote root ordering. If the selected path was removed, selection falls back to the nearest surviving visible ancestor or root header without panicking or retaining a dangling identifier.
5. Refresh is safe when there are no displayed roots, when a displayed root has been deleted or is inaccessible, and when the panel is inactive. It reports an actionable warning/telemetry event and leaves the last valid tree visible rather than crashing.
6. Remote-backed roots continue to be updated by their existing server-push path. The local refresh action does not issue an unsupported local filesystem scan for a remote root, and refreshing a mixed local/remote display refreshes only local roots.
7. After Phase 2, ordinary filesystem operations continue to update Project Explorer without manual intervention on native macOS, Windows, and Linux roots. A create/remove/rename that is coalesced into a parent-only or otherwise incomplete native notification is reconciled by the affected loaded parent/root.
8. On WSL2 roots, or on any root whose native watcher cannot be registered because of a resource/watch-limit failure, Project Explorer uses a polling/snapshot backend in the same filesystem namespace as the root. Changes made inside the guest become visible within the documented polling bound without requiring a restart.
9. Polling and native events are deduplicated: one filesystem change produces at most one visible tree transition, and a transient watcher error does not cause an unbounded rescan loop or duplicate entries.

*Design alternatives* (per decision point with more than one reasonable approach):
- Recovery strategy — (a) full re-index on every native event (simple, but expensive and causes visible churn), (b) trust exact event paths (cheap, but fails when FSEvents/inotify/WSL coalesces or drops notifications), or (c) keep incremental updates and add bounded parent/root reconciliation plus an explicit full refresh. Select (c): it preserves current performance while making missed notifications recoverable.
- User affordance — (a) a button only (discoverable but not keyboard/automation friendly), (b) a Command Palette action only (keyboard friendly but easy to miss), or (c) one shared action exposed by both. Select (c), with one `WorkspaceAction`/`FileTreeAction` implementation to prevent behavior drift.
- Watch backend — (a) native watcher only, (b) polling every root, or (c) adaptive native-first with polling fallback. Select (c): native delivery remains efficient, while WSL2 and watch-limit failures remain usable without imposing polling cost on healthy roots.
- Directory coverage — (a) register one recursive watch everywhere (simple but prone to Linux watch/filter/resource limits), (b) one non-recursive watch per directory (complete but can exhaust inotify), or (c) retain the existing hybrid root/per-expanded-directory model and reconcile parent directories when a notification is incomplete. Select (c), adding explicit registration/teardown for newly created and removed expanded directories.
- WSL2 execution — (a) continue watching from the Windows host (does not see Linux ext4 mutations), (b) disable Project Explorer in WSL (current limitation, no recovery), or (c) poll/read snapshots through the WSL filesystem namespace. Select (c), using the existing WSL session/command boundary where required so the path is observed by the side that owns the filesystem.

*Proposed changes:*

*Phase 1 — explicit recovery*
1. Add a `Refresh Project Explorer` action to the shared action plumbing. Register it as an editable Command Palette entry with the `SHOW_PROJECT_EXPLORER` context predicate and a stable binding name. Route it through the active pane group's `FileTreeView`, so all invocations use the same code path.
2. Add a refresh button to the Project Explorer surface (not to the unrelated “open Project Explorer” toolbelt toggle). Use the existing Warp icon-button/tooltip patterns, preserve mouse-state ownership, and expose the action through accessibility text. Disable or show progress while a refresh is in flight.
3. Add a forced local reconciliation API to `RepoMetadataModel`/`LocalRepoMetadataModel` that accepts a displayed root and schedules a complete filesystem tree build even when the repository is already indexed. It must reuse existing watcher registrations, cancel/coalesce an in-flight refresh for the same root, and emit the existing metadata events only after the replacement state is installed.
4. For lazy roots, refresh the loaded/root-visible portions without eagerly materializing the entire repository; preserve expanded directories and use the existing `load_directory` path for expanded descendants. For complete git repositories, rebuild the complete indexed state using the same ignore/force-included rules as initial indexing.
5. In `FileTreeView`, snapshot UI state before dispatching refresh, refresh all local displayed roots, and restore/remap expansion and selection after each root completes. Keep remote roots untouched and retain the last valid state on per-root errors.
6. Update the current WSL/unsupported-session gating so a WSL2 local session can display Project Explorer through the polling backend; keep truly unsupported remote/terminal modes on their existing disabled state.

*Phase 2 — reliable automatic delivery*
1. Treat native watcher events as reconciliation hints. Normalize create/remove/rename/modify batches into repository-scoped changes, and when an event identifies a directory or a platform may coalesce events, rescan the nearest loaded parent/root rather than assuming every child path was delivered.
2. Make recursive-watch coverage explicit for newly created directories that are expanded or otherwise required by the tree. Register them exactly once, unregister deleted/moved subtrees, and handle registration failures as a backend state transition rather than silently leaving stale bookkeeping.
3. Add an adaptive polling/snapshot backend to the watcher abstraction. Select it for WSL2 roots and fall back to it when native registration reports an inotify/resource/watch-limit failure. Poll only the affected roots, diff directory/file metadata into `BulkFilesystemWatcherEvent`-compatible mutations, debounce/coalesce repeated snapshots, and stop polling when the root is removed or becomes healthy under a new registration.
4. Ensure WSL2 polling executes against the WSL filesystem namespace (not a Windows host-only watcher). Reuse the existing session/command boundary or an equivalent platform abstraction; do not parse shell output in the UI layer.
5. Add structured logging/telemetry for backend selection, native registration failure, polling fallback, refresh completion/failure, and dropped/coalesced event batches. Do not include file contents or sensitive paths beyond existing safe path logging conventions.
6. Keep event application serialized per repository. A manual refresh, native event, and polling diff for the same root must coalesce or queue deterministically so an older async result cannot overwrite a newer tree.

*Open questions resolved:* The ticket's “phased” proposal is implemented as one contract with Phase 1 as the immediate user-visible recovery and Phase 2 as the automatic-delivery hardening; no separate user approval is required between phases. Scope is local roots displayed by the active Project Explorer; remote roots remain server-push driven. Polling is adaptive and bounded to affected roots, not a global setting. Existing ignore rules, hidden-file preference, lazy-loading limits, and per-pane-group view ownership remain authoritative.

*Risks / blast radius:* Full reconciliation can be CPU/IO-heavy for large repositories; mitigate with per-root serialization, cancellation/coalescing, existing lazy limits, and progress UI. Replacing indexed state can invalidate selection/index identifiers; mitigate by path-based state remapping and ancestor fallback. Polling can increase CPU and wakeups; mitigate with native-first selection, bounded intervals, and teardown on root removal. WSL command execution can fail or hang; use cancellation/timeouts and retain the last valid tree. Watch-limit and registration errors must not crash the app; expose fallback state and telemetry. Remote metadata and unrelated panels must not be rebuilt by local refresh events.
