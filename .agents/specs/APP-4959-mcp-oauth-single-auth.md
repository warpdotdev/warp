*Spec: Serialize MCP OAuth authentication across Warp instances*

## PRODUCT

**Summary:** When the same MCP installation is started by multiple Warp processes, one process must own the interactive OAuth attempt and open one authorization page. Every other process must wait for the result from shared secure storage, then continue using the credentials without opening another page. The behavior applies to TUI loopback callbacks and desktop custom-URL-scheme callbacks on every native platform.

**Key design choices:** Use a per-channel, per-installation OS file lease held with `fs4` for crash-safe leader election; do not fall back to uncoordinated browser launches when the lease cannot be created. Publish credentials through a globally serialized read/merge/write path so the existing secure-storage map remains backward compatible. Route desktop callbacks to the lease owner over the existing cross-platform `crates/ipc` transport while retaining the current loopback peer/path/CSRF validation.

**Behavior invariants:**

1. For one `(Warp channel, MCP installation UUID)` with no usable credentials, concurrent native Warp processes have at most one active OAuth leader. The leader is the only process that prepares the interactive callback receiver, emits `AuthenticationRequired`, stores an authorization URL, and calls `ctx.open_url`; a successful happy-path authentication therefore opens exactly one browser page regardless of the number of running instances.
2. A non-leader never opens a browser, calls `requires_authentication`, binds a TUI loopback port, or creates a desktop CSRF mapping. It enters a visible waiting state, reloads shared credentials, and starts the MCP server as soon as valid credentials are published.
3. The leader rechecks shared credentials after acquiring the lease and before opening the page. Credentials written by an already-running process win over a stale in-memory snapshot, so a follower cannot unnecessarily start a second flow.
4. After a successful token exchange, every waiting process can load the newly persisted credentials and use the MCP server without restarting Warp. The persisted refresh token and client secret are retained according to the current carry-forward behavior.
5. If the leader returns an OAuth error, token-exchange error, callback timeout, or is killed, its lease is released. Waiting processes race to acquire the lease; exactly one successor may become leader and open a replacement page. Other processes continue waiting. A replacement page is allowed only after the prior owner has failed or timed out; concurrent pages are never allowed.
6. TUI and desktop processes coordinate on the same lease key. A TUI leader uses its dynamically allocated `127.0.0.1` redirect and the existing loopback receiver; a desktop leader uses the channel custom scheme. If a desktop callback is delivered to a non-owner process, that process forwards it to the current owner and does not consume it locally.
7. Loopback callbacks continue to accept only the existing callback path, loopback peer, bounded request size, and matching CSRF state. Custom-scheme callbacks are accepted only by the current lease owner after the same CSRF-state check; an unknown, stale, duplicated, or malformed callback cannot supply credentials to a follower.
8. Waiting state is explicit in the TUI snapshot (`WaitingForAuthentication` or an equivalently named status). It shows that another Warp instance is authenticating, has no `authorization_url`, and cannot perform `ReopenAuthorization`. The state changes to `Authenticating` only when that process wins promotion, and to `Running` after credentials are loaded.
9. Credential updates for different installations and auto-refresh writes cannot overwrite one another. A failed serialization, secure-storage read, or write leaves the previous valid value intact and reports a controlled MCP authentication error rather than corrupting the shared map.
10. Headless/autonomous MCP startup remains non-interactive: it does not acquire an interactive lease or wait indefinitely, and it returns the current “authenticate in the Warp desktop app first” error when no usable credentials exist. A later headless retry can consume credentials written by an interactive leader.
11. The lease is scoped to the Warp channel and current user, so preview/development/release channels and different OS users cannot elect one another’s leaders or consume one another’s credentials.

The defect is the boundary mismatch: the manager singleton and OAuth callback channels stop at a process boundary, while the user-visible authorization and the secure credential map are shared by all instances. A change limited to `crates/warp_tui` would leave desktop startup and the desktop URI callback path unsynchronized.

### Design alternatives

- **Leader primitive — selected: an OS file lease via `fs4`.** Create/open an owner-only lock file in the current user’s channel-scoped runtime directory and hold an exclusive OS lock for the entire interactive flow. `try_lock_exclusive` elects one leader without polling a PID file; RAII releases it on normal completion and the OS releases it on crash on Unix and Windows. A stale empty lock file is reusable. A PID/JSON lock was rejected because stale-owner expiry can overlap a slow legitimate flow; a Windows-only named mutex was rejected because TUI and Unix/macOS need the same semantics; a long-lived global broker was rejected because it adds a daemon lifecycle and migration surface.
- **Coordination location — selected: an OAuth coordinator in `crates/mcp`, integrated by the native manager.** The coordinator owns the lease, wait/promotion state, and callback-owner metadata while `app/src/ai/mcp/templatable_manager/native.rs` remains responsible for WarpUI events and server state. Keeping election next to `make_authenticated_client` covers all native callers and avoids duplicating the rule in TUI and desktop code. `crates/local_control` and the single-instance manager are patterns for permissions and IPC, not new MCP-specific dependencies.
- **Desktop callback delivery — selected: lease-owner IPC relay.** The leader starts a short-lived `crates/ipc` service at an instance-derived, channel-scoped address and writes only the endpoint/instance nonce to an owner-only metadata record. `UriHost::Mcp` forwards a callback received by any process to that endpoint; the leader alone validates the CSRF state and sends the result to its local OAuth channel. A shared callback file was rejected because it would persist authorization codes and require another race-prone polling protocol. Treating every process as a callback consumer was rejected because a custom scheme may be delivered to a follower that has no matching CSRF state.
- **Credential persistence — selected: serialize the existing map read/merge/write.** Add a second global secure-storage lease for the credential namespace. Every load and write takes a shared/exclusive lock as appropriate; the writer reloads the latest map, merges only the installation key, serializes it, and writes through `write_value_with_owner_only_fallback`. This preserves existing keys and migration behavior. Per-installation keys would avoid map contention but require a storage migration and would break current map readers; allowing independent whole-map writes is the current data-loss bug.
- **Waiting UX — selected: an explicit manager/TUI state with no URL.** The follower’s state is observable and testable while retaining the existing `AuthenticationRequired` event for the leader. Automatically opening the URL from every follower was rejected; silently showing `Authenticating` was rejected because it implies that `ReopenAuthorization` is safe even though the follower has no callback owner.

### Proposed changes

1. **Add a cross-process coordinator to the native OAuth path.**
   - Introduce a coordinator keyed by `(ChannelState::channel(), installation UUID, credential namespace)` and a small `AuthLease` RAII type. Use a private per-user runtime directory with mode `0700`; lock files and owner metadata are `0600`, contain no access/refresh tokens or authorization codes, and are derived from the channel and UUID rather than accepting arbitrary paths.
   - Acquire the lease only after the initial credential read says that no valid access/refresh token is available and only for non-headless flows. Re-read the credential map while holding the lease; if another process populated it, release the lease and continue without interaction.
   - Make acquisition non-blocking to the UI executor. A follower retries on a bounded interval while loading the shared credentials. The wait deadline is the existing five-minute OAuth callback deadline plus a small handoff grace period; expiry is a controlled “authentication in another instance did not complete” error, not a browser launch by every waiter.
   - Do not call `OAuthCallbackMode::prepare`, discover metadata, register a client, or create a CSRF state until the process is the leader. This ensures a follower does not reserve a loopback port or create a browser-visible authorization URL.
   - Release the lease in every success, OAuth error, callback timeout, token-exchange error, cancellation, and server-shutdown path. A promotion attempt must acquire the OS lease before setting leader state, so simultaneous followers cannot both open a replacement page.

2. **Preserve and harden both callback modes.**
   - **TUI loopback:** after leadership, call the existing `LoopbackOAuthReceiver::bind()`, use its dynamically allocated redirect URI in the authorization URL, and retain the five-minute timeout and all peer/path/size/CSRF checks. The receiver is dropped with the lease on every terminal path.
   - **Desktop custom scheme:** after leadership, create the existing `oauth_result_tx` and CSRF mapping, publish a short-lived owner endpoint for callback relay, and then call `ctx.open_url`. Update `UriHost::Mcp`/`handle_oauth_callback` so a process with no matching local CSRF mapping forwards the complete validated-scheme URL to the current owner endpoint. The owner validates the state and delivers exactly one `CallbackResult` to its local channel. If the owner endpoint is absent or unreachable, return a controlled callback error; do not open another URL or accept the callback in the follower.
   - The owner endpoint uses the existing `crates/ipc` cross-platform transport (`ServerBuilder::with_fixed_address`), an instance-derived address, and an owner-only runtime directory. The metadata is atomically published and removed on release; a callback relay includes a protocol version, installation UUID, owner nonce, and URL, but never logs or persists the authorization code. The owner compares the URL’s scheme/host/path and CSRF state before forwarding it to OAuth.
   - A callback arriving after owner failure is rejected or forwarded to the new owner only if its state matches the new owner’s current flow. Old CSRF mappings are removed during shutdown/promotion.

3. **Serialize secure-storage access without changing the stored schema.**
   - Add a credential-store lock per existing secure-storage key (templatable and file-based namespaces may use distinct lock files). Reads used for startup and follower polling take the lock; writes take it exclusively.
   - Replace the current “write the manager’s entire in-memory map” behavior with `read latest → deserialize (or treat missing as empty) → merge one UUID/hash → serialize → secure write`. Use `write_value_with_owner_only_fallback`; reject malformed existing JSON rather than replacing it with an empty map, and leave the prior value untouched when serialization or the backend write fails.
   - Keep refresh-token carry-forward and client-secret persistence intact. Emit `CredentialsChanged` only after the shared write succeeds; followers then reload and seed their own `AuthorizationManager`.
   - Add a process-safe test seam for the secure store and lock directory so tests can run against temporary owner-only directories without touching a developer’s keychain.

4. **Expose leader/waiter state through the manager and TUI.**
   - Add a process-local authentication role/state for each installation (leader, waiting, or idle) and clear it on success, failure, shutdown, and logout. The manager must not put a follower URL into `authorization_urls`.
   - Extend `TuiMcpServerStatus` and snapshot mapping with `WaitingForAuthentication` (or the final equivalent). The waiting snapshot includes `has_credentials = false`, `authorization_url = None`, and a stable message such as “Waiting for authentication in another Warp instance…”. `ReopenAuthorization` is a no-op with a user-visible explanation while waiting.
   - Keep `Authenticating` and `authorization_url` for the elected leader. Emit manager events on role transitions, shared credential publication, promotion, and coordination errors so the TUI refreshes without polling the UI model directly.
   - Desktop notification/toast copy must distinguish “authentication in another Warp instance” from a local browser launch. A successful shared credential load may use the existing authenticated notification only once per process.

5. **Scope, compatibility, and failure semantics.**
   - Scope lock and endpoint names with the current channel and user runtime directory; reject unsafe ownership/permissions and refuse to launch interactive auth if the coordination directory or endpoint cannot be initialized. The failure must be actionable and must not revert to N independent pages.
   - Existing secure-storage maps remain readable. Existing valid credentials bypass coordination exactly as today. Logout removes only the selected installation under the credential-store lock and wakes any waiter, which then observes missing credentials and may elect a new leader.
   - Preserve the existing headless error path, server lifecycle/abort behavior, telemetry, and callback security. Add safe telemetry/log fields for `leader`, `waiter`, `promotion`, `timeout`, and `coordination_error`; never include auth URLs, codes, tokens, or client secrets.
   - Use explicit exhaustive matches for the new role/status variants and keep WASM builds unchanged through the existing native cfg boundary.

### Open questions resolved

- **Does “exactly one” include recovery after a failure?** The invariant is one active page at a time. A failed or timed-out leader may be replaced by exactly one serialized successor, so a human can see a second page only when the first flow demonstrably failed.
- **Should the follower bind a loopback receiver in advance?** No. Binding and callback preparation happen only after lease acquisition; this avoids unnecessary ports and makes the leader boundary testable.
- **Can a desktop callback be delivered to a different process?** Yes; custom URL handlers are process/platform dependent. The receiving process forwards to the current owner over IPC, and only the owner validates/consumes the state.
- **What is the lock timeout?** Use the current five-minute callback timeout as the interactive attempt deadline and a bounded handoff grace interval for promotion. Tests inject shorter durations; production must not create a second page while the current owner is still alive and within its deadline.
- **How are simultaneous credential refreshes handled?** Refresh persistence uses the same credential-store lock and merge path as initial OAuth. The in-memory `PersistingCredentialStore` remains process-local, but every secure write is serialized and merged with the latest map.
- **What happens if coordination is unavailable?** Fail closed for interactive OAuth with a clear retryable error. It is safer to ask the user to retry than to recreate the reported N-page behavior.
- **Are non-MCP OAuth flows affected?** No. The coordinator is only constructed for the MCP `AuthContext`; existing GitHub/account auth and the WASM implementation are out of scope.
- **Is a new daemon required?** No. The lease and callback relay live only for the interactive flow and are released with the owning process/task.

### Risks and mitigations

- **Lease implementation differs across platforms:** use `fs4`’s native locking API, keep the file descriptor alive in `AuthLease`, and run subprocess contention/crash tests on Linux, macOS, and Windows CI.
- **A follower sees stale in-memory credentials:** reload under the credential-store lock before opening and after every wait interval; never trust the manager cache as the cross-process source of truth.
- **Whole-map writes lose another installation:** serialize read/merge/write and test two UUIDs plus simultaneous refresh updates. Reject malformed values and preserve the last good value.
- **A callback reaches the wrong process or a stale owner:** use owner-derived IPC endpoints, owner nonce/installation matching, and final CSRF validation at the leader; remove metadata on release and reject unreachable/stale endpoints.
- **UI claims local authentication while waiting:** add snapshot fixtures and a running TUI check that assert the exact waiting state, absent URL, and promotion transition.
- **A lock or endpoint path leaks secrets:** derive names from channel/UUID/instance ID, use owner-only permissions, and add tests that inspect lock/metadata contents for tokens, codes, and client secrets.

## Validation & verification criteria (all must pass before merge)

1. **Environment/reproduction record:** Preserve the documented environment-mismatch reproduction note from APP-4959 (the available sandbox cannot launch multiple authenticated Warp TUI instances), and add a deterministic local harness for the same sequence: start N native MCP auth tasks for one installation with no credentials and instrument `requires_authentication`/`ctx.open_url`. The pre-change harness must show N launches; the implementation must show one active launch.
2. **Regression test:** Add a named test such as `only_one_process_opens_mcp_oauth_authorization` that starts at least three concurrent coordinator clients against one temporary channel/installation key, asserts exactly one leader and one `AuthenticationRequired`/open-url event, and fails against the current implementation because every task invokes the callback.
3. **Follower preparation/UX test:** Add a test such as `mcp_oauth_followers_wait_without_preparing_callbacks` asserting that non-leaders do not call `OAuthCallbackMode::prepare`, bind a loopback listener, create a CSRF mapping, persist an authorization URL, or call `ctx.open_url`; assert their exposed state is `WaitingForAuthentication` with no reopenable URL.
4. **Shared credential completion test:** Have the elected leader complete a fake OAuth exchange and persist credentials, then assert every waiting process reloads the same access token, refresh token, and client secret, reaches `Running`, and emits `CredentialsChanged` only after the shared write succeeds.
5. **Exactly-one stress test:** Repeat the happy path with N=2, 3, 5, and 10 processes/tasks and randomized start barriers. Assert one page per attempt, one leader, N usable clients, and no follower URL. Run the stress fixture repeatedly enough to exercise simultaneous lock acquisition rather than relying on a single scheduling order.
6. **Failure/promotion test:** Inject leader failure before callback, token-exchange failure, and a shortened callback timeout. Assert the first lease is released, exactly one follower becomes leader, exactly one replacement `AuthenticationRequired`/open-url event is emitted, all other followers remain waiters, and the eventual credentials unblock every process.
7. **Crash/stale-lock test:** Use subprocess helpers to kill the leader while holding the OS lease. Assert the successor can acquire the same lock on Unix and Windows, a leftover empty lock file is reusable, and no stale PID/metadata can cause two leaders. A live leader must retain exclusivity past the test timeout.
8. **Credential merge/concurrency test:** Run concurrent writes for different installation UUIDs and concurrent auto-refresh updates through the fake secure store. After all writes, deserialize the stored map and assert every entry and refresh-token carry-forward value remains present; inject a backend/serialization failure and assert the last valid value remains readable.
9. **Storage permission/content test:** On native platforms, assert the coordinator directory is owner-only, lock/owner metadata is non-world-readable, and no file contains an authorization URL, OAuth code, access token, refresh token, or client secret. Assert unsafe/symlinked/unwritable coordination paths fail closed without opening a page.
10. **TUI loopback path test:** Exercise the leader with a fake provider and a real `127.0.0.1` receiver. Assert the authorization URL contains only the leader’s loopback port, a valid callback succeeds, and invalid peer/path/oversized/missing-state/mismatched-state callbacks fail exactly as the existing `crates/mcp/src/oauth/loopback.rs` tests require. Assert followers never bind a port.
11. **Desktop custom-scheme path test:** Deliver a valid `warp` MCP callback to a follower process and assert it forwards to the current owner, which consumes it once and completes authentication. Deliver a wrong-state, stale-owner, duplicate, malformed-host, and malformed-path callback and assert no process receives credentials. Cover owner endpoint startup failure and callback relay timeout.
12. **Cross-mode promotion test:** Start a TUI follower and a desktop follower around a leader, fail the leader, and assert exactly one promoted process opens the replacement page using its own callback mode. The other mode remains waiting and consumes the resulting shared credentials.
13. **Headless/legacy compatibility tests:** Assert headless startup with missing credentials returns the existing non-interactive error without acquiring a lease or opening a page; assert valid pre-existing credential maps bypass coordination; assert logout removes only one installation while preserving other map entries.
14. **TUI snapshot/render test:** Add render-to-lines/model fixtures for leader, waiting, promotion, success, timeout, and coordination-error states. `WaitingForAuthentication` must have the exact stable copy, no authorization URL, and a disabled/no-op `ReopenAuthorization` action. The fixture must prove two process snapshots converge to `Running` after one credential publication.
15. **Running UI verification:** After the implementation builds, run two real TUI instances against the deterministic fake OAuth provider and use the repository’s TUI verification/computer-use workflow to capture the leader’s single authentication state, the follower’s waiting state, and the post-authentication running state.
