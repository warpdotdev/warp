*Proposed change: Add Terraform and GCP tooling to the agent-dev image*

*Summary:* Extend `docker/agent-dev/Dockerfile` so the development image can use SSH/network diagnostics, the Google Cloud CLI and GKE auth plugin, and Terraform managed by tfenv.
*Key design choices:* Mirror the already-merged [`warp-agent-docker` PR #173](https://github.com/warpdotdev/warp-agent-docker/pull/173), keep the existing Ubuntu apt-install layer and signed apt-keyring pattern, and pin tfenv to `v3.2.2` plus Terraform `1.15.5` for reproducible image contents.
*Design alternatives:* No meaningful alternatives for this tightly scoped image enhancement; installing Terraform directly would bypass the requested tfenv workflow, while an unpinned tfenv checkout would reduce reproducibility.
*Root cause / approach:* The current image at `docker/agent-dev/Dockerfile` (`@ 726c1b64ba2ee7e0944d30527b3ce68762bdb0e9`) lacks the command-line dependencies needed by Terraform/GKE workflows. Add `openssh-client`, `netcat-openbsd`, and `lsof` to the existing apt block; add a Google Cloud SDK repository using a dearmored key in `/usr/share/keyrings`, then install `google-cloud-cli` and `google-cloud-cli-gke-gcloud-auth-plugin`; expose `TFENV_ROOT=/usr/local/tfenv` and its `bin` directory on `PATH`; clone tfenv at tag `v3.2.2`, install and select Terraform `1.15.5`, and verify it during the build. Existing `gnupg2` and `unzip` packages satisfy the repository-key and tfenv archive prerequisites, so do not add duplicate packages.
*Affected files:* `docker/agent-dev/Dockerfile`
*Open questions resolved:* The requested Terraform version is `1.15.5`, tfenv is pinned to `v3.2.2`, and the image-wide install location is `/usr/local/tfenv`, matching the ticket and reference PR. This is a Dockerfile-only infrastructure change; no application code or runtime API behavior is in scope.
*Risks / blast radius:* Image builds depend on the Google apt repository, GitHub, and Terraform release availability, and add image size plus external package supply-chain surface. Mitigate with the signed Google keyring, pinned tfenv/Terraform versions, apt-list cleanup, and build-time version checks. The existing x86_64-only image constraint remains unchanged.

*Validation & verification criteria* (must ALL pass before merge):
1. The apt-install block contains `openssh-client`, `netcat-openbsd`, and `lsof` exactly once, while retaining the existing `gnupg2` and `unzip` prerequisites; inspect `docker/agent-dev/Dockerfile`.
2. The Dockerfile adds the Google Cloud SDK apt source with a dearmored keyring and `signed-by=/usr/share/keyrings/cloud.google.gpg`, installs both `google-cloud-cli` and `google-cloud-cli-gke-gcloud-auth-plugin`, removes `/var/lib/apt/lists/*` after installation, and successfully runs `gke-gcloud-auth-plugin --version` during the build.
3. The image declares `TFENV_ROOT=/usr/local/tfenv` and includes `/usr/local/tfenv/bin` on `PATH`; the tfenv checkout is pinned to tag `v3.2.2`, and the build installs/selects Terraform `1.15.5` before running `terraform version`.
4. Run `docker build -f docker/agent-dev/Dockerfile -t warp-agent-dev:app-4957 .` from the repository root; the build completes successfully without missing packages, repository-key errors, or failed version checks.
5. Run a container from the built image and verify `ssh`, `nc`, `lsof`, `gcloud`, `gke-gcloud-auth-plugin`, `tfenv`, and `terraform` resolve on `PATH`; assert `terraform version` reports `1.15.5` and the Google tools return version output.
6. Because this is a testing-exempt configuration/infrastructure change with no testable application logic, no new Rust or application regression test is required; the Docker build, shell parsing of the changed `RUN` blocks, and runtime command/version checks above are the regression coverage.
