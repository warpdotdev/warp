# Spec: Add shell commands to the up-arrow history menu (CODE-1906)

== PRODUCT ==
*Summary:* The headless TUI's up-arrow history menu (`crates/warp_tui/src/prompt_history_menu.rs`) â€” the menu that opens when the user presses Up with the caret on the first input row â€” today lists only previously-submitted agent prompts. Extend it so it also surfaces the user's executed shell commands, so Up recalls both prompts and commands from one menu. This fulfills the shell-command scope that CODE-1871 (which built the TUI prompt-history menu) explicitly deferred (see `specs/CODE-1871/PRODUCT.md` Non-goals: "Command (shell) history â€¦ out of scope for this ticket. â€¦ Shell command history is out of scope for this ticket.").

*Scope:* This change is **strictly TUI-only**. No GUI menu is modified â€” not the standard GUI `InlineHistoryMenuView` (which already shows commands via tabs) and not the cloud-mode-v2 GUI history menu (`app/src/terminal/input/cloud_mode_v2_history_menu.rs`, prompts-only behind the `CloudModeInputV2` dogfood flag). Both GUI surfaces remain untouched.

*Key design choices:*
1. Extend the existing TUI prompt-history menu into a combined history menu (prompts + commands) rather than adding a second menu â€” one Up menu, reusing CODE-1871's lifecycle/preview/restore machinery.
2. Source rows from the shared `up_arrow_suggestions_for_terminal_view` combiner (already used by the GUI) so ordering, de-duplication, session-scoping, the `include_agent_commands_in_history` setting, and ignored-suggestions all behave identically across the TUI and GUI.
3. Open the menu on Up in both shell (`!`) and agent modes; the menu's content config selects kinds: shell mode â†’ commands only; agent mode â†’ prompts + commands.
4. Accepting a command fills the input and executes it as a shell command (mirrors the GUI `AcceptCommand` â†’ fill + `input_enter`); accepting a prompt keeps today's fill + submit-as-prompt behavior.
5. *Shared settings:* every setting this feature depends on â€” `AISettings::include_agent_commands_in_history` (whether agent-executed commands appear), the ignored-suggestions model (`IgnoredSuggestionsModel`, `SuggestionType::ShellCommand`/`AIQuery`), and the AI/Agent-Mode-enabled gates for prompts â€” is read from the **same shared singletons** the GUI reads, because both surfaces source rows through the one shared `up_arrow_suggestions_for_terminal_view` combiner. The TUI introduces **no** TUI-only duplicate of these settings; toggling any of them affects the GUI and TUI identically.

*Behavior* (numbered, testable invariants from the user's view):
1. With the prompt input focused, no other inline menu open, and a single caret on the first visual row, pressing Up (`â†‘` or `Ctrl+P`) opens the history menu â€” in both agent (AI) input mode and `!` shell input mode. (CODE-1871 invariant 25 â€” "shell mode does not open the menu" â€” is superseded by this ticket.)
2. In agent (non-`!`) mode the menu lists both prior agent prompts and prior shell commands, interleaved in a single list by the shared up-arrow ordering: items from other sessions precede items from the current session, and within each group items are ordered oldest-first so the most-recent current-session item sits at the bottom, closest to the input.
3. In `!` shell mode the menu lists only shell commands (no prompts), using the same ordering.
4. Commands and prompts are each de-duplicated by text, keeping the most recent occurrence of each (a command and a prompt that happen to share text both appear, once each).
5. Whitespace-only and empty commands and prompts are never shown.
6. Shell commands the user has marked as ignored (ignored-suggestions, `SuggestionType::ShellCommand`) are not shown; agent-executed commands are excluded when the "include agent-executed commands in history" setting (`AISettings::include_agent_commands_in_history`) is off. Prompts are excluded when AI/Agent Mode is disabled (existing gate in `up_arrow_suggestions_for_terminal_view`). These are the **same shared settings** the GUI honors (see Key design choices #5); the TUI does not maintain separate copies, so changing a setting in Settings affects both surfaces identically.
7. Whatever text is in the input when the menu opens becomes the initial prefix filter and is preserved as the "original buffer" for restoration. Opening with an empty input lists everything (subject to the above).
8. Typing filters the list by prefix: an item is shown only when one of its lines starts with the trimmed query (commands are single-line; multi-line prompts match on any line, as today). Clearing the query shows the full de-duplicated list again.
9. On open the selection defaults to the item nearest the input (the most-recent current-session row) and that item's text is previewed into the input. Previewing a command sets the input to shell (`!`) mode so the shell indicator renders while cycling commands; previewing a prompt sets AI mode. Up/Down move the selection toward older/newer items and re-preview.
10. Pressing Down past the last (newest) item, or Down on an empty list, closes the menu and restores the original buffer and the original input type (agent vs shell) the user had before opening.
11. Pressing Escape closes the menu and restores the original buffer and original input type, discarding any preview text. One Escape closes only the menu.
12. Pressing Enter with a prompt highlighted fills the input with that prompt's text, closes the menu, and submits it as an agent prompt immediately (unchanged from today).
13. Pressing Enter with a command highlighted fills the input with that command's text, closes the menu, and executes it as a shell command in the active session (the equivalent of typing the command at a `!` prompt and pressing Enter).
14. With an empty or filtered-to-nothing list, Enter does not accept a history item; it behaves as a normal submit of whatever is in the input (so submitting empty is a no-op, as today).
15. Any event that would normally dismiss inline menus (accepting another action, starting a conversation, losing input focus, a running process taking over the input) closes the history menu and restores the original buffer and input type rather than leaving preview text stranded.
16. Empty history: opening in agent mode with no prompts and no commands shows an explicit empty state "No history"; in shell mode with no commands, "No history". A query that matches nothing shows "No matching history".
17. While a process owns the input (alt-screen app or an inline long-running command under user control), Up is forwarded to the process and the menu does not open (unchanged).
18. History updates while the menu is open (a background conversation records a new prompt, or a new command lands) refresh the list in place without losing the current selection where the selected item still exists.
19. The menu header reads "History" (was "Prompt history") â€” this is UI/display copy only; code identifiers use `PromptAndCommandHistory` (see Tech step 6). Command rows and prompt rows are visually distinguishable: command rows are prefixed with the same `!` marker the TUI transcript uses for shell-command rows, rendered in the transcript's shell-command prefix color (bold bright-green â€” `TuiUiBuilder::shell_command_prefix_style()`'s foreground, `terminal_colors().bright.green` + `Modifier::BOLD`), and **without** the pale-green tinted row background the transcript applies (`shell_command_background()` is NOT used on menu rows). Prompt rows have no `!` prefix. The `!` reuses the existing transcript prefix style's foreground so the marker matches what users already see in the transcript.
