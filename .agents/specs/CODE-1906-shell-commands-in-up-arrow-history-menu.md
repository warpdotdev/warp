# Spec: Add shell commands to the up-arrow history menu (CODE-1906)

== PRODUCT ==
*Summary:* The headless TUI's up-arrow history menu (`crates/warp_tui/src/prompt_history_menu.rs`) — the menu that opens when the user presses Up with the caret on the first input row — today lists only previously-submitted agent prompts. Extend it so it also surfaces the user's executed shell commands, so Up recalls both prompts and commands from one menu. This fulfills the shell-command scope that CODE-1871 (which built the TUI prompt-history menu) explicitly deferred (see `specs/CODE-1871/PRODUCT.md` Non-goals: "Command (shell) history … out of scope for this ticket. … Shell command history is out of scope for this ticket.").

*Key design choices:*
1. Extend the existing TUI prompt-history menu into a combined "History" menu (prompts + commands) rather than adding a second menu — one Up menu, reusing CODE-1871's lifecycle/preview/restore machinery.
2. Source rows from the shared `up_arrow_suggestions_for_terminal_view` combiner (already used by the GUI) so ordering, de-duplication, session-scoping, the `include_agent_commands_in_history` setting, and ignored-suggestions all behave identically across the TUI and GUI.
3. Open the menu on Up in both shell (`!`) and agent modes; the menu's content config selects kinds: shell mode → commands only; agent mode → prompts + commands.
4. Accepting a command fills the input and executes it as a shell command (mirrors the GUI `AcceptCommand` → fill + `input_enter`); accepting a prompt keeps today's fill + submit-as-prompt behavior.

*Behavior* (numbered, testable invariants from the user's view):
1. With the prompt input focused, no other inline menu open, and a single caret on the first visual row, pressing Up (`↑` or `Ctrl+P`) opens the history menu — in both agent (AI) input mode and `!` shell input mode. (CODE-1871 invariant 25 — "shell mode does not open the menu" — is superseded by this ticket.)
2. In agent (non-`!`) mode the menu lists both prior agent prompts and prior shell commands, interleaved by the shared up-arrow ordering: items from other sessions precede items from the current session, and within each group items are ordered oldest-first so the most-recent current-session item sits at the bottom, closest to the input.
3. In `!` shell mode the menu lists only shell commands (no prompts), using the same ordering.
4. Commands and prompts are each de-duplicated by text, keeping the most recent occurrence of each (a command and a prompt that happen to share text both appear, once each).
5. Whitespace-only and empty commands and prompts are never shown.
6. Shell commands the user has marked as ignored (ignored-suggestions, `SuggestionType::ShellCommand`) are not shown; agent-executed commands are excluded when the "include agent-executed commands in history" setting (`AISettings::include_agent_commands_in_history`) is off. Prompts are excluded when AI/Agent Mode is disabled (existing gate in `up_arrow_suggestions_for_terminal_view`).
7. Whatever text is in the input when the menu opens becomes the initial prefix filter and is preserved as the "original buffer" for restoration. Opening with an empty input lists everything (subject to the above).
8. Typing filters the list by prefix: an item is shown only when one of its lines starts with the trimmed query (commands are single-line; multi-line prompts match on any line, as today). Clearing the query shows the full de-duplicated list again.
9. On open the selection defaults to the item nearest the input (the most-recent current-session row) and that item's text is previewed into the input. Previewing a command sets the input to shell (`!`) mode so the shell indicator renders while cycling commands; previewing a prompt sets AI mode. Up/Down move the selection toward older/newer items and re-preview.
10. Pressing Down past the last (newest) item, or Down on an empty list, closes the menu and restores the original buffer and the original input type (agent vs shell) the user had before opening.
11. Pressing Escape closes the menu and restores the original buffer and original input type, discarding any preview text. One Escape closes only the menu.
12. Pressing Enter with a prompt highlighted fills the input with that prompt's text, closes the menu, and submits it as an agent prompt immediately (unchanged from today).
13. Pressing Enter with a command highlighted fills the input with that command's text, closes the menu, and executes it as a shell command in the active session (the equivalent of typing the command at a `!` prompt and pressing Enter).
14. With an empty or filtered-to-nothing list, Enter does not accept a history item; it behaves as a normal submit of whatever is in the input (so submitting empty is a no-op, as today).
15. Any event that would normally dismiss inline menus (accepting another action, starting a conversation, losing input focus, a running process taking over the input) closes the history menu and restores the original buffer and input type rather than leaving preview text stranded.
16. Empty history: opening in agent mode with no prompts and no commands shows an explicit empty state "No history"; in shell mode with no commands, "No history". A query that matches nothing shows "No matching history".
17. While a process owns the input (alt-screen app or an inline long-running command under user control), Up is forwarded to the process and the menu does not open (unchanged).
18. History updates while the menu is open (a background conversation records a new prompt, or a new command lands) refresh the list in place without losing the current selection where the selected item still exists.
19. The menu header reads "History" (was "Prompt history"). Command rows and prompt rows are visually distinguishable in the rendered menu (e.g. a `$` affordance/state-suffix on command rows) so the user can tell at a glance which kind a row is.

*Design alternatives* (per decision point with more than one reasonable approach):
- *Where the combined list is built* — (a) Add a new owned, TUI-friendly item type + a combined getter `up_arrow_history_for_terminal_view(terminal_view_id, session_id, config, app) -> Vec<TuiHistoryItem>` that wraps `up_arrow_suggestions_for_terminal_view` and maps to owned `{ text, kind, linked_workflow_data }`; export it via `tui_export.rs`. **Selected:** clean ownership (no `&HistoryEntry` lifetime in the TUI), reuses the shared sort/dedup/setting/ignored logic verbatim. (b) Export `up_arrow_suggestions_for_terminal_view` + `HistoryInputSuggestion` directly. Rejected: the borrowed `Command` variant is awkward to store in the menu model and would leak `pub(crate)` types to the TUI crate. (c) Re-implement command fetching inside the TUI. Rejected: diverges ordering/dedup from the GUI and duplicates logic.
- *Menu surface* — (a) Extend the existing TUI prompt-history menu into the combined history menu. **Selected:** reuses CODE-1871's open/preview/restore/filter infra; one Up menu. (b) A separate command-history menu. Rejected: two menus keyed on the same Up gesture is confusing and doubles lifecycle code.
- *Accept semantics for commands* — (a) Fill + execute as a shell command. **Selected:** mirrors GUI `AcceptCommand` and matches "recall a command → run it". (b) Fill only and let the user press Enter. Rejected: inconsistent with the prompt's accept-submits behavior and with the GUI.
- *Opening in shell mode* — (a) Open in both shell and agent modes; config selects kinds. **Selected:** makes shell commands recallable from `!` mode (the natural place) and fulfills the request. (b) Agent mode only. Rejected: commands are most useful in shell mode; agent-mode-only would half-deliver.
- *Content config derivation* — TUI-specific: `include_commands = true; include_prompts = !is_shell_mode`. **Selected:** agent mode shows both; shell mode shows commands only. Rejected: reusing `UpArrowHistoryConfig::for_input_config` directly — it returns prompts-only when the input is locked to AI, which would NOT add commands in agent mode and would regress the goal.

*Risks / blast radius:*
- The TUI Up gesture is currently the sole entry point for the prompt-history menu; widening it to shell mode changes Up behavior in `!` mode (today: cursor move / no-op on the first row). Mitigation: still gated on `single_cursor_on_first_row` and no other menu open; shell-mode users not on the first row see no change.
- Accept-routing change touches the shared `TuiInlineMenuAccepted` enum and the input-view submit match; a missed arm would drop an accepted item. Mitigation: exhaustive match (no wildcard) per repo convention; the accept-routing unit tests cover both kinds.
- Input-type snapshot/restore is new for this menu; getting it wrong could leave the user stranded in shell mode after dismissing a command preview. Mitigation: explicit dismiss/escape tests that assert the original input type is restored.
- The rename (if done) is mechanical but spans several TUI files; a stale reference would fail to compile. Mitigation: `cargo clippy`/build catches it; criteria require a clean build.
